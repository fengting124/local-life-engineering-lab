# 仓库结构

## 1. 结构目标

仓库按“代码、基础设施、文档”三条线分开，避免后期目录失控。

## 2. 根目录规划

```text
Personal_Projections/
├─ docs/
├─ infra/
├─ local-life-server/
├─ local-life-copilot/
├─ findings.md
├─ pom.xml
├─ progress.md
├─ README.md
├─ task_plan.md
└─ .gitignore
```

## 3. 各目录职责

### `docs/`

存放全过程文档，包括：

- `01-project/`：项目蓝图、边界、架构、路线
- `02-environment/`：环境与基础设施文档
- `03-process/`：规划过程入口
- `04-notes/`：个人学习笔记
- `05-reference/`：参考资料与规范底稿

核心内容包括：

- 环境搭建
- 项目边界
- 领域模型
- ER 图
- 时序图
- 技术选型
- 缓存设计
- MQ 与一致性
- Agent/RAG/MCP 设计
- 测试与压测
- 故障演练
- 复盘与简历表达

### `infra/`

存放本地开发和后续部署所需基础设施：

- `docker-compose.dev.yml`
- 容器环境变量模板
- 初始化 SQL
- 监控配置

### `local-life-server/`

传统后端主项目模块。  
后续会承载门店、内容、社交、营销、交易、搜索、对账补偿等系统能力。

### `local-life-copilot/`

AI 应用项目模块。  
后续会承载 WebFlux、SSE、Agent Controller、Tool Router、RAG Pipeline、Eval、MCP 等能力。

## 4. 模块边界

### `local-life-server`

阶段一只保留最小可运行骨架。  
业务模块拆分延后到领域模型和边界文档完成后再做。

### `local-life-copilot`

阶段一只保留最小可运行骨架。  
AI 依赖、模型接入、MCP 和评测模块延后到主项目数据链路稳定后再接入。

## 5. 当前目录策略

- 先有骨架，再有模块
- 先有文档，再有实现
- 先有边界，再有接口
- 根目录只保留仓库入口和过程状态文件
