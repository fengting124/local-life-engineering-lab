# 技术选型文档

## 1. 文档目标

本文档解释 `LocalLife` 和 `LocalLife Copilot` 的技术选型。

每个选型都按五个问题展开：

1. 用在哪里
2. 解决什么问题
3. 为什么选择它
4. 替代方案是什么
5. 当前阶段做到什么深度

该文档用于指导后续编码、压测、复盘和面试表达。

## 2. 总体原则

### 2.1 双线技术边界

| 项目 | 核心目标 | 技术重心 |
| --- | --- | --- |
| `LocalLife` | 证明 Java 后端工程能力 | Java、MySQL、Redis、RocketMQ、Elasticsearch、高并发、一致性 |
| `LocalLife Copilot` | 证明企业级 Agent 应用能力 | LangGraph、MCP、RAG、HITL、Evals、Tracing、Guardrails |

### 2.2 选型原则

1. 优先选择能支撑面试深挖的技术。
2. 优先选择能落地到业务链路的技术。
3. 优先复用同一套业务数据和基础设施。
4. 先做可控闭环，再扩展复杂能力。
5. 对每个暂缓技术说明原因，避免面试时被动。

## 4. LocalLife 主项目技术选型

### 4.1 Java 17

| 项 | 内容 |
| --- | --- |
| 使用位置 | 所有 Java 后端服务 |
| 解决问题 | 提供稳定运行时、现代语言特性、长期维护版本 |
| 选择理由 | Spring Boot 3.x 基线要求 Java 17，生态稳定，校招面试覆盖 JVM、并发、GC、类加载 |
| 替代方案 | Java 8、Java 21 |
| 当前深度 | 使用 Java 17，重点学习集合、JVM、并发、AQS、线程池、类加载 |

选择 Java 17 的原因：

1. Spring Boot 3.x 要求 Java 17 起步。
2. Java 17 是 LTS 版本，稳定性高。
3. 面试中的 JVM、锁、线程池、类加载都能围绕 Java 17 展开。
4. Java 21 虽然更新，但引入虚拟线程等话题后会增加学习面。

### 4.2 Spring Boot 3.x

| 项 | 内容 |
| --- | --- |
| 使用位置 | `local-life-server`、后续 Java MCP Server |
| 解决问题 | 快速构建 Web API、配置管理、依赖整合、监控接入 |
| 选择理由 | Java 后端主流事实标准，和 MySQL、Redis、MQ、监控生态兼容 |
| 替代方案 | Spring Cloud、Quarkus、Micronaut |
| 当前深度 | 第一阶段先做单体模块化，后续按边界拆服务 |

选择策略：

1. 第一阶段使用 Spring Boot 单体 + Maven 多模块。
2. 模块按 DDD 聚合根划分，例如 `user`、`shop`、`post`、`coupon`、`order`、`payment`。
3. 代码包结构强约束模块边界，Controller、Service、Repository 只访问本模块公开接口。
4. 暂不引入 Spring Cloud，避免过早增加注册中心、配置中心、网关和链路治理复杂度。
5. 后续如果拆服务，优先拆订单、营销、搜索、Copilot MCP Server。

### 4.3 Spring MVC 与 WebFlux

| 场景 | 选择 |
| --- | --- |
| LocalLife 业务 API | Spring MVC |
| Copilot SSE 流式输出 | FastAPI SSE 或 Spring WebFlux SSE |
| 高并发长连接 | WebFlux 可作为候选 |

LocalLife 主业务使用 Spring MVC。原因：

1. CRUD、交易、缓存、MQ 等同步业务链路更直观。
2. 调试和压测成本低。
3. 与 MyBatis、事务、拦截器集成更直接。

Copilot 流式输出可以使用 FastAPI SSE。原因：

1. Python Agent 服务天然接 LangGraph 和 LLM 生态。
2. SSE 只需要单向流式输出，复杂度低。
3. Java 侧保留 MCP Server 和业务能力。

### 4.4 MySQL 8

| 项 | 内容 |
| --- | --- |
| 使用位置 | 用户、商家、门店、内容、券、订单、支付、审计、会话 |
| 解决问题 | 保存业务事实，提供事务、索引、唯一约束、状态机持久化 |
| 选择理由 | Java 后端面试核心，覆盖索引、MVCC、事务、redo log、binlog、主从复制 |
| 替代方案 | PostgreSQL、TiDB、MongoDB |
| 当前深度 | 单库单表起步，重点做好索引和事务边界 |

MySQL 在项目中的职责：

1. 保存用户、门店、订单、支付单等主数据。
2. 通过唯一索引兜底幂等。
3. 通过状态字段支撑订单、支付、退款状态机。
4. 通过事务保证单库内一致性。
5. 后续通过 binlog 或补偿任务修复缓存和索引。

暂缓分库分表：

| 技术 | 暂缓原因 |
| --- | --- |
| PostgreSQL | 国内互联网后端主流训练和面试共识更偏 MySQL；PostgreSQL 在复杂查询和 GIS 上更强，但当前项目差异化需求有限 |
| ShardingSphere | 当前先完成单库链路，订单表后期再做分片演示 |
| TiDB | 分布式数据库运维复杂，项目核心先放在业务一致性 |
| MongoDB | 当前数据强关系、强事务，关系型数据库更合适 |

### 4.5 MyBatis / MyBatis-Plus

| 项 | 内容 |
| --- | --- |
| 使用位置 | Repository / Mapper 层 |
| 解决问题 | 让 SQL 可控，便于讲索引和慢 SQL 优化 |
| 选择理由 | 面试中能直接解释 SQL、索引命中、执行计划 |
| 替代方案 | JPA、JdbcTemplate |
| 当前深度 | 基础 CRUD 可用 MyBatis-Plus，复杂查询写自定义 SQL |

JPA 暂不优先。原因：

1. 自动生成 SQL 不利于训练索引意识。
2. 复杂查询和状态更新更需要 SQL 可控。
3. 本项目目标是后端工程能力，SQL 需要能讲清楚。

### 4.6 Redis 7

| 项 | 内容 |
| --- | --- |
| 使用位置 | 登录态、验证码、热点缓存、秒杀库存、限购、计数、限流 |
| 解决问题 | 承接高频读写，保护 MySQL，提供 TTL 和原子能力 |
| 选择理由 | 面试高频，能覆盖数据结构、持久化、网络模型、Lua、分布式锁 |
| 替代方案 | Memcached、KeyDB、本地缓存 |
| 当前深度 | 先做单 Redis，后续补充哨兵或集群讨论 |

Redis 数据结构使用：

| 结构 | 用途 |
| --- | --- |
| String | 验证码、Token、库存 |
| Hash | 笔记计数、用户摘要 |
| Set | 共同关注、用户抢券记录 |
| ZSet | Feed 流、排行榜候选 |
| Bitmap | 后续扩展签到、活跃用户、共同关注优化 |
| Bloom Filter | 防止门店、券模板、用户 ID 缓存穿透 |

### 4.7 Caffeine + Redis 多级缓存

| 项 | 内容 |
| --- | --- |
| 使用位置 | 热点门店、热点券模板、热点笔记 |
| 解决问题 | 降低 Redis 热点压力，缩短热点数据读取延迟 |
| 选择理由 | 本地缓存无网络开销，Redis 作为跨节点共享缓存 |
| 替代方案 | 只用 Redis、Guava Cache |
| 当前深度 | 后续在缓存设计文档中展开 |

分工：

| 层级 | 作用 |
| --- | --- |
| L1 Caffeine | 单机热点读，极低延迟 |
| L2 Redis | 跨实例共享缓存 |
| MySQL | 最终事实数据 |

一致性策略：

1. 写操作先更新 MySQL。
2. 删除 Redis 缓存。
3. 本地 Caffeine 通过 TTL 兜底。
4. 后续用 MQ 或 binlog 事件做二次删除和修复。

L1 Caffeine 失效策略：

1. TTL 兜底，热点对象 1 到 5 分钟自动失效。
2. 后续可接入 Redis Pub/Sub 或 RocketMQ 广播失效消息。
3. 关键热点 Key 可使用短 TTL + Stale-While-Revalidate，先返回旧值，再异步刷新。
4. 接受 L1 短窗口不一致，主链路以 Redis 和 MySQL 为准。

### 4.8 RocketMQ

| 项 | 内容 |
| --- | --- |
| 使用位置 | 秒杀异步下单、延时关单、支付成功事件、索引同步、补偿任务 |
| 解决问题 | 削峰、异步解耦、最终一致性、延时消息 |
| 选择理由 | 订单交易链路更需要事务消息、延时消息、重试、死信 |
| 替代方案 | Kafka、RabbitMQ |
| 当前深度 | 主用 RocketMQ，Kafka 和 RabbitMQ 写入选型对比 |

MQ 对比：

| 维度 | RocketMQ | Kafka | RabbitMQ |
| --- | --- | --- | --- |
| 典型场景 | 交易消息、延时消息、事务消息 | 日志、流处理、大吞吐 | 轻量队列、传统路由 |
| 延时消息 | 支持较好 | 需要额外设计 | 支持但吞吐较弱 |
| 事务消息 | 支持 | 原生语义不适合交易事务 | 支持能力有限 |
| 项目适配 | 高 | 中 | 中 |

本项目选择 RocketMQ 的原因：

1. 秒杀异步下单需要削峰。
2. 订单超时关闭需要延时消息。
3. 支付成功后需要异步通知履约、券发放、对账。
4. 死信和重试可以支撑补偿链路。

### 4.9 Elasticsearch 8

| 项 | 内容 |
| --- | --- |
| 使用位置 | 门店搜索、内容搜索、地理位置搜索 |
| 解决问题 | 全文检索、倒排索引、聚合、地理位置查询 |
| 选择理由 | MySQL 不适合复杂全文搜索和相关性排序 |
| 替代方案 | MySQL LIKE、OpenSearch、Solr |
| 当前深度 | 先做门店和笔记搜索，后续做索引同步一致性 |

使用边界：

1. MySQL 保存事实数据。
2. Elasticsearch 保存搜索索引。
3. 搜索结果可以短暂延迟。
4. 详情页以 MySQL 为准。
5. 索引同步失败由补偿任务修复。

### 4.10 Sentinel

| 项 | 内容 |
| --- | --- |
| 使用位置 | 登录、抢券、Copilot 工具调用、LLM 调用入口 |
| 解决问题 | 限流、熔断、热点参数保护、降级 |
| 选择理由 | 高并发入口需要保护 MySQL、Redis、MQ 和 LLM 成本 |
| 替代方案 | Resilience4j、Guava RateLimiter、Redis 限流 |
| 当前深度 | 第一阶段先文档设计，编码阶段从 Redis 限流起步 |

策略：

1. 登录接口做手机号和 IP 维度限流。
2. 秒杀接口做 QPS 限流和热点参数限流。
3. Copilot 做用户级限流、工具级限流和 token 预算。
4. Redis 故障时对高风险入口直接降级。

### 4.11 Micrometer + Prometheus + Grafana

| 项 | 内容 |
| --- | --- |
| 使用位置 | JVM、接口耗时、Redis、MQ、MySQL、业务指标 |
| 解决问题 | 压测、瓶颈定位、故障复盘 |
| 选择理由 | Spring Boot 生态集成成熟 |
| 替代方案 | SkyWalking、OpenTelemetry、ELK |
| 当前深度 | 主项目先接基础指标，后续补链路追踪 |

关键指标：

| 类型 | 指标 |
| --- | --- |
| JVM | GC、堆内存、线程数 |
| HTTP | QPS、P95、P99、错误率 |
| Redis | 命中率、耗时、错误数 |
| MySQL | 慢 SQL、连接池、事务耗时 |
| MQ | 消息堆积、消费失败、死信数 |
| 业务 | 抢券成功率、支付成功率、对账修复数 |

### 4.12 Docker Compose

| 项 | 内容 |
| --- | --- |
| 使用位置 | 本地 MySQL、Redis、RocketMQ、Elasticsearch、Milvus |
| 解决问题 | 统一开发环境，避免手工安装污染系统 |
| 选择理由 | 本地可复现，适合多中间件项目 |
| 替代方案 | 本机安装、Kubernetes、云服务 |

本地中间件通过 Docker 跑在本机或 WSL2 后端中。代码可以在 Windows 工作区开发，服务通过 localhost 连接容器端口。

## 5. LocalLife 工程问题选型

### 5.1 缓存一致性

主方案：

```text
写 MySQL
   |
   v
删除 Redis 缓存
   |
   v
本地缓存 TTL 兜底
   |
   v
MQ 或 binlog 修复链路
```

选择删除缓存的原因：

1. 更新缓存容易写入旧值。
2. 删除缓存让下一次读请求重建数据。
3. 业务层删除低延迟，补偿链路兜底异常情况。

暂缓直接引入 Canal：

1. 第一阶段先做业务层删除和 MQ 补偿。
2. Canal 后续用于搜索索引和缓存修复增强。
3. 过早引入会增加环境复杂度。

### 5.2 秒杀库存

主方案：

```text
Redis 预热库存
   |
   v
Lua 原子预扣库存和限购
   |
   v
RocketMQ 异步下单
   |
   v
MySQL 创建订单
```

选择 Redis + Lua 的原因：

1. Redis 承接高并发请求。
2. Lua 保证库存扣减和用户限购在 Redis 内原子执行。
3. MySQL 只处理预扣成功后的订单写入。
4. MQ 负责削峰和失败重试。

### 5.3 分布式 ID

主方案：雪花算法。

| 项 | 内容 |
| --- | --- |
| 使用位置 | 用户、门店、订单、支付单、消息事件 |
| 解决问题 | 避免数据库自增 ID 在分库分表时受限 |
| 选择理由 | 趋势递增，适合 MySQL B+ 树索引 |
| 替代方案 | UUID、数据库自增、Redis 自增、Leaf、UidGenerator、Tinyid |

暂缓复杂时钟回拨策略：

1. 第一阶段可先实现基础雪花 ID。
2. 第二阶段补时钟回拨检测和机器号配置。
3. 面试时解释 UUID 对聚簇索引不友好。

暂缓 Leaf / UidGenerator / Tinyid：

1. 这些方案优势在大集群部署和号段治理，本项目单服务起步不需要。
2. 雪花算法已能满足趋势递增和分布式唯一。
3. 后续如果需要号段模式或跨机房 ID 管理，可引入 Leaf。

### 5.4 接口幂等

主方案：

| 场景 | 幂等方式 |
| --- | --- |
| 登录 | 验证码一次性删除 |
| 抢券 | Redis Set + MySQL 唯一索引 |
| 下单 | requestId + Redis SETNX |
| 支付回调 | 渠道流水唯一索引 + 状态机更新 |
| MQ 消费 | eventId + 业务唯一键 |

后续可封装统一幂等注解：

```text
@Idempotent(key = "userId + requestId", ttl = 60)
```

### 5.5 对账补偿

主方案：

1. 支付渠道为资金事实来源。
2. MySQL 保存本地订单和支付单事实。
3. RocketMQ 处理异步通知。
4. 定时任务扫描异常订单。
5. 所有修复操作必须幂等。

适用场景：

| 场景 | 补偿 |
| --- | --- |
| 支付成功但本地未更新 | 补更新支付单和订单 |
| 订单关闭后支付成功 | 退款或人工处理 |
| MQ 消费失败 | 重试、死信、补偿任务 |
| Redis 预扣和 MySQL 不一致 | 按订单事实修复库存 |

## 6. Copilot 技术选型

### 6.1 Java MCP Server

| 项 | 内容 |
| --- | --- |
| 使用位置 | `locallife-mcp-server` |
| 解决问题 | 把 Java 主服务业务能力标准化为 Agent 工具 |
| 选择理由 | 业务规则保留在 Java，Agent 只通过协议调用工具 |
| 替代方案 | Python 直接 HTTP 调 Java API、OpenAPI Tool |
| 当前深度 | 第二阶段实现查询类工具，再实现高风险执行类工具 |

选择 MCP 的原因：

1. 工具描述、输入、输出可以标准化。
2. Agent 服务和业务服务解耦。
3. 工具可被兼容 MCP 的 Host 复用。
4. 权限、审计、限流可以集中在工具层做。

### 6.2 Python + FastAPI Agent Service

| 项 | 内容 |
| --- | --- |
| 使用位置 | `copilot-agent-service` |
| 解决问题 | 承载 LangGraph、LLM 调用、RAG、Evals |
| 选择理由 | Python AI 生态成熟，LangGraph、Milvus、评测工具接入成本低 |
| 替代方案 | Java Spring AI、Node.js |
| 当前深度 | 第二阶段启动，先跑通 ReAct 和 MCP Client |

Java 侧不承担 Agent 编排的原因：

1. Python 对 LangGraph、RAG、Evals 的支持更完整。
2. Java 保留业务工程能力。
3. 两者通过 MCP 协议解耦，边界清晰。

### 6.3 LangGraph

| 项 | 内容 |
| --- | --- |
| 使用位置 | Agent 编排层 |
| 解决问题 | 多步 Agent、状态管理、checkpoint、HITL 恢复 |
| 选择理由 | 能表达 ReAct 循环、人工审批、失败恢复和长任务 |
| 替代方案 | LangChain Agent、OpenAI Agents SDK、自研 Workflow Engine |
| 当前深度 | 第二阶段实现单 Agent，不做多 Agent |

选择 LangGraph 的原因：

1. 状态图适合表达 Agent 多步执行。
2. Checkpoint 支撑异常恢复和 HITL。
3. interrupt 支撑人工审批。
4. 比自研工作流更容易落地和解释。

### 6.4 ReAct + Self-Reflection

| 项 | 内容 |
| --- | --- |
| 使用位置 | 订单异常排查、活动配置辅助 |
| 解决问题 | 让 Agent 根据工具反馈决定下一步 |
| 选择理由 | 适合工具调用和多步证据收集 |
| 替代方案 | 固定 Workflow、单轮 Function Calling |
| 当前深度 | 单 Agent ReAct，Reflection 只在失败、每 5 步和最终回答前触发 |

使用边界：

1. 简单知识问答走 RAG 单轮。
2. 简单数据查询走单工具调用。
3. 异常排查和活动配置走 ReAct 多步。

### 6.5 LLM Provider

| 项 | 内容 |
| --- | --- |
| 使用位置 | Agent 决策、自然语言生成、Query Rewrite、草稿生成 |
| 解决问题 | 理解用户意图，选择工具，生成答案 |
| 选择理由 | 通过环境变量切换模型供应商，避免绑定单一模型 |
| 替代方案 | 本地模型、单一云模型 |
| 当前深度 | 先抽象 Provider 接口，通过环境变量切换模型 |

当前候选模型：

| 用途 | 模型 |
| --- | --- |
| 默认对话模型 | `deepseek-v4-flash` |
| 可切换云模型 | Claude / OpenAI / Qwen |
| 本地模型备选 | Ollama 兼容 OpenAI API |
| Embedding | `intfloat/multilingual-e5-base` |
| Reranker | `BAAI/bge-reranker-base` |

选择策略：

1. 业务代码不直接绑定具体模型。
2. 模型供应商通过 Provider 接口抽象。
3. 日常开发默认 DeepSeek flash；如果要做模型对比，只改 `.env`，不改业务代码。
4. Query Rewrite、摘要、低风险生成可切到成本更低的模型。

Provider 接口需要统一：

1. Chat completion。
2. Tool calling。
3. Streaming。
4. Token usage。
5. Timeout 和 retry。

### 6.6 Embedding 与 Reranker

| 模块 | 选型 | 理由 |
| --- | --- | --- |
| Embedding | `text-embedding-3-small` 或同级别通用 embedding | 成本低，效果够用 |
| Reranker | `bge-reranker-v2-m3` | 适合中英文重排，开源可控 |

使用策略：

1. 向量召回 top 20。
2. metadata 权限过滤。
3. reranker 取 top 5。
4. 分数低于阈值拒答。

### 6.7 Milvus

| 项 | 内容 |
| --- | --- |
| 使用位置 | 商家知识库、平台规则、FAQ、复盘文档向量检索 |
| 解决问题 | 语义检索和 metadata 权限过滤 |
| 选择理由 | 向量检索能力明确，支持 filtered search |
| 替代方案 | pgvector、Elasticsearch vector、Chroma |
| 当前深度 | 第二阶段接入，先做权限感知 RAG |

暂缓 pgvector：

1. 当前主数据库是 MySQL。
2. 为向量检索额外引入 PostgreSQL 会增加运维复杂度。
3. Milvus 更适合作为独立向量检索组件。

### 6.8 Elasticsearch 在 Copilot 中的作用

| 场景 | 作用 |
| --- | --- |
| 门店搜索 | BM25、地理位置、类目过滤 |
| 内容搜索 | 关键词、标题、正文检索 |
| RAG 混合检索 | 后续可与 Milvus 组合做 Hybrid Search |

Copilot 不重复建设全文检索系统，直接复用 LocalLife 的 Elasticsearch 索引。

### 6.9 MySQL 与 Redis 在 Copilot 中的作用

| 数据 | 存储 |
| --- | --- |
| 会话 | MySQL |
| 消息 | MySQL |
| 工具审计 | MySQL |
| HITL 审批 | MySQL |
| Checkpoint | MySQL |
| 活跃会话 | Redis |
| 限流计数 | Redis |
| RAG 结果缓存 | Redis |
| token 预算 | Redis |

原则：

1. 需要审计和回放的数据进 MySQL。
2. 高频、短期、带 TTL 的数据进 Redis。
3. Agent 状态可恢复，不能只放内存。

### 6.10 HITL

| 项 | 内容 |
| --- | --- |
| 使用位置 | 退款、补券、活动发布、批量操作 |
| 解决问题 | 防止 Agent 自动执行高风险动作 |
| 选择理由 | 企业级 Agent 必须有人审、可追责、可回放 |
| 替代方案 | 全自动执行、只读 Agent |
| 当前深度 | 第二阶段先做审批表和简版审批工作台 |

### 6.11 Evals

| 项 | 内容 |
| --- | --- |
| 使用位置 | Agent 质量评估 |
| 解决问题 | 用指标判断 Agent 是否真的变好 |
| 选择理由 | 简历和面试需要量化结果 |
| 替代方案 | 只靠人工体验 |
| 当前深度 | 第二阶段构建 50 条评测集 |

指标：

1. 工具调用准确率。
2. 任务完成率。
3. Recall@5。
4. 回答事实一致性。
5. 平均延迟。
6. 平均 token 成本。

该 6 项指标与 `07-Copilot企业级Agent设计.md` 保持一致。

### 6.12 LangSmith 与自建 Trace

| 方案 | 作用 |
| --- | --- |
| LangSmith | 开发期可视化观察 Agent 执行轨迹 |
| 自建 Trace 表 | 生产持久化审计，支持回放和复盘 |

选择双轨方案的原因：

1. LangSmith 调试效率高。
2. 自建审计表更容易和业务权限、会话、HITL 关联。
3. 面试可以讲清开发期观测和生产审计的差异。

第一阶段自建 Trace 表先落到 `tool_audit_log`，用于记录工具调用。后续再扩展 LLM 调用、RAG 检索和 HITL 审批明细。

### 6.13 Guardrails

| 风险 | 方案 |
| --- | --- |
| Prompt Injection | 输入检测、工具层权限强校验 |
| 权限越界 | 服务端注入 merchant_id，Agent 无权覆盖 |
| 编造答案 | RAG 低分拒答 |
| 私有信息泄露 | 输出前检查商家边界 |
| 高风险动作误执行 | HITL |

## 9. 面试表达要点

### 9.1 LocalLife

| 问题 | 回答方向 |
| --- | --- |
| 为什么用 RocketMQ | 秒杀下单、延时关单、支付事件都需要异步和延时能力 |
| 为什么用 Redis Lua | 库存和限购必须原子执行 |
| 为什么缓存一致性主方案是删除缓存 | 更新缓存容易写旧值，删除后由读请求重建 |
| Redis 挂了怎么办 | 秒杀入口降级，热点读用本地缓存兜底，保护 MySQL |
| 为什么不用 Seata | 当前更适合 MQ 最终一致性和对账补偿 |
| 为什么订单和支付单分开 | 订单是业务事实，支付单是资金事实 |

### 9.2 Copilot

| 问题 | 回答方向 |
| --- | --- |
| 为什么用 MCP | 标准化工具协议，解耦 Agent 和业务系统 |
| 为什么用 LangGraph | 需要状态管理、checkpoint、HITL、长任务恢复 |
| 为什么业务规则留在 Java | Java 主服务负责业务事实和权限，Agent 只负责编排 |
| 为什么需要 HITL | 退款、补券、活动发布属于高风险动作 |
| RAG 怎么做权限隔离 | Milvus metadata 过滤，服务端注入 merchant_id |
| 怎么证明 Agent 有效果 | Evals 指标，包括工具调用准确率、Recall@5、任务完成率 |
