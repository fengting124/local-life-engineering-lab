# 项目增强路线图

## 1. 文档目标

本文档用于收敛 `LocalLife` 和 `LocalLife Copilot` 的增强项，防止项目被新技术不断打断。

本文档只回答四个问题：

1. 哪些增强项必须进入项目
2. 哪些增强项延后进入项目
3. 哪些增强项明确不做
4. 每个增强项放到什么阶段落地

## 2. 当前基线能力

### 2.1 LocalLife 主项目基线

| 能力 | 当前状态 |
| --- | --- |
| Java 后端基础 | Java 17 + Spring Boot 3.x |
| 数据库 | MySQL 8 |
| 缓存 | Redis 7 + Caffeine |
| 秒杀 | Redis Lua 预扣 + RocketMQ 异步下单 |
| 订单支付 | 订单状态机、支付单、回调幂等、延时关单 |
| 消息 | RocketMQ 事件、重试、死信、补偿设计 |
| 搜索 | Elasticsearch 作为门店和内容搜索索引 |
| 观测 | Micrometer + Prometheus + Grafana |
| 限流 | Sentinel 或 Redis 限流 |
| 补偿 | 对账任务、异常修复、定时兜底 |

### 2.2 LocalLife Copilot 基线

| 能力 | 当前状态 |
| --- | --- |
| Agent 架构 | Python FastAPI + LangGraph |
| 工具协议 | Java MCP Server + Python MCP Client |
| 决策机制 | ReAct + Self-Reflection |
| RAG | Milvus + Reranker + 权限过滤 |
| HITL | 高风险动作审批、Checkpoint 恢复 |
| Evals | 50 条评测集 + 6 项指标 |
| Trace | LangSmith + `tool_audit_log` |
| Guardrails | 权限、输入、输出、工具调用保护 |

结论：当前地基已经足够。后续增强必须围绕业务闭环、工程深度和可量化指标展开。

## 3. 必须增强项

这些能力会显著提升项目档次，且能直接服务核心业务链路。

### 3.1 消息防丢三件套

| 项 | 内容 |
| --- | --- |
| 所属项目 | LocalLife |
| 优先级 | P0 |
| 涉及链路 | 抢券、下单、支付、对账补偿 |
| 解决问题 | MQ 发送失败、消费失败、消息重复、死信无人处理 |
| 落地阶段 | LocalLife 第 4 到 6 周 |

设计内容：

1. RocketMQ 事务消息用于强一致消息场景。
2. 本地消息表用于业务事务和消息发送解耦。
3. 死信队列用于消费失败后的人工或自动补偿。
4. 消费者通过 `eventId` 和业务唯一键保证幂等。

推荐落地顺序：

```text
支付成功事件
   |
   v
本地消息表
   |
   v
RocketMQ 投递
   |
   v
消费者幂等处理
   |
   v
失败进入死信
   |
   v
补偿任务重投或人工处理
```

面试价值：

| 追问 | 回答方向 |
| --- | --- |
| MQ 发送失败怎么办 | 本地消息表记录，定时任务重发 |
| 消费失败怎么办 | RocketMQ 重试，超过阈值进死信 |
| 重复消费怎么办 | eventId + 业务唯一索引幂等 |
| 本地事务和消息怎么一致 | 事务消息或业务表与消息表同事务 |

### 3.2 Canal + Binlog 同步管道

| 项 | 内容 |
| --- | --- |
| 所属项目 | LocalLife |
| 优先级 | P1 |
| 涉及链路 | 缓存一致性、ES 索引同步、审计 |
| 解决问题 | 业务双写不一致、缓存修复、索引延迟 |
| 落地阶段 | LocalLife 第 5 到 6 周后 |

设计形态：

```text
MySQL Binlog
   |
   v
Canal Server
   |
   v
RocketMQ Topic
   |
   +--> CacheRepairConsumer
   |
   +--> SearchIndexConsumer
   |
   +--> DataAuditConsumer
```

使用边界：

1. 主链路仍然由业务代码写 MySQL。
2. 缓存删除仍然在业务层做第一道处理。
3. Canal 作为补偿和派生数据同步链路。
4. ES 索引同步可以优先通过 MQ 事件做，Canal 后续做兜底。

暂缓原因：

1. Canal 增加本地环境复杂度。
2. 第一阶段先把业务链路跑通。
3. 编码阶段中后期再引入更合适。

### 3.3 SkyWalking / OpenTelemetry 全链路追踪

| 项 | 内容 |
| --- | --- |
| 所属项目 | LocalLife + Copilot |
| 优先级 | P1 |
| 涉及链路 | 登录、抢券、下单支付、MCP、Agent |
| 解决问题 | 跨服务调用定位困难 |
| 落地阶段 | 压测与观测阶段 |

设计形态：

```text
Chat UI
   |
   v
copilot-agent-service
   |
   v
locallife-mcp-server
   |
   v
LocalLife Server
   |
   v
MySQL / Redis / RocketMQ / Elasticsearch
```

追踪策略：

1. Java 服务接 SkyWalking Agent 或 OpenTelemetry Java Agent。
2. Python Agent 服务接 OpenTelemetry。
3. MCP 调用透传 `trace_id`。
4. MQ 消息携带 `trace_id` 和 `eventId`。
5. 日志通过 MDC 输出 `trace_id`。

面试价值：

| 问题 | 回答方向 |
| --- | --- |
| 压测瓶颈怎么定位 | trace + metrics + logs 联合定位 |
| MQ 异步链路怎么串起来 | 消息体携带 trace_id |
| Copilot 调 Java 工具怎么追踪 | MCP 请求头或上下文透传 trace_id |

### 3.4 Streaming Tool Call

| 项 | 内容 |
| --- | --- |
| 所属项目 | LocalLife Copilot |
| 优先级 | P1 |
| 涉及链路 | Agent SSE 输出 |
| 解决问题 | 用户等待时间长、工具调用过程不可见 |
| 落地阶段 | Copilot 第 10 周体验优化 |

设计形态：

```text
用户提问
   |
   v
SSE: 正在分析问题
   |
   v
SSE: 调用 query_order
   |
   v
SSE: 返回订单状态
   |
   v
SSE: 调用 query_coupon_issue_log
   |
   v
SSE: 输出最终结论
```

事件类型：

| SSE event | 含义 |
| --- | --- |
| `agent_step` | Agent 进入新步骤 |
| `tool_call` | 即将调用工具 |
| `tool_result` | 工具返回结果 |
| `reflection` | Agent 反思 |
| `final_answer` | 最终回答 |
| `error` | 异常 |

量化指标：

1. 首字响应时间 `TTFT`。
2. 端到端耗时。
3. 工具调用可见率。
4. 用户取消率。

### 3.5 Prompt Injection 防护

| 项 | 内容 |
| --- | --- |
| 所属项目 | LocalLife Copilot |
| 优先级 | P1 |
| 涉及链路 | 输入、工具调用、输出 |
| 解决问题 | Prompt Injection、权限绕过、隐私泄露 |
| 落地阶段 | Copilot Guardrails 阶段 |

三层防护：

| 层级 | 策略 |
| --- | --- |
| 输入层 | 检测忽略系统规则、角色伪装、工具泄露请求 |
| 工具层 | 服务端强制注入 `user_id`、`merchant_id`、`role` |
| 输出层 | 检查跨商家数据、手机号、支付流水等敏感信息 |

关键原则：

1. Agent 不能自行决定权限。
2. Agent 不能覆盖服务端注入的身份上下文。
3. 高风险工具必须 HITL。
4. 工具返回给模型的数据需要最小化。

### 3.6 ShardingSphere 订单分库分表

| 项 | 内容 |
| --- | --- |
| 所属项目 | LocalLife |
| 优先级 | P2 |
| 涉及链路 | 订单、支付、用户订单列表 |
| 解决问题 | 单表数据量膨胀、订单写入和查询压力 |
| 落地阶段 | LocalLife 主链路完成后 |

建议设计：

| 维度 | 策略 |
| --- | --- |
| 分库键 | `user_id` |
| 分表键 | `created_at` 月份 |
| 分片组件 | ShardingSphere-JDBC |
| 跨片查询 | 通过 ES 订单索引承接 |
| ID | 雪花算法 |

暂缓原因：

1. 当前要先完成单库订单链路。
2. 分库分表会影响事务、分页、唯一约束和查询路由。
3. 适合在主项目稳定后作为增强项。

## 4. 延后增强项

这些能力有价值，但必须等主链路完成后再做。

### 4.1 模型路由

| 项 | 内容 |
| --- | --- |
| 所属项目 | Copilot |
| 优先级 | P2 |
| 解决问题 | LLM 成本和延迟 |
| 前置条件 | Evals 跑通、Provider 接口稳定 |

策略：

1. 简单数据查询走低成本模型。
2. 复杂异常排查走主力模型。
3. Query Rewrite、摘要、分类走低成本模型。
4. 通过 Evals 对比任务完成率和成本。

### 4.2 Hybrid Search

| 项 | 内容 |
| --- | --- |
| 所属项目 | Copilot |
| 优先级 | P2 |
| 解决问题 | 专有名词、订单号、券号、规则编号召回不稳定 |
| 前置条件 | Milvus RAG 和 Elasticsearch 索引都已完成 |

设计：

```text
Query
   |
   +--> Milvus 向量召回
   |
   +--> Elasticsearch BM25 召回
   |
   v
结果融合
   |
   v
Reranker
   |
   v
Top 5 Context
```

量化指标：`Recall@5`。

### 4.3 Agent Memory

| 项 | 内容 |
| --- | --- |
| 所属项目 | Copilot |
| 优先级 | P3 |
| 解决问题 | 商家偏好、历史上下文、长期运营习惯 |
| 前置条件 | 会话、消息、RAG、Evals 已稳定 |

当前不急着做。原因：

1. 长期记忆容易引入隐私和权限问题。
2. 需要清晰的记忆写入、读取、删除策略。
3. 第一阶段 Copilot 重点是工具调用和异常排查。

### 4.4 智能运营报告

| 项 | 内容 |
| --- | --- |
| 所属项目 | Copilot |
| 优先级 | P3 |
| 方向 | Proactive Agent |
| 前置条件 | 数据查询工具、RAG、报告生成、调度任务已完成 |

设计：

```text
每天凌晨
   |
   v
拉取昨日经营数据
   |
   v
识别异常指标
   |
   v
检索活动规则和历史案例
   |
   v
生成运营日报
   |
   v
推送给商家或运营
```

该能力适合作为最终展示亮点，不进入第一阶段。

## 5. 明确不做项

| 技术 | 不做原因 |
| --- | --- |
| Seata | 当前用 MQ 最终一致性 + 对账补偿，更符合交易场景 |
| Kubernetes | 本地开发 Docker Compose 足够，K8s 运维成本高 |
| A2A 多 Agent | 当前单 Agent 能覆盖业务，强行多 Agent 会稀释主线 |
| 自训练 LLM / Fine-tuning | 成本高、收益不确定，当前重点是工具、RAG、评测 |
| TiDB | MySQL 足够支撑项目训练目标 |
| GraphRAG | 本地生活交易平台不是强图谱场景 |
| 实时音视频 / 多模态 | 与当前业务主线弱相关 |

## 7. 简历亮点映射

| 增强项 | 简历表达 |
| --- | --- |
| 消息防丢三件套 | 构建事务消息、本地消息表、死信处理和补偿任务，保障支付成功事件、抢券下单事件最终可达 |
| Canal + Binlog | 基于 binlog 订阅构建缓存修复和 ES 索引同步链路，降低业务双写不一致风险 |
| SkyWalking / OpenTelemetry | 建立 Java + Python + MCP 跨语言链路追踪，支持秒杀和 Agent 工具调用瓶颈定位 |
| Streaming Tool Call | 基于 SSE 实现 Agent 工具调用过程流式展示，降低首字响应时间 |
| Prompt Injection 防护 | 构建输入、工具、输出三层 Guardrails，阻止越权工具调用和跨商家数据泄露 |
| ShardingSphere | 对订单表按用户和时间维度分片，解决订单数据膨胀和跨片查询问题 |
| 模型路由 | 按任务复杂度动态选择模型，降低 token 成本和延迟 |
| Hybrid Search | 结合向量召回和 BM25，提高专有名词和业务编号的召回率 |

## 8. 执行原则

1. 主链路优先，增强项不能阻断核心业务编码。
2. 每个增强项都必须能回答业务价值。
3. 每个增强项都必须能产出可验证指标。
4. 先做 LocalLife 的交易、缓存、消息、搜索，再做 Copilot 的 Agent 深化。
5. 遇到新技术建议，先放入本文档评估，不直接插入当前编码计划。
