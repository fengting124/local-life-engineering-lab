# 双线项目路线

## 1. 总体原则

项目按同一业务域拆成两条线：

1. 传统后端主项目，负责证明 Java 基础、数据库、中间件、分布式和高并发处理能力。
2. AI 应用项目，负责证明 Agent、RAG、工具调度、评测、流式链路和 MCP 设计能力。

两条线共用同一套业务数据、事件流和知识库，避免做成两个互不相关的简历项目。

## 2. 统一业务域

业务域定为：`本地生活内容交易平台`

核心场景：

- 用户浏览门店、内容笔记、活动和商品
- 用户关注达人和商家，形成社交关系链
- 平台发券、抢券、下单、支付、退款、对账
- 商家需要内容运营、活动配置、异常订单排查和知识问答

这样做的好处：

- 后端线天然覆盖内容社区、交易系统、本地生活三类典型场景
- AI 线天然有 Tool、RAG、工作流、MCP 的业务落点
- 面试时可以讲同一域内的数据流、事件流、知识流

## 3. 传统后端主项目

项目名先定为：`LocalLife`

### 3.1 核心模块

- 用户与鉴权：登录、Token、权限、风控基础
- 商家与门店：商家资料、营业状态、门店地理位置、类目
- 内容社区：探店笔记、评论、点赞、收藏、举报
- 社交关系：关注、共同关注、Feed 基础能力
- 营销系统：优惠券、活动、秒杀场次、限购规则
- 交易系统：购物车、订单、支付单、退款单、状态流转
- 搜索系统：门店搜索、内容搜索、地理位置搜索
- 对账补偿：支付结果对账、库存校正、消息补偿、死信处理

### 3.2 必做工程问题

- 多级缓存：`Caffeine + Redis`
- 缓存一致性：删除缓存主方案 + Binlog/MQ 修复链路
- Redis 宕机兜底：限流、降级、本地缓存兜底、热点白名单
- 秒杀预扣减：`Redis + Lua` 原子预扣，异步下单
- MQ 选型：主用 `RocketMQ`，单独写一份 Kafka/RocketMQ/RabbitMQ 选型文档
- 搜索方案：`Elasticsearch`，支持全文检索、联想、地理位置检索
- 布隆过滤器：门店、券模板、用户 ID 防穿透
- 共同关注计算：`Redis Set` 交集，后续扩展 Bitmap 方案
- QPS 压测：`wrk/JMeter + Prometheus + Grafana`
- 对账补偿：订单、支付、库存三方对账 + 定时修复

### 3.3 关键技术决策

- Java：`17`
- 框架：`Spring Boot 3.x`
- 数据库：`MySQL 8`
- 缓存：`Redis 7`
- 本地缓存：`Caffeine`
- MQ：`RocketMQ`
- 搜索：`Elasticsearch 8`
- 可观测性：`Micrometer + Prometheus + Grafana`
- 部署：`Docker Compose`

### 3.4 面试深挖点

- 为什么缓存一致性主方案选删除缓存，不选更新缓存
- Redis 挂掉时读流量如何控住，不让数据库被打穿
- 秒杀库存如何避免超卖、少卖、重复下单
- MQ 消息重复、乱序、丢失怎么处理
- 搜索索引延迟和数据库一致性怎么平衡
- 共同关注为什么先用 Set，再谈 Bitmap
- 压测瓶颈如何定位，怎么判断在 JVM、DB、Redis、网络还是锁
- 对账补偿如何保证最终一致性，如何避免修复任务重复执行

## 4. AI 应用项目

项目名先定为：`LocalLife Copilot`

### 4.1 项目定位

这个项目服务于商家运营、平台客服和运营审批，直接消费 `LocalLife` 的业务数据、文档和工具。

升级后的定位是：`企业级自主决策 Agent 应用`。

核心边界：

- `LocalLife` 主服务负责订单、支付、券、门店等真实业务规则
- `locallife-mcp-server` 负责把 Java 业务能力标准化为 MCP 工具
- `copilot-agent-service` 负责 LangGraph 编排、ReAct 决策、RAG、HITL 和评测
- Chat UI 和审批工作台负责交互与高风险动作确认

目标能力：

- 商家知识问答
- 活动配置辅助
- 订单异常排查
- 券投放规则解释
- 内容运营分析
- 门店数据查询

### 4.2 核心模块

- Chat Gateway：对前端提供 `SSE` 流式输出
- Agent Runtime：基于 `LangGraph` 管理状态、checkpoint、恢复和终止
- ReAct Controller：执行 `Thought -> Action -> Observation -> Reflection -> Final`
- Tool Router：按角色、任务类型和风险等级筛选工具子集
- MCP Layer：Java 主服务通过 MCP 暴露标准化业务工具
- RAG Pipeline：解析、切分、召回、重排、压缩、生成
- HITL Center：退款、补券、活动发布等高风险动作审批
- Eval Center：离线评测集、在线指标、错误样本回放
- Governance：RBAC、审计、Trace、限流、Guardrails

### 4.3 必做工程问题

- Tool 调用：输入输出 Schema、超时、重试、幂等、审计日志
- Agent 主循环：`Thought -> Action -> Observation -> Reflection -> Final`
- Controller：工具白名单、最大步数、token 预算、失败终止条件、回退策略
- Tool Router：按领域动态下发工具子集，避免把所有工具塞进上下文
- RAG 评测：`Recall@K`、`MRR`、回答相关性、事实一致性
- 上下文压缩：Query Rewrite、TopK 裁剪、Rerank、片段压缩
- 异常兜底：工具失败降级、检索失败拒答、模型超时回退
- 流式返回：前端使用 SSE，后端使用 `Spring WebFlux`
- MCP 服务设计：本地走 `stdio`，远程走 `Streamable HTTP`
- HITL：退款、补券、活动发布等高风险动作必须人工审批
- Evals：工具调用准确率、任务完成率、Recall@5、事实一致性、延迟、成本

### 4.4 第一批工具

- `query_order`
- `query_payment`
- `query_coupon_issue_log`
- `query_mq_dead_letter`
- `shop_metrics_query`
- `coupon_policy_lookup`
- `knowledge_search`
- `campaign_draft_generate`
- `execute_refund`
- `issue_compensation_coupon`

### 4.5 面试深挖点

- 为什么需要 Router，为什么不能把所有工具直接给模型
- 工具太多时如何治理命名、Schema、版本和观测
- RAG 召回率和回答准确率如何定义、怎么测
- 为什么用户流式输出用 SSE，但 MCP 远程传输优先选 Streamable HTTP
- 工具失败时如何保证 Agent 不乱答
- 如何判断问题该走检索、工具还是纯模型回答
- MCP 和 Skill 的边界怎么划
- 为什么业务规则保留在 Java 主服务，Agent 只做编排
- 为什么高风险动作必须 HITL
- 如何用 Evals 证明 Agent 质量提升

## 5. 双线之间的关系

`LocalLife` 提供业务系统、数据库、缓存、事件流和搜索索引。  
`LocalLife Copilot` 消费这些能力，并通过 MCP 对工具进行标准化暴露。

共享内容：

- 用户、商家、订单、活动、内容数据模型
- MQ 事件流
- Elasticsearch 索引
- 监控与日志体系
- 故障案例和复盘文档

## 6. 开发顺序

顺序固定：

1. 先做 `LocalLife` 主链路和工程问题。
2. 再做 `LocalLife Copilot`，直接复用主项目数据和能力。

原因：

- 先有真实业务系统，AI 项目才有可靠工具和知识源
- 先把 Java、MySQL、Redis、MQ 打牢，Agent 项目才有可讲的后端含量
- 简历表达也更稳，主项目讲工程底座，副项目讲 AI 增量能力

## 7. 八周路线

### 第 1 周

- 写产品边界文档
- 画领域模型和 ER 图
- 画核心时序图：登录、发笔记、抢券、下单、支付回调
- 输出技术选型文档

### 第 2 周

- 搭项目骨架
- 完成用户、商家、内容、关注基础模块
- 建立统一异常、日志、鉴权、接口规范

### 第 3 周

- 上 Redis 基础缓存
- 做缓存穿透、击穿、雪崩防护
- 做共同关注与 Feed 基础能力

### 第 4 周

- 做优惠券、秒杀、异步下单
- 引入 MQ、幂等、防重复下单
- 做支付超时取消

### 第 5 周

- 做搜索、索引同步、Geo 搜索
- 做压测、监控、热点治理
- 做 Redis 宕机降级演练

### 第 6 周

- 做对账补偿
- 做死信处理
- 做核心链路复盘文档

### 第 7 周

- 搭 AI 网关、SSE 流式返回
- 实现 Tool Router、Controller、第一批工具
- 接入基础 RAG

### 第 8 周

- 做评测集
- 做召回率、回答准确率、工具成功率统计
- 做 MCP Server
- 输出简历版项目描述和面试题库

## 9. 简历表达骨架

### 传统后端主项目

- 基于 `Spring Boot + MySQL + Redis + RocketMQ + Elasticsearch` 设计并实现本地生活内容交易平台，覆盖门店检索、内容互动、优惠券秒杀、订单支付、对账补偿等核心链路。
- 围绕高并发与一致性问题实现多级缓存、Lua 预扣库存、异步下单、消息补偿、热点治理与降级兜底，完成压测与瓶颈定位。
- 构建搜索索引同步、共同关注计算、支付对账和异常修复链路，补齐典型 CRUD 项目缺失的工程能力。

### AI 应用项目

- 基于 `LangGraph + MCP + Milvus + MySQL + Redis` 构建商家运营和平台客服 Agent，覆盖订单异常排查、券规则解释、商家知识问答、活动配置辅助等场景。
- 设计 Java MCP Server + Python Agent Service 架构，将订单、支付、券、门店、知识库等业务能力标准化为 Agent 工具，业务规则保留在 Java 主服务。
- 实现 ReAct + Self-Reflection 主循环、Tool Router、HITL 审批、权限感知 RAG 和 SSE 流式响应，构建工具调用准确率、Recall@5、任务完成率、事实一致性、延迟和成本评测体系。
