# 接口规范文档

## 1. 文档目标

本文档固定 `LocalLife` 后端所有模块的接口约定，覆盖：

- 统一响应结构
- HTTP 状态码
- 业务错误码
- 鉴权规范
- 分页规范
- 幂等规范
- 数据格式规范
- URL 路径规范

**编码原则**：所有模块的接口实现必须遵守本文档，新增约定在本文档追加，不允许各模块自定义响应结构。

---

## 2. 统一响应结构

### 2.1 响应包装器

所有接口统一使用 `Result<T>` 包装：

```json
{
  "code": "OK",
  "message": "操作成功",
  "data": { ... },
  "timestamp": "2026-05-26T20:00:00+08:00"
}
```

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `code` | String | 业务码。成功固定为 `OK`，失败为业务错误码字符串 |
| `message` | String | 面向用户的可读描述，失败时可直接展示 |
| `data` | T | 泛型，失败时为 `null` |
| `timestamp` | String | 服务端响应时间，ISO 8601，含时区 |

### 2.2 成功示例

```json
{
  "code": "OK",
  "message": "操作成功",
  "data": {
    "userId": "10001",
    "nickname": "小明"
  },
  "timestamp": "2026-05-26T20:00:00+08:00"
}
```

### 2.3 失败示例

```json
{
  "code": "COUPON_STOCK_EXHAUSTED",
  "message": "优惠券已抢完",
  "data": null,
  "timestamp": "2026-05-26T20:00:00+08:00"
}
```

### 2.4 为什么 code 用字符串而不是数字

- 数字码（如 `40009`）在排查问题时需要查手册才知道含义
- 字符串码（如 `COUPON_STOCK_EXHAUSTED`）一眼可读，on-call 时效率更高
- 面试时也更容易讲清楚：「这个错误码代表优惠券库存耗尽，返回 400」比「这个 40009 代表...」更流畅
- 缺点是字符串稍大，但在业务接口场景中可忽略不计

## 3. HTTP 状态码规范

### 3.1 使用的状态码

本项目只使用以下 6 个 HTTP 状态码，不引入其他状态码：

| 状态码 | 语义 | 典型场景 |
| --- | --- | --- |
| `200` | 成功 | 任何成功的请求 |
| `400` | 请求错误 | 参数校验失败、业务规则拒绝、资源不存在 |
| `401` | 未认证 | 未携带 Token 或 Token 已失效 |
| `403` | 无权限 | 有登录态但无权访问该资源 |
| `429` | 请求过频 | 触发限流（发验证码、发笔记、秒杀） |
| `500` | 服务错误 | 中间件异常、未捕获异常 |

### 3.2 为什么不使用 404 和 409

- **不用 404**：资源不存在统一走 `400 + SHOP_NOT_FOUND` 等业务码。理由：前端只需按 `code` 字段处理业务逻辑，不需要再分支判断 HTTP 状态码；混用 404 会让前端要同时处理两套错误机制。
- **不用 409**：并发冲突（如重复抢券）统一走 `400 + COUPON_ALREADY_RECEIVED`。理由：409 在不同团队的理解差异较大，业务码更清晰。

## 4. 业务错误码规范

### 4.1 命名规则

格式：`{模块前缀}_{错误描述}`，全大写，用下划线分隔。

| 前缀 | 归属模块 |
| --- | --- |
| `AUTH_` | 登录与鉴权 |
| `USER_` | 用户 |
| `SHOP_` | 门店 |
| `POST_` | 内容笔记 |
| `COUPON_` | 优惠券 |
| `SECKILL_` | 秒杀 |
| `ORDER_` | 订单 |
| `PAYMENT_` | 支付 |
| `SYS_` | 系统级（通用） |

### 4.2 第一批错误码清单

覆盖四条核心链路：登录、发笔记、抢券、下单支付。

| 错误码 | HTTP 状态码 | 说明 |
| --- | --- | --- |
| `AUTH_CODE_INVALID` | 400 | 验证码错误或已过期 |
| `AUTH_CODE_SEND_TOO_FREQUENT` | 429 | 验证码发送过于频繁（手机号或 IP 限流） |
| `AUTH_TOKEN_MISSING` | 401 | 请求未携带 Token |
| `AUTH_TOKEN_EXPIRED` | 401 | Token 已失效，需重新登录 |
| `USER_ACCOUNT_DISABLED` | 403 | 账号被禁用，无法访问 |
| `SHOP_NOT_FOUND` | 400 | 门店不存在 |
| `SHOP_NOT_ONLINE` | 400 | 门店未上线，暂不可操作 |
| `POST_PUBLISH_TOO_FREQUENT` | 429 | 发笔记过于频繁，触发用户维度限流 |
| `COUPON_STOCK_EXHAUSTED` | 400 | 优惠券库存已耗尽 |
| `COUPON_ALREADY_RECEIVED` | 400 | 已领取过该优惠券，不可重复领取 |
| `SECKILL_NOT_STARTED` | 400 | 秒杀场次尚未开始 |
| `SECKILL_EXPIRED` | 400 | 秒杀场次已结束 |
| `ORDER_NOT_FOUND` | 400 | 订单不存在 |
| `ORDER_STATUS_ILLEGAL` | 400 | 当前订单状态不允许此操作（如已支付不可取消） |
| `PAYMENT_VERIFY_FAILED` | 400 | 支付回调验签失败 |
| `PAYMENT_AMOUNT_MISMATCH` | 400 | 支付金额与订单金额不符 |
| `SYS_BUSY` | 500 | 系统繁忙（Redis、MQ 等中间件不可用时的兜底） |
| `SYS_PARAM_INVALID` | 400 | 通用参数校验失败（@Valid 等框架级校验） |

### 4.3 新增错误码规则

- 每个模块的错误码集中维护在该模块的枚举类中，如 `AuthErrorCode`、`OrderErrorCode`
- 新增错误码必须同步更新本文档
- 禁止在业务代码中直接写字符串 code，必须引用枚举

## 5. 鉴权规范

### 5.1 Token 方案

| 项 | 说明 |
| --- | --- |
| Token 格式 | 随机 UUID，无状态不可解析（有别于 JWT） |
| 存储位置 | Redis，Key: `login:token:{token}`，TTL: 7 天 |
| 传递方式 | HTTP Header: `Authorization: Bearer {token}` |
| 验证位置 | 鉴权拦截器（`AuthInterceptor`），在业务 Controller 前执行 |
| 验证成功 | 从 Redis 读取用户摘要，绑定到 `UserContext`（ThreadLocal） |
| 验证失败 | 返回 `401 + AUTH_TOKEN_EXPIRED` 或 `AUTH_TOKEN_MISSING` |

### 5.2 Redis 中存储的用户摘要

只存高频鉴权所需的最小字段，不存完整用户信息：

```json
{
  "userId": "10001",
  "mobile": "138****8000",
  "nickname": "用户昵称",
  "status": "ENABLED"
}
```

业务接口如需完整用户信息，从 MySQL 按需查询，不通过 Token 携带。

### 5.3 鉴权白名单（无需 Token 的接口）

| 路径 | 说明 |
| --- | --- |
| `POST /api/v1/auth/code` | 发送验证码，登录前操作 |
| `POST /api/v1/auth/login` | 登录，登录前操作 |
| `GET /api/v1/shops/{shopId}` | 门店详情，公开内容 |
| `GET /api/v1/posts/{postId}` | 笔记详情，公开内容 |
| `POST /api/v1/payments/callback` | 支付回调，由支付渠道调用，走渠道验签 |

### 5.4 Redis 不可用时的处理

- **强制拒绝请求，返回 `500 + SYS_BUSY`**
- 不允许在 Redis 不可用时自动降级放行
- 原因：登录态无法验证时放行等于绕过鉴权，是安全漏洞，安全优先于可用性

### 5.5 为什么 Token 不用 JWT

- Redis 方案可以随时主动踢出（账号封禁、检测到异常登录、用户主动退出）
- JWT 一旦签发在过期前无法作废，需要额外维护黑名单，复杂度反而更高
- 当前项目是单体 + 少量微服务，Redis 鉴权的网络开销可接受
- JWT 可作为后续增强项讨论，届时关注点是「无状态横向扩展」vs「主动踢出能力」的取舍

## 6. 分页规范

### 6.1 标准分页（列表查询）

适用场景：订单列表、我的优惠券、门店搜索结果等有总数的场景。

**请求参数（Query String）**

| 参数 | 类型 | 默认值 | 约束 | 说明 |
| --- | --- | --- | --- | --- |
| `pageNumber` | int | 1 | ≥ 1 | 页码，从 1 开始 |
| `pageSize` | int | 20 | 1 ~ 100 | 每页条数 |

**响应结构**

```json
{
  "code": "OK",
  "message": "操作成功",
  "data": {
    "total": 1000,
    "pageNumber": 1,
    "pageSize": 20,
    "items": [ ... ]
  },
  "timestamp": "2026-05-26T20:00:00+08:00"
}
```

### 6.2 游标分页（Feed 流）

适用场景：关注 Feed 流。Feed 流数据实时更新，传统分页会有「重复」和「漏条」问题，使用游标分页解决。

**请求参数（Query String）**

| 参数 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `lastId` | String | 不传 | 上一页最后一条记录的 ID，首页不传 |
| `pageSize` | int | 10 | 每批条数 |

**响应结构**

```json
{
  "code": "OK",
  "message": "操作成功",
  "data": {
    "hasMore": true,
    "lastId": "180000000000000050",
    "items": [ ... ]
  },
  "timestamp": "2026-05-26T20:00:00+08:00"
}
```

| 字段 | 说明 |
| --- | --- |
| `hasMore` | 是否还有下一页 |
| `lastId` | 当前页最后一条记录的 ID，传给下次请求的 `lastId` |

### 6.3 为什么分页从 1 开始

- 符合用户认知（没有人说「第 0 页」）
- 与 Spring Data Pageable 的 `one-indexed-parameters: true` 配置对齐
- 从 0 开始容易在前后端对接时出现 off-by-one 错误

## 7. 幂等规范

### 7.1 幂等 Header

创建类接口（可能被用户重复点击）支持客户端传入幂等键：

| Header | 类型 | 说明 |
| --- | --- | --- |
| `X-Idempotency-Key` | String (UUID) | 客户端生成的请求唯一标识，可选传入 |

**适用接口**

| 接口 | 说明 |
| --- | --- |
| `POST /api/v1/orders` | 下单，防止用户重复点击提交两笔订单 |
| `POST /api/v1/seckill` | 抢券，防止前端重复提交 |

### 7.2 服务端幂等方案

不同场景使用不同方案，选择最适合的而不是统一一种：

| 场景 | 方案 | 实现 |
| --- | --- | --- |
| 下单（客户端主动） | Redis SETNX | Key: `idempotent:order:{X-Idempotency-Key}`，TTL 5 分钟，成功后返回已有订单 ID |
| 支付回调（第三方） | DB 唯一索引 | `uk_payment_channel_trade_no(channel, channel_trade_no)`，重复插入忽略 |
| MQ 消费（异步事件） | eventId + 业务唯一键 | 先查 eventId 是否已消费，未消费则执行业务逻辑，结果与 eventId 一起落库 |
| 抢券（Redis Lua） | Redis Set | `SADD seckill:user:{sessionId}:{couponTemplateId} {userId}`，Lua 脚本原子判断 |

### 7.3 幂等实现原则

1. 幂等检查在业务逻辑之前执行
2. 重复请求返回和首次请求相同的结果（不报错，不重复执行）
3. 幂等 Key 的 TTL 需大于最长可能的重试窗口

## 8. 数据格式规范

### 8.1 时间格式

- 统一使用 **ISO 8601**，含时区，精确到秒
- 固定东八区：`2026-05-26T20:00:00+08:00`
- Spring Boot 配置：`spring.jackson.date-format=yyyy-MM-dd'T'HH:mm:ssXXX`
- 禁止使用时间戳（Long）或 `yyyy/MM/dd` 等非标准格式

### 8.2 金额格式

- 所有金额以**分**为单位，Integer 或 Long 类型
- 9.99 元 → `999`
- **禁止使用浮点数（float/double）传递金额**
- 原因：`9.99` 在浮点数中可能表示为 `9.989999...`，支付场景绝对不能出现精度问题

### 8.3 ID 格式

- 所有 ID（userId、shopId、orderId 等）以**字符串**形式传递，不传数字
- 正确：`"userId": "10001"`
- 错误：`"userId": 10001`
- 原因：雪花算法 ID 最大可达 2^63，超过 JavaScript `Number.MAX_SAFE_INTEGER`（2^53-1），JSON 传 Long 型数字会被前端静默截断，导致数据错误

### 8.4 布尔值

- 使用 `true` / `false`
- 禁止使用 `0` / `1` 或 `"Y"` / `"N"` 表示布尔值

### 8.5 空值

- 字段不存在时返回 `null`，不省略字段
- 禁止用空字符串 `""` 代替 `null`

## 9. URL 路径规范

### 9.1 路径设计原则

| 原则 | 示例 |
| --- | --- |
| 版本前缀 `/api/v1/` | `/api/v1/shops` |
| 资源名小写复数 | `/api/v1/shops`、`/api/v1/posts` |
| 层级表示从属关系 | `/api/v1/shops/{shopId}/posts` |
| 动作用 HTTP 方法表达，不用动词路径 | ✅ `DELETE /api/v1/follows/{userId}`，❌ `/api/v1/follows/cancel` |
| 路径参数用小驼峰 | `{shopId}`、`{postId}` |

### 9.2 HTTP 方法映射

| 方法 | 语义 | 幂等性 |
| --- | --- | --- |
| GET | 查询，不修改数据 | 幂等 |
| POST | 创建资源或触发动作 | 非幂等（创建）|
| PUT | 全量更新资源 | 幂等 |
| PATCH | 局部更新资源 | 非幂等（部分更新）|
| DELETE | 删除资源 | 幂等 |

## 10. 限流规范

### 10.1 限流实现

基于 Redis ZSet 滑动窗口，通过 `@RateLimit` 注解声明在 Controller 方法上，`RateLimitInterceptor` 拦截执行。

### 10.2 限流维度

| 维度 | 说明 | 适用接口 |
| --- | --- | --- |
| `USER`（用户维度） | 按登录用户 ID 限流，精准到人 | 需登录的写操作 |
| `IP`（IP 维度） | 按客户端 IP 限流，兼容代理场景（读 X-Forwarded-For） | 公开接口 |

### 10.3 各接口限流配置

| 接口 | 维度 | 限制 | 说明 |
| --- | --- | --- | --- |
| `POST /api/v1/auth/code` | IP | 1次/60s | 防短信轰炸 |
| `POST /api/v1/auth/login` | IP | 10次/60s | 防暴力破解 |
| `POST /api/v1/seckill` | USER | 1次/5s | 防机器刷券 |
| `POST /api/v1/posts` | USER | 3次/60s | 防内容刷屏 |
| `POST /api/v1/posts/.../comments` | USER | 5次/10s | 防评论刷屏 |
| `POST /api/v1/orders` | USER | 2次/10s | 防重复下单 |
| `GET /api/v1/search/shops` | IP | 10次/1s | 保护 ES |
| `GET /api/v1/search/posts` | IP | 10次/1s | 保护 ES |

响应 Body（与 `Result<T>` 格式一致）：

  "message": "请求过于频繁，请稍后再试",
  "data": null,
  "timestamp": "2026-05-28T20:00:00+08:00"
}

## 11. 面试深挖点汇总

编写代码时每个决策都应能回答「为什么」，以下是本规范的主要决策点：

| 决策 | 问题 | 回答要点 |
| --- | --- | --- |
| code 用字符串 | 为什么不用数字错误码？ | 字符串码一眼可读；数字码在 on-call 时要查手册；字符串码方便面试表达 |
| 金额用分 | 为什么不用元？ | 浮点数精度问题，9.99 元 float 可能是 9.989999...，支付场景不允许精度误差 |
| ID 用字符串 | 为什么不直接传 Long？ | JS Number 最大安全整数约 9 千万亿，雪花 ID 可达 2^63，JSON 传 Long 会被前端截断 |
| Token 不用 JWT | 为什么选 Redis Token？ | Redis 方案支持主动踢出（封号、异常登录），JWT 签发后无法作废；当前单体场景无需 JWT 的无状态优势 |
| Redis 不可用时拒绝 | 为什么不降级放行？ | 无法验证登录态时放行等于绕过鉴权，是安全漏洞，安全优先于可用性 |
| 不用 404/409 | 为什么都走 400？ | 让前端只需处理 `code` 字段，无需再分支判断 HTTP 状态码；减少前后端约定的歧义 |
| 分页从 1 开始 | 为什么不从 0 开始？ | 符合用户认知；对齐 Spring Data 配置；从 0 开始容易 off-by-one |
| Feed 游标分页 | 为什么不用传统分页？ | Feed 数据实时更新，传统分页会出现同条记录重复展示或漏条问题；游标分页按 ID 定位，稳定 |
| 幂等 Key 用 UUID | 为什么用客户端生成而不是服务端？ | 客户端重试时携带同一 Key，服务端才能识别是同一次请求的重试，而不是新请求 |
