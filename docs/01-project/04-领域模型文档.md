# 领域模型文档

## 1. 文档目标

这份文档解决五个问题：

1. 项目拆成哪些领域
2. 每个领域里有哪些核心实体
3. 实体之间是什么关系
4. 哪些对象是聚合根
5. 哪些状态流转必须提前固定

边界文档定义“项目做什么”。  
领域模型文档定义“项目里面有什么，以及这些东西如何组织”。

## 2. 建模原则

当前阶段的建模遵循 4 条原则：

1. 先围绕业务闭环建模，不围绕表建模
2. 先固定主链路对象，不一次性铺满所有扩展对象
3. 先明确聚合边界，再考虑缓存、MQ、搜索
4. 先让语言稳定，再进入代码和 ER 图

## 3. 一级领域划分

当前项目拆成 6 个一级领域：

1. 用户域
2. 商家门店域
3. 内容域
4. 社交域
5. 营销域
6. 交易域

这 6 个领域已经足够覆盖第一阶段和主后端项目的大部分核心链路。

## 4. 各领域职责

### 4.1 用户域

负责平台用户身份、登录状态和基础资料。

关注问题：

- 用户是谁
- 用户如何登录
- 用户是否可访问某些功能
- 用户的基本身份数据如何被其他领域引用

### 4.2 商家门店域

负责商家主体和门店实体。

关注问题：

- 商家是谁
- 商家下面有哪些门店
- 门店对外展示哪些基础信息
- 门店和后续内容、活动、交易如何建立关联

### 4.3 内容域

负责探店笔记、评论和内容互动。

关注问题：

- 用户发了什么内容
- 内容属于哪个门店
- 内容如何被评论
- 内容的展示状态和审核状态如何变化

### 4.4 社交域

负责关注关系和共同关注。

关注问题：

- 用户关注了谁
- 关注关系如何表达
- 共同关注如何计算
- 后续 Feed 能力如何扩展

### 4.5 营销域

负责券、活动和秒杀配置。

关注问题：

- 优惠券是什么
- 活动如何配置
- 用户是否有资格参与
- 库存如何被预扣减

### 4.6 交易域

负责订单和支付基础链路。

关注问题：

- 订单怎么创建
- 订单如何流转
- 支付结果如何回写
- 对账和补偿围绕哪些对象展开

## 5. 核心实体清单

第一阶段先固定 10 个核心实体。

### 5.1 User

定义：平台消费者身份主体。

职责：

- 持有登录身份
- 发内容
- 评论
- 关注他人
- 参与领券和下单

关键属性：

- userId
- mobile
- nickname
- avatar
- status
- registerTime

### 5.2 Merchant

定义：平台商家主体。

职责：

- 持有商家身份
- 拥有多个门店
- 配置活动
- 查看经营数据

关键属性：

- merchantId
- merchantName
- contactName
- contactPhone
- status

### 5.3 Shop

定义：面向用户展示和交易承接的门店实体。

职责：

- 对外展示门店信息
- 承接内容和活动
- 承接订单
- 被搜索和推荐

关键属性：

- shopId
- merchantId
- shopName
- categoryId
- address
- longitude
- latitude
- businessStatus

### 5.4 Post

定义：用户发布的探店内容。

职责：

- 形成内容资产
- 与门店关联
- 提供评论和互动入口
- 为 AI 总结和搜索提供素材

关键属性：

- postId
- authorUserId
- shopId
- title
- content
- status
- publishTime

### 5.5 Comment

定义：用户对内容的评论。

职责：

- 表达反馈
- 形成讨论
- 为内容总结提供输入

关键属性：

- commentId
- postId
- userId
- content
- status
- createTime

### 5.6 FollowRelation

定义：用户与用户、用户与商家之间的关注关系。

职责：

- 维护社交连接
- 支撑共同关注计算
- 支撑后续 Feed 能力

关键属性：

- relationId
- followerUserId
- targetType
- targetId
- createTime

### 5.7 CouponTemplate

定义：商家或平台配置的优惠券模板。

职责：

- 定义券规则
- 定义库存与限领逻辑
- 作为秒杀或普通领券的基础对象

关键属性：

- couponTemplateId
- shopId
- couponName
- totalStock
- receiveLimit
- startTime
- endTime
- status

### 5.8 SeckillSession

定义：具备强时效和高并发特征的抢券活动实体。

职责：

- 管理秒杀活动时间窗
- 管理活动库存入口
- 为 Redis 预扣减和异步下单提供业务归属

关键属性：

- seckillSessionId
- couponTemplateId
- beginTime
- endTime
- sessionStatus

### 5.9 Order

定义：用户交易的主业务单据。

职责：

- 承接下单行为
- 持有订单生命周期
- 对接支付和补偿

关键属性：

- orderId
- userId
- shopId
- orderAmount
- orderStatus
- createTime
- expireTime

### 5.10 PaymentOrder

定义：围绕订单支付行为生成的支付单据。

职责：

- 记录支付请求
- 接收支付结果
- 回写订单状态
- 支撑对账

关键属性：

- paymentOrderId
- orderId
- payAmount
- payStatus
- channel
- paidTime

## 6. 实体关系

### 6.1 用户域相关关系

- 一个 `User` 可以发布多个 `Post`
- 一个 `User` 可以发表多个 `Comment`
- 一个 `User` 可以发起多条 `FollowRelation`
- 一个 `User` 可以创建多个 `Order`

### 6.2 商家门店域相关关系

- 一个 `Merchant` 拥有多个 `Shop`
- 一个 `Shop` 可以绑定多个 `CouponTemplate`
- 一个 `Shop` 可以被多个 `Post` 关联
- 一个 `Shop` 可以产生多个 `Order`

### 6.3 内容域相关关系

- 一个 `Post` 属于一个 `User`
- 一个 `Post` 可关联一个 `Shop`
- 一个 `Post` 下有多个 `Comment`

### 6.4 社交域相关关系

- 一条 `FollowRelation` 由一个 `User` 发起
- `FollowRelation.targetType` 决定目标是用户还是商家

### 6.5 营销域相关关系

- 一个 `CouponTemplate` 可属于一个 `Shop`
- 一个 `CouponTemplate` 可被一个或多个 `SeckillSession` 使用

### 6.6 交易域相关关系

- 一个 `Order` 属于一个 `User`
- 一个 `Order` 对应一个 `Shop`
- 一个 `Order` 可关联一个优惠活动上下文
- 一个 `Order` 对应一个或多个 `PaymentOrder`

## 7. 聚合划分

当前第一阶段建议固定 6 个聚合。

### 7.1 User 聚合

聚合根：`User`

原因：

- 用户身份是其他行为的发起基础
- 登录、状态控制和资料维护都围绕用户进行

### 7.2 Merchant-Shop 聚合

聚合根：`Merchant`

子实体：`Shop`

原因：

- 商家与门店是明显的拥有关系
- 管理后台视角下，门店归属于商家

注意：

- 搜索和内容展示层更多以 `Shop` 为直接查询对象
- 但管理边界上仍然归商家聚合控制

### 7.3 Post 聚合

聚合根：`Post`

子实体：`Comment`

原因：

- 内容是一级展示对象
- 评论围绕内容存在
- 内容的状态流转决定评论可见性和后续 AI 总结素材范围

### 7.4 Follow 聚合

聚合根：`FollowRelation`

原因：

- 关注本身就是最小独立业务事实
- 共同关注属于衍生计算能力，不是独立聚合根

### 7.5 Coupon-Marketing 聚合

聚合根：`CouponTemplate`

子实体：`SeckillSession`

原因：

- 秒杀活动本质上是券模板在特定时间窗下的并发发放场景
- 活动规则围绕券模板定义

### 7.6 Order 聚合

聚合根：`Order`

子实体：`PaymentOrder`

原因：

- 订单是交易主事实
- 支付单围绕订单存在
- 对账和补偿都以订单为主中心展开

## 8. 核心状态流

第一阶段先固定 5 类状态流。

### 8.1 User 状态

- ENABLED
- DISABLED

### 8.2 Shop 状态

- DRAFT
- ONLINE
- OFFLINE
- CLOSED

### 8.3 Post 状态

- DRAFT
- PENDING_REVIEW
- PUBLISHED
- REJECTED
- HIDDEN

### 8.4 CouponTemplate / SeckillSession 状态

- NOT_STARTED
- RUNNING
- EXPIRED
- CLOSED

### 8.5 Order / PaymentOrder 状态

Order：

- CREATED
- WAIT_PAY
- PAID
- CANCELED
- REFUNDING
- REFUNDED

PaymentOrder：

- INIT
- PAYING
- SUCCESS
- FAILED
- CLOSED

## 9. 领域间依赖方向

依赖方向先固定为：

- 用户域为基础域
- 商家门店域为基础域
- 内容域依赖用户域和门店域
- 社交域依赖用户域
- 营销域依赖门店域
- 交易域依赖用户域、门店域、营销域

这样可以避免早期循环依赖。

## 10. 对后续技术设计的影响

这份领域模型会直接影响后续几个设计动作：

### 10.1 对 ER 图

- 决定建哪些表
- 决定主外键关系
- 决定哪些表先出

### 10.2 对缓存设计

- `Shop`、`CouponTemplate`、`Order` 会成为重点缓存对象
- `FollowRelation` 会进入 Redis Set 计算体系

### 10.3 对 MQ 设计

- 订单创建、支付成功、券库存变化、索引同步会围绕聚合根发事件

### 10.4 对搜索设计

- `Shop` 和 `Post` 是最先进入 Elasticsearch 的两个主对象

### 10.5 对 AI 工具设计

- `shop_search`
- `order_exception_query`
- `coupon_policy_lookup`
- `merchant_knowledge_search`

这些工具背后都依赖当前领域模型是否清晰。

## 11. 当前阶段不展开的对象

当前先不展开下面这些对象：

- Like
- Favorite
- Report
- Category
- Tag
- Cart
- RefundOrder
- InventoryLedger

这些对象未来大概率会有，但现在展开只会稀释主线。
