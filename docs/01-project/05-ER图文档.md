# ER 图文档

## 1. 文档目标

这份文档解决 5 个问题：

1. 第一阶段到底建哪些核心表
2. 每张表的主键是什么
3. 表和表之间怎么关联
4. 哪些字段需要唯一约束
5. 哪些字段需要优先建索引

这一步不是直接写 SQL。  
这一步是先把数据库结构语言固定下来。

## 2. 建模范围

当前只覆盖第一阶段 10 个核心实体：

1. `user`
2. `merchant`
3. `shop`
4. `post`
5. `comment`
6. `follow_relation`
7. `coupon_template`
8. `seckill_session`
9. `order_info`
10. `payment_order`

说明：

- 订单表命名使用 `order_info`
- 原因是 `order` 在多种 SQL 环境下容易与关键字冲突

## 3. 命名规则

统一规则如下：

- 表名：小写下划线
- 主键：`id`
- 外键：`xxx_id`
- 时间字段：`create_time`、`update_time`
- 状态字段：`status`
- 逻辑删除字段先不引入

## 4. 核心表设计

### 4.1 user

用途：

- 平台用户基础身份
- 登录主体
- 内容、关注、订单等行为的发起方

核心字段：

- `id`
- `mobile`
- `nickname`
- `avatar`
- `status`
- `register_time`
- `create_time`
- `update_time`

约束：

- 主键：`id`
- 唯一：`mobile`

索引候选：

- `uk_mobile(mobile)`
- `idx_status(status)`

### 4.2 merchant

用途：

- 商家主体
- 门店拥有者

核心字段：

- `id`
- `merchant_name`
- `contact_name`
- `contact_phone`
- `status`
- `create_time`
- `update_time`

约束：

- 主键：`id`

索引候选：

- `idx_status(status)`
- `idx_contact_phone(contact_phone)`

### 4.3 shop

用途：

- 门店核心实体
- 内容、活动、订单的挂载对象
- 搜索主对象之一

核心字段：

- `id`
- `merchant_id`
- `shop_name`
- `category_id`
- `address`
- `longitude`
- `latitude`
- `business_status`
- `create_time`
- `update_time`

约束：

- 主键：`id`
- 外键逻辑关系：`merchant_id -> merchant.id`

索引候选：

- `idx_merchant_id(merchant_id)`
- `idx_category_id(category_id)`
- `idx_business_status(business_status)`
- 地理位置索引后续交给 Elasticsearch，不在 MySQL 做主能力

### 4.4 post

用途：

- 探店内容主体
- 内容域核心对象

核心字段：

- `id`
- `author_user_id`
- `shop_id`
- `title`
- `content`
- `status`
- `publish_time`
- `create_time`
- `update_time`

约束：

- 主键：`id`
- 外键逻辑关系：
  - `author_user_id -> user.id`
  - `shop_id -> shop.id`

索引候选：

- `idx_author_user_id(author_user_id)`
- `idx_shop_id(shop_id)`
- `idx_status_publish_time(status, publish_time)`

说明：

- `content` 作为大字段，不进入普通索引
- 内容搜索主能力后续交给 Elasticsearch

### 4.5 comment

用途：

- 内容评论
- 内容总结和互动输入

核心字段：

- `id`
- `post_id`
- `user_id`
- `content`
- `status`
- `create_time`
- `update_time`

约束：

- 主键：`id`
- 外键逻辑关系：
  - `post_id -> post.id`
  - `user_id -> user.id`

索引候选：

- `idx_post_id(post_id)`
- `idx_user_id(user_id)`
- `idx_post_id_create_time(post_id, create_time)`

### 4.6 follow_relation

用途：

- 关注关系存储
- 共同关注计算基础数据

核心字段：

- `id`
- `follower_user_id`
- `target_type`
- `target_id`
- `create_time`

约束：

- 主键：`id`
- 唯一：`(follower_user_id, target_type, target_id)`

索引候选：

- `uk_follower_target(follower_user_id, target_type, target_id)`
- `idx_target(target_type, target_id)`

说明：

- `target_type` 用来区分关注的是用户还是商家
- 这是一个多态关联表

### 4.7 coupon_template

用途：

- 优惠券模板
- 活动规则载体

核心字段：

- `id`
- `shop_id`
- `coupon_name`
- `total_stock`
- `receive_limit`
- `start_time`
- `end_time`
- `status`
- `create_time`
- `update_time`

约束：

- 主键：`id`
- 外键逻辑关系：`shop_id -> shop.id`

索引候选：

- `idx_shop_id(shop_id)`
- `idx_status_time(status, start_time, end_time)`

### 4.8 seckill_session

用途：

- 秒杀活动时间窗
- 高并发抢券活动主对象

核心字段：

- `id`
- `coupon_template_id`
- `begin_time`
- `end_time`
- `session_status`
- `create_time`
- `update_time`

约束：

- 主键：`id`
- 外键逻辑关系：`coupon_template_id -> coupon_template.id`

索引候选：

- `idx_coupon_template_id(coupon_template_id)`
- `idx_session_status_time(session_status, begin_time, end_time)`

### 4.9 order_info

用途：

- 交易主单
- 订单生命周期主体

核心字段：

- `id`
- `order_no`
- `user_id`
- `shop_id`
- `coupon_template_id`
- `order_amount`
- `order_status`
- `create_time`
- `expire_time`
- `pay_time`
- `update_time`

约束：

- 主键：`id`
- 唯一：`order_no`
- 外键逻辑关系：
  - `user_id -> user.id`
  - `shop_id -> shop.id`
  - `coupon_template_id -> coupon_template.id`

索引候选：

- `uk_order_no(order_no)`
- `idx_user_id(user_id)`
- `idx_shop_id(shop_id)`
- `idx_order_status(order_status)`
- `idx_user_status_create_time(user_id, order_status, create_time)`

说明：

- 券关联先直接挂 `coupon_template_id`
- 后续如果需要更细的订单项和营销明细，再扩展子表

### 4.10 payment_order

用途：

- 支付过程单据
- 支付回调和对账对象

核心字段：

- `id`
- `payment_no`
- `order_id`
- `pay_amount`
- `pay_status`
- `channel`
- `paid_time`
- `create_time`
- `update_time`

约束：

- 主键：`id`
- 唯一：`payment_no`
- 外键逻辑关系：`order_id -> order_info.id`

索引候选：

- `uk_payment_no(payment_no)`
- `idx_order_id(order_id)`
- `idx_pay_status(pay_status)`
- `idx_channel_paid_time(channel, paid_time)`

## 5. 表关系总览

### 5.1 一对多关系

- `merchant -> shop`
- `user -> post`
- `shop -> post`
- `post -> comment`
- `user -> comment`
- `shop -> coupon_template`
- `coupon_template -> seckill_session`
- `user -> order_info`
- `shop -> order_info`
- `order_info -> payment_order`

### 5.2 多态关系

- `follow_relation.target_type + target_id`

说明：

- 关注目标可以是用户
- 也可以是商家

### 5.3 弱关联关系

- `order_info -> coupon_template`

说明：

- 订单可以不一定使用券
- 所以 `coupon_template_id` 允许为空

## 6. 第一阶段建议字段规范

所有核心表建议统一包含：

- `id`
- `create_time`
- `update_time`

部分表增加：

- `status`

时间字段建议：

- 使用 `datetime` 或 `timestamp`
- 统一由服务端写入

主键建议：

- 第一阶段先用 `bigint`
- 生成策略后续统一决定，可以用雪花算法或数据库外部分配

## 7. 第一阶段先建表顺序

先建表顺序固定如下：

1. `user`
2. `merchant`
3. `shop`
4. `post`
5. `comment`
6. `follow_relation`
7. `coupon_template`
8. `seckill_session`
9. `order_info`
10. `payment_order`

原因：

- 先建基础主实体
- 再建内容与社交
- 再建营销
- 最后建交易

## 8. 当前阶段不进入的表

当前不进入下面这些扩展表：

- `post_like`
- `post_favorite`
- `post_report`
- `shop_category`
- `shop_tag_relation`
- `order_item`
- `refund_order`
- `inventory_log`
- `settlement_bill`

原因：

- 当前阶段目标是稳定主链路
- 这些扩展表会增加复杂度
- 等主链路跑通后再扩

## 9. ER 图文本版

```text
merchant (1) ------ (n) shop
user     (1) ------ (n) post
shop     (1) ------ (n) post
post     (1) ------ (n) comment
user     (1) ------ (n) comment
user     (1) ------ (n) follow_relation
shop     (1) ------ (n) coupon_template
coupon_template (1) ---- (n) seckill_session
user     (1) ------ (n) order_info
shop     (1) ------ (n) order_info
coupon_template (0..1) - (n) order_info
order_info (1) ---- (n) payment_order
```

## 10. 对后续代码结构的影响

这份 ER 图文档会直接影响后续代码目录设计：

- `user` 对应用户模块
- `merchant`、`shop` 对应商家门店模块
- `post`、`comment` 对应内容模块
- `follow_relation` 对应社交模块
- `coupon_template`、`seckill_session` 对应营销模块
- `order_info`、`payment_order` 对应交易模块

后续包结构、Mapper、Repository 都围绕这组表稳定展开。
