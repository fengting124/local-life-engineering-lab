# 代码级实现地图

本文按“业务问题 → 接口 → 代码入口 → 数据/中间件 → 测试 → 面试考点”组织，目的是让你能从文档一路追到代码。

## 1. 总体服务边界

| 服务 | 端口 | 职责 | 代码入口 |
| --- | --- | --- | --- |
| LocalLife Server | 8080 | C 端/B 端核心业务：用户、门店、笔记、券、秒杀、订单、支付、搜索 | `local-life-server/src/main/java/.../server` |
| LocalLife Copilot MCP | 8081 | 把业务能力封装成 MCP 工具，做 RBAC、限流、审计 | `local-life-copilot/src/main/java/.../copilot` |
| Copilot Agent Service | 8000 | 对话、Fast Path、ReAct、RAG、HITL、会话持久化 | `copilot-agent-service` |
| Embedding/Reranker | 8100/8101 | RAG 向量化与重排 | `copilot-agent-service/model_services` |

核心原则：

- Java 主服务是业务事实来源。
- Java MCP 是 Agent 的工具网关。
- Python Agent 是编排层，不直接改业务库。

面试常问：

- 为什么拆成三个服务？
  因为业务后端、工具网关、LLM 编排的发布节奏、权限边界和故障模式不同。Java 主服务可以不依赖 Agent 独立提供业务；Agent 挂了不影响下单支付；MCP 统一做工具权限和审计。

## 2. 读代码的统一方法

每条链路都按这个顺序读：

1. Swagger 找接口。
2. Controller 看请求参数、鉴权、返回 VO。
3. Service 看事务、幂等、缓存、MQ、业务状态机。
4. Mapper/Entity 看表结构和索引。
5. resources 看 SQL、Lua、ES mapping、sharding 配置。
6. tests 看系统认为哪些行为最值得守住。

常用资源文件：

| 文件 | 用途 |
| --- | --- |
| `local-life-server/src/main/resources/db/migration/*.sql` | 主业务表结构 |
| `local-life-copilot/src/main/resources/db/migration/V1__init_copilot_tables.sql` | Agent/MCP 审计、会话、HITL 表 |
| `local-life-server/src/main/resources/lua/seckill.lua` | 秒杀原子扣减 |
| `local-life-server/src/main/resources/lua/rate_limit.lua` | Redis 滑动窗口限流 |
| `local-life-server/src/main/resources/sharding.yaml` | 本地订单分片配置 |
| `local-life-server/src/main/resources/sharding-docker.yaml` | Docker 内订单分片配置 |

## 3. 登录与鉴权

业务目标：用户用手机号验证码登录，后续请求通过 token 识别用户。

| 层 | 代码 |
| --- | --- |
| 接口 | `AuthController`: `POST /api/v1/auth/code`, `POST /api/v1/auth/login`, `POST /api/v1/auth/logout` |
| 业务 | `AuthService` |
| 鉴权 | `AuthInterceptor`, `UserContext`, `WebMvcConfig` |
| 限流 | `RateLimit`, `RateLimitInterceptor`, `lua/rate_limit.lua` |
| 数据 | `User`, `UserMapper`, `V1__init_user.sql` |
| 测试 | `AuthInterceptorTest` |

关键实现：

- 验证码放 Redis，登录成功后 token 对应用户摘要也放 Redis。
- `AuthInterceptor` 从请求头取 token，查 Redis 后写入 `UserContext`。
- `UserContext` 基于 ThreadLocal，业务代码不用层层传 userId。
- 公共接口通过白名单放行，写接口和个人数据接口必须登录。

面试常问：

- 为什么不用 JWT？
  这个项目选 Redis token，方便服务端主动失效、踢人、控制过期和存用户摘要。JWT 优点是无状态，但一旦签发，撤销和权限变更更麻烦。

- ThreadLocal 有什么风险？
  线程复用会导致脏上下文，所以拦截器必须在 finally 中清理。异步线程不会自动继承 ThreadLocal，MCP 审计异步写入时需要显式捕获上下文快照。

- 限流为什么用 Lua？
  滑动窗口要读旧请求、删过期、判断数量、写入本次请求，多步操作必须原子化；Lua 在 Redis 单线程里执行，避免并发穿透。

## 4. 门店、缓存与搜索

业务目标：用户高频浏览门店，商家维护门店，搜索支持关键字和地理位置。

| 层 | 代码 |
| --- | --- |
| 接口 | `ShopController`: `GET /api/v1/shops`, `GET /api/v1/shops/{shopId}`, `POST /api/v1/shops` |
| 业务 | `ShopService`, `ShopCacheService` |
| 防穿透 | `BloomFilterService` |
| 搜索 | `SearchController`, `ShopSearchService`, `ShopSearchRepository`, `ShopDocument` |
| 缓存失效 | `ShopCacheInvalidationListener` |
| 数据 | `Shop`, `Merchant`, `ShopMapper`, `MerchantMapper`, `V2__init_merchant_shop.sql` |
| 测试 | `ShopServiceTest` |

关键实现：

- 读门店详情走 Cache Aside：先查本地/Redis 缓存，miss 后查 DB，再回填缓存。
- BloomFilter 用来拦截明显不存在的 shopId，降低缓存穿透。
- 门店搜索走 ES，业务事实仍以 MySQL 为准。
- 门店更新后要处理缓存失效，避免旧数据长期停留。

面试常问：

- 缓存一致性为什么常用“删除缓存”而不是“更新缓存”？
  更新缓存容易把并发事务中的旧值写回缓存；删除缓存让下一次读从 DB 重建，语义更简单。代价是短暂 cache miss。

- BloomFilter 有什么缺点？
  可能误判存在，但不会误判不存在。误判存在时最多多查一次 DB，所以适合防穿透，不适合做最终权限判断。

- ES 和 MySQL 谁是准的？
  MySQL 是事实来源，ES 是检索索引。ES 允许短暂延迟，不能直接作为交易事实。

## 5. 内容、点赞、评论和关注

业务目标：用户发笔记、浏览门店笔记、点赞评论、查看共同关注。

| 层 | 代码 |
| --- | --- |
| 笔记接口 | `PostController`: `POST /api/v1/posts`, `GET /api/v1/posts/{postId}`, `GET /api/v1/shops/{shopId}/posts` |
| 互动接口 | `POST /api/v1/posts/{postId}/likes`, 评论相关接口 |
| 关注接口 | `FollowController` |
| 业务 | `PostService`, `FollowService` |
| 搜索 | `PostSearchService`, `PostSearchRepository`, `PostDocument` |
| 数据 | `Post`, `Comment`, `FollowRelation`, `PostMapper`, `CommentMapper`, `FollowRelationMapper`, `V3__init_post_comment_follow.sql` |

关键实现：

- 点赞状态和计数适合放 Redis，DB 保存最终事实或快照。
- 共同关注适合用 Redis Set/ZSet 做集合运算，避免应用层拉全量列表。
- Feed/列表类接口要重点关注分页方式，避免深分页。

面试常问：

- 为什么点赞不用每次都直接 update DB？
  高频互动直接打 DB 会放大写压力，Redis 更适合承接热点计数和状态，DB 做最终持久化或异步同步。

- 共同关注为什么适合集合结构？
  共同关注本质是两个用户关注集合求交集，Redis 集合运算比把列表查回应用层再求交更直接。

## 6. 优惠券和秒杀

业务目标：用户抢券，系统不能超卖，不能重复领取。

| 层 | 代码 |
| --- | --- |
| 接口 | `SeckillController`: `GET /api/v1/coupons/templates`, `POST /api/v1/seckill`, `GET /api/v1/seckill/result`, `GET /api/v1/coupons` |
| 业务 | `CouponService`, `SeckillService` |
| 原子脚本 | `lua/seckill.lua` |
| MQ | `SeckillSuccessEvent`, `SeckillSuccessConsumer` |
| 数据 | `CouponTemplate`, `SeckillSession`, `UserCoupon`, `CouponTemplateMapper`, `SeckillSessionMapper`, `UserCouponMapper`, `V4__init_coupon_seckill.sql` |
| 测试 | `CouponServiceTest`, `SeckillServiceTest`, `SeckillConcurrencyIT`, `UserCouponRepositoryIT` |

关键实现：

- Redis Lua 同时判断库存和用户是否已抢，成功后扣库存并记录用户。
- DB 层用唯一索引兜底，防止 Redis 或异步链路异常导致重复领取。
- 秒杀成功后可以通过 MQ 异步落库，降低接口同步耗时。
- 查询结果接口用于把“抢券请求已接收”和“最终落库成功”解耦。

面试常问：

- Redis 扣库存成功但 DB 落库失败怎么办？
  需要异步重试、失败补偿或对账任务。Redis 是前置削峰和原子准入，DB 才是最终事实。

- 怎么证明没有超卖？
  看 Redis Lua 原子性、DB 唯一索引和并发集成测试。压测时验收口径是成功领取数不能超过库存。

- 为什么不能只靠 synchronized？
  synchronized 只管单 JVM，多实例部署时失效；Redis/DB 这种集中式状态才能跨实例生效。

## 7. 订单、支付和 Outbox

业务目标：用户下单、支付、回调、自动关单，支付成功事件不能丢。

| 层 | 代码 |
| --- | --- |
| 订单接口 | `OrderController`: `POST /api/v1/orders`, `GET /api/v1/orders`, `GET /api/v1/orders/{orderId}` |
| 支付接口 | `PaymentController`: `POST /api/v1/payments`, `POST /api/v1/payments/callback`, `GET /api/v1/payments/mock-pay` |
| 业务 | `OrderService`, `PaymentService` |
| MQ/Outbox | `OutboxService`, `PaymentSuccessConsumer`, `OrderCloseConsumer`, `MqTopics` |
| 分表 | `sharding.yaml`, `sharding-docker.yaml`, `V8__init_order_sharding.sql` |
| 数据 | `OrderInfo`, `PaymentOrder`, `OutboxMessage`, `OrderInfoMapper`, `PaymentOrderMapper`, `OutboxMessageMapper`, `V5__init_order_payment.sql`, `V7__init_outbox_message.sql`, `V9__outbox_auto_recovery.sql` |
| 测试 | `OrderServiceTest`, `PaymentServiceTest` |

关键实现：

- 下单需要幂等，避免用户重复点击产生多笔订单。
- 订单和支付单分离，一笔订单可以有多次支付尝试，支付历史可追踪。
- 支付回调必须幂等，因为第三方支付平台可能重试通知。
- 支付成功事件写 Outbox，再由 Relay/消费者投递 MQ，避免“DB 成功但 MQ 发送失败”。
- 订单按 `user_id % 4` 分片，用户查自己的订单可以命中单分片。

面试常问：

- Outbox 解决什么问题？
  解决本地事务和 MQ 发送之间的不一致。业务变更和 outbox 消息在同一个 DB 事务里提交，后续异步投递失败可以重试。

- 为什么不用分布式事务/2PC？
  2PC 成本高、耦合强、可用性差。交易系统里很多事件通知可以接受最终一致，Outbox 更适合。

- 支付回调为什么不能只看请求来更新订单？
  必须校验幂等、金额、订单状态和支付状态，否则重复回调、伪造回调、乱序回调都会造成资损。

- 为什么按 user_id 分片？
  用户订单列表是高频查询，按 user_id 分片能单分片命中；按 order_id 分片虽然均匀，但用户订单列表要广播查询。

## 8. Java MCP 工具服务

业务目标：让 Agent 能安全查询和执行业务能力，同时保留权限、审计和限流。

| 层 | 代码 |
| --- | --- |
| 入口 | `McpController`: `POST /mcp` |
| 协议 DTO | `McpRequest`, `McpResponse`, `McpError`, `ToolDefinition` |
| 工具注册 | `ToolRegistry`, `McpTool` |
| 权限 | `RbacFilter`, `RbacContext` |
| 限流 | `ToolRateLimiter` |
| 审计 | `ToolAuditService`, `ToolAuditLog`, `ToolAuditMapper` |
| 业务工具 | `QueryOrderTool`, `QueryPaymentTool`, `ShopMetricsQueryTool`, `CouponPolicyLookupTool`, `QueryCouponIssueLogTool`, `QueryMqDeadLetterTool`, `ExecuteRefundTool`, `IssueCompensationCouponTool`, `CampaignDraftGenerateTool` |
| 查询 | `CopilotOrderMapper`, `CopilotCouponMapper` |
| 内部写操作 | `LocalLifeInternalClient`, `InternalController`, `InternalService` |
| 测试 | `McpControllerTest`, `RbacFilterTest`, `ToolAuditServiceTest`, `QueryOrderToolTest` |

关键实现：

- MCP 请求是 JSON-RPC 2.0，`tools/list` 返回工具 schema，`tools/call` 执行工具。
- `RbacFilter` 从 Header 构造身份上下文，工具执行时按角色校验。
- L1 查询工具可自动执行，高风险写工具必须配合 HITL 审批。
- 每次工具调用都写审计日志，便于复盘 Agent 做过什么。

当前注意点：

- `query_order` 的 schema 参数名叫 `order_id`，但当前实现按业务订单号 `order_no` 查询。演示时传 `BULK2026061000000000` 这种订单号。

面试常问：

- MCP 和直接 REST 调用有什么区别？
  REST 是给人类/前端业务流程用的；MCP 工具给 Agent 用，必须有工具描述、结构化错误、权限边界、审计和可恢复语义。

- 为什么工具错误要结构化？
  LLM 需要根据 `parameter_error`、`permission_denied`、`business_conflict` 等错误类型决定下一步，而不是解析一段自然语言报错。

- 为什么商家查不到别人的订单时返回 not_found 而不是 permission_denied？
  防枚举。否则攻击者可以通过错误类型判断订单是否存在。

## 9. Python Agent 服务

业务目标：把用户问题转成可控的工具调用，必要时检索知识库或请求人工审批。

| 层 | 代码 |
| --- | --- |
| 应用入口 | `main.py` |
| 对话接口 | `api/chat.py`: `POST /chat`, `POST /chat/resume` |
| 会话接口 | `api/session.py` |
| 审批接口 | `api/hitl.py` |
| Fast Path | `api/chat.py` 的 `_try_fast_path` |
| ReAct 图 | `agent/graph.py`, `agent/nodes.py`, `agent/state.py` |
| 工具选择 | `agent/tool_router.py` |
| MCP 客户端 | `mcp/mcp_client.py` |
| RAG | `rag/pipeline.py`, `rag/vector_store.py`, `rag/bm25_store.py`, `rag/embedding_client.py`, `rag/reranker_client.py` |
| Guardrails | `guardrails/input_checker.py` |
| 状态持久化 | `session/manager.py`, `session/checkpointer.py`, `session/hitl.py`, `session/models.py` |
| 模型服务 | `model_services/embedding_service/app.py`, `model_services/reranker_service/app.py` |
| 测试 | `tests/test_agent_graph.py`, `tests/test_agent_nodes.py`, `tests/test_chat_api.py`, `tests/test_mcp_client.py`, `tests/test_rag_pipeline.py`, `tests/test_guardrails.py`, `tests/contract/test_mcp_contract.py` |

关键实现：

- 简单经营问题走 Fast Path，直接调用稳定查询逻辑，减少 LLM 成本和不确定性。
- 复杂排障问题走 ReAct，让 LLM 根据工具结果决定下一步。
- Tool Router 按角色、任务类型和上下文裁剪工具，减少无关工具干扰。
- HITL 把高风险动作挂起，审批后从 checkpoint 恢复执行。
- RAG 结合向量召回、BM25 和 reranker，回答平台规则和排障案例。

面试常问：

- Fast Path 和 ReAct 怎么取舍？
  可确定、结构化、低推理需求的问题走 Fast Path；需要多步判断和工具反馈的问题走 ReAct。

- Checkpoint 和聊天历史有什么区别？
  聊天历史只能回放对话；checkpoint 保存图执行位置、消息、pending action，能从中断点恢复。

- RAG 为什么不能只做向量搜索？
  订单号、规则编号、券名这类精确词向量召回不稳定；BM25 补关键词，reranker 再做语义精排。

- Prompt Injection 怎么防？
  输入侧 Guardrails 拦截明显注入；工具侧 RBAC 和 schema 校验保证 LLM 即使被诱导也越不过权限；输出侧避免泄露系统配置和敏感信息。

## 10. 质量保障和压测

| 类型 | 代码/配置 |
| --- | --- |
| Java 单元测试 | `local-life-server/src/test/java` |
| Java 集成测试 | `SeckillConcurrencyIT`, `UserCouponRepositoryIT` |
| MCP 测试 | `local-life-copilot/src/test/java` |
| Python 单元/E2E/契约 | `copilot-agent-service/tests` |
| CI | `.github/workflows/*.yml` |
| 压测 | `performance-tests`, `scripts/seed-perf-data.sh` |
| 批量业务模拟 | `scripts/business-bulk-simulate.sh` |

当前测试数字：

- `local-life-server`: 74 单元 + 5 集成 = 79。
- `local-life-copilot`: 76。
- `copilot-agent-service`: 187。

面试常问：

- 为什么要加变异测试？
  行覆盖只说明代码执行过，不能证明断言有效。变异测试故意改坏代码，如果测试杀不死变异，说明断言不够强。

- 为什么 Testcontainers 比 H2 更适合？
  秒杀唯一索引、Redis Lua、MySQL 隔离级别、真实连接池行为，H2 和 Mock 都测不出来。

- 压测怎么证明秒杀正确？
  不是只看 QPS，还要看成功领取数是否超过库存、重复领取是否被拒、失败是否是业务可预期失败。

## 11. 你应该能从代码回答的问题

学完整个项目后，至少能顺着代码回答这些问题：

1. `POST /api/v1/seckill` 从 Controller 到 Redis Lua 再到 DB 唯一索引经历了什么？
2. `POST /api/v1/payments/callback` 如何保证重复回调不资损？
3. `query_order` 如何从 MCP Header 拿到角色，再查订单、支付、券状态？
4. `POST /chat` 如何判断走 Fast Path 还是 ReAct？
5. Agent 触发高风险工具时，HITL 如何挂起和恢复？
6. RAG 入库后，用户问平台规则时如何召回、重排和回答？
7. CI 失败时应该先看哪个 workflow、哪个报告、哪个测试文件？

如果这些能讲顺，你就不是“知道项目用了什么技术”，而是能把需求、实现、原理和工程质量串起来。
