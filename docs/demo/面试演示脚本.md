# LocalLife 面试演示脚本

本文用于后端 + Agent 实习面试时的 8-12 分钟演示。目标不是把所有功能点都点一遍，而是展示你能把后端交易链路、Agent 工具调用、RAG、可观测性、测试和压测讲成一个闭环。

## 1. 演示前准备

启动本地开发环境：

```bash
cd infra
docker compose -f docker-compose.dev.yml up -d
```

确认健康状态：

```bash
curl -s http://localhost:8080/actuator/health
curl -s http://localhost:8081/actuator/health
curl -s http://localhost:8000/health
```

准备业务演示数据：

```bash
cd ..
bash scripts/business-simulate.sh
```

可选：跑端到端 smoke：

```bash
NO_PROXY='*' no_proxy='*' bash scripts/e2e-smoke.sh
```

## 2. 开场讲法

推荐用 60 秒讲清项目定位：

> 这是一个本地生活交易系统 + 企业级 Copilot 项目。后端用 Spring Boot 实现用户、商家、门店、内容、优惠券、秒杀、订单、支付和 outbox 最终一致性；Agent 侧用 FastAPI + LangGraph/ReAct，接入 MCP 工具、RAG 知识库、HITL 人工审批和 DeepSeek flash 模型。这个项目重点不是单纯调 LLM，而是把真实业务系统的可查询、可执行、可审计能力安全地暴露给 Agent。

## 3. 第一段：后端业务链路

打开 Swagger：

- LocalLife Server: http://localhost:8080/swagger-ui.html
- MCP Server: http://localhost:8081/swagger-ui.html
- Agent Service: http://localhost:8000/docs

演示重点：

1. `GET /api/v1/shops/{shopId}`：公开门店详情，说明 Caffeine + Redis 双层缓存思路。
2. `GET /api/v1/coupons/templates`：公开秒杀活动列表，说明活动、券模板、库存的拆分。
3. `POST /api/v1/seckill`：说明 Redis Lua 原子扣减、防重复领取、DB 最终兜底。
4. `GET /api/v1/orders`：说明订单按 `user_id % 4` 分片，用户订单列表命中单分片。
5. `POST /api/v1/payments/callback`：说明支付回调、幂等、outbox 事务消息。

面试官常问：

| 问题 | 回答抓手 |
| --- | --- |
| 为什么订单要分片？ | 单表订单到千万级后索引树变深；用户查订单是高频路径，用 user_id 分片可以单分片命中。 |
| 秒杀怎么防超卖？ | Redis Lua 原子校验库存和用户领取集合，DB 唯一索引兜底。 |
| 支付成功但 MQ 发送失败怎么办？ | 事务内写 outbox_message，Relay 异步投递，失败重试和死信自动恢复。 |

## 4. 第二段：Agent + MCP 链路

先直接调 MCP，证明工具层可用：

```bash
curl -s http://localhost:8081/mcp \
  -H 'Content-Type: application/json' \
  -H 'X-User-Id: 880000000001' \
  -H 'X-User-Role: merchant' \
  -H 'X-Merchant-Id: 880000100001' \
  -d '{"jsonrpc":"2.0","id":"demo-tools","method":"tools/list","params":{}}'
```

查询一笔正常订单：

```bash
curl -s http://localhost:8081/mcp \
  -H 'Content-Type: application/json' \
  -H 'X-User-Id: 880000000001' \
  -H 'X-User-Role: merchant' \
  -H 'X-Merchant-Id: 880000100001' \
  -d '{"jsonrpc":"2.0","id":"demo-order","method":"tools/call","params":{"name":"query_order","arguments":{"order_id":"202606100001"}}}'
```

查询一笔异常订单：

```bash
curl -s http://localhost:8081/mcp \
  -H 'Content-Type: application/json' \
  -H 'X-User-Id: 880000000001' \
  -H 'X-User-Role: merchant' \
  -H 'X-Merchant-Id: 880000100001' \
  -d '{"jsonrpc":"2.0","id":"demo-abnormal","method":"tools/call","params":{"name":"query_order","arguments":{"order_id":"202606100002"}}}'
```

讲解要点：

- Agent 不直接连业务库，而是通过 MCP 工具访问受控能力。
- MCP Server 做 RBAC、审计、限流和结构化错误。
- 高风险动作，如退款和补券，不直接执行，必须走 HITL。
- Tool schema 里写了业务提示，减少 LLM 盲目 ReAct 循环。

## 5. 第三段：Agent 对话效果

用 Agent 发起一条真实问题：

```bash
curl -N http://localhost:8000/chat \
  -H 'Content-Type: application/json' \
  -H 'X-User-Id: 880000000001' \
  -H 'X-User-Role: merchant' \
  -H 'X-Merchant-Id: 880000100001' \
  -d '{"message":"帮我排查订单 202606100002 为什么用户支付成功但订单还显示待支付，并给出处理建议","session_id":880001}'
```

理想观察点：

1. SSE 中出现 `tool_call` 和 `tool_result`。
2. Agent 调用 `query_order` 或 `query_payment`。
3. 最终回答能指出订单、支付状态不一致。
4. 如涉及退款/补券，Agent 应该要求人工审批，而不是直接执行。

如果现场 LLM API 不稳定，可以切换到 MCP 直调演示，并说明 Agent 链路此前 smoke 已验证。

## 6. 第四段：测试和压测

测试体系可以这样讲：

- Java: 单元测试、集成测试、JaCoCo、PIT 变异测试门禁。
- Python Agent: LangGraph 节点、Tool Router、RAG、Chat API、E2E 测试。
- 端到端: `scripts/e2e-smoke.sh` 覆盖 Server、MCP、Agent、RAG。
- 压测: Locust 已有综合场景，k6 新增秒杀专项脚本。

秒杀压测准备：

```bash
bash scripts/seed-perf-data.sh
```

k6 秒杀专项：

```bash
k6 run performance-tests/k6/seckill.js
```

验收口径：

- `claimed_success` 不超过 Redis 预置库存。
- `http_req_failed` 不把售罄、重复领取、限流这类业务正常失败算作系统错误。
- 关注 P95/P99、RPS、Redis 慢日志、JVM GC、数据库连接池。

## 7. 收尾讲法

推荐 30 秒总结：

> 这个项目我最想表达的是：我不是只会把 LLM API 接进来，而是能把后端真实业务系统、数据一致性、权限审计、可观测性、测试门禁和 Agent 工具调用合成一个可运行的工程闭环。后续如果上线，我会优先补生产级密钥管理、灰度部署、告警规则、压测基线和真实支付沙箱。

## 8. 演示时避免踩坑

| 风险 | 处理 |
| --- | --- |
| DeepSeek API 慢或额度不足 | 先演示 MCP 直调，再展示已保存的 smoke 结果和日志。 |
| Docker 服务未全健康 | 先跑 `docker compose ps`，重点看 8080/8081/8000、MySQL、Redis、Milvus。 |
| Swagger 访问 401 | Java 服务已放行 `/swagger-ui.html` 和 `/v3/api-docs`，MCP Filter 已放行文档路径。 |
| k6 未安装 | 用 Locust 现有脚本替代，或只展示 k6 脚本和验收口径。 |
