# 企业级测试实践：集成测试 · 覆盖率 · 压测

> 本文记录在 `local-life-server` 与 `copilot-agent-service` 上落地的三件「企业级测试」工作：
> ① 用 **Testcontainers** 拉真实 MySQL / Redis 跑集成测试；② **JaCoCo + pytest-cov** 覆盖率统计；
> ③ 秒杀模块的**并发正确性测试 + Locust 压测**。
> 配套的「单元测试」明细见 [测试体系与面试答辩指南.md](./测试体系与面试答辩指南.md)。

---

## 一、为什么需要这一层：测试金字塔

```
        /\          E2E / 压测（少量，慢，验证「系统在真实负载下的行为」）
       /  \         → Locust HTTP 压测、秒杀并发正确性
      /----\
     /      \       集成测试（中量，中速，验证「与真实中间件的交互」）
    /        \      → Testcontainers 真 MySQL / 真 Redis
   /----------\
  /            \    单元测试（大量，快，验证「单个类的分支逻辑」）
 /______________\   → JUnit + Mockito / pytest + mock
```

**单元测试的盲区**，正是这次补的：

| 单元测试测不到的东西 | 为什么测不到 | 用什么补 |
|---------------------|------------|---------|
| Lua 脚本在高并发下到底超不超卖 | `stringRedisTemplate.execute()` 被 Mock 了 | 真 Redis + 1000 线程并发 IT |
| 唯一索引能不能挡住重复领券 | Mock 的 mapper 不会抛唯一键冲突 | 真 MySQL 唯一约束 IT |
| 逻辑删除改写的 SQL 在 MySQL 上对不对 | H2/Mock 的方言与 MySQL 有差异 | 真 MySQL IT |
| 接口在 200 并发下的 P99 | 单元测试没有「负载」概念 | Locust 压测 |

## 二、Testcontainers 集成测试

### 2.1 选型理由（面试常问：为什么不用 H2 / 嵌入式 Redis）

- **行为保真**：H2 不是 MySQL——唯一索引冲突的异常类型、`ON UPDATE CURRENT_TIMESTAMP`、
  `utf8mb4` 排序规则、隔离级别语义都不一样。集成测试的价值就在于「测真实依赖的真实行为」，
  换成 H2 等于测了个寄生环境，假绿。
- **环境一致**：容器镜像与生产同款（`mysql:8.4`、`redis:7.4-alpine`，与
  `infra/docker-compose.dev.yml` 对齐），本地 / CI 跑出来的结果一致。
- **隔离 & 可重复**：容器随测试启动、测试结束即销毁，不污染开发库，不存在「上一次跑剩的脏数据」。

### 2.2 Maven 工程化：单元测试与集成测试分离

```
surefire  → **/*Test.java   绑定 test 阶段     （mvn test，快，每次提交都跑）
failsafe  → **/*IT.java      绑定 verify 阶段   （mvn verify，慢，依赖 Docker）
```

这是 Maven 区分两类测试的标准约定：开发循环里 `mvn test` 秒回；
集成测试 `mvn verify` 在 CI 或本地按需跑，不拖慢日常。

### 2.3 测试一：SeckillConcurrencyIT（真 Redis + 真 Lua，防超卖）

**文件**：`local-life-server/.../module/seckill/SeckillConcurrencyIT.java`

这是整个测试体系**技术含量最高**的一个。它用一个真实 Redis 容器跑真实的 `seckill.lua`，
然后用线程池制造极限并发争用：

```
1000 个线程，全部 await 同一个 CountDownLatch（发令枪）
       ↓ countDown 瞬间「同时」起跑，最大化争用
争抢预置库存 = 100
       ↓
断言：成功数 == 100（一个不多）、Redis 最终库存 == 0、已抢集合 size == 100
```

第二个用例反向验证「一人一单」：**同一个用户** 50 个并发请求，断言**恰好 1 次成功**。

**它证明了什么**：Lua 在 Redis 单线程模型里整段原子执行，判重 + 判库存 + 扣减不会被打断。
如果把这三步拆成普通 Redis 命令，这个测试会立刻变红（多个线程同时读到「有库存」→ 超卖）。
**这就是「我的秒杀不会超卖」这句话的可执行证据**，比口头描述有力得多。

**工程取舍**：没有启动整个 Spring 容器，而是手动 `new SeckillService(...)`，
DB 类依赖用 Mockito Mock，只让 `StringRedisTemplate` 连真容器——聚焦被测点（Redis 原子性），
启动快、不被无关组件干扰。

### 2.4 测试二：UserCouponRepositoryIT（真 MySQL，唯一索引兜底）

**文件**：`local-life-server/.../module/seckill/UserCouponRepositoryIT.java`

用真实 MySQL 8.4 验证仓储层三件事：

| 用例 | 验证 |
|------|------|
| `insert_thenSelect_roundTrips` | MyBatis-Plus 字段↔列映射、`@TableField(fill)` 自动填充在真库往返一致 |
| `uniqueIndex_blocksDoubleClaim_forSameUserAndTemplate` | `uk_user_coupon_template (user_id, coupon_template_id)` 抛 `Duplicate entry`——**一人一单的 DB 最后防线** |
| `logicDelete_hidesRowFromSelect_butKeepsItPhysically` | `@TableLogic` 把 DELETE 改写成 `UPDATE deleted=1`，查询自动追加 `WHERE deleted=0`，但物理行还在 |

**核心叙事**：Redis 判重是第一道防线（快、挡住 99.9%），但 Redis 可能宕机 / Key 过期 /
预热遗漏；**数据库唯一索引是最后一道兜底**——即使 Redis 整个失效，DB 也保证一人一单。
这个测试就是兜底防线的可执行证明。

**工程取舍**：本服务的全 `@SpringBootTest` 会连带拉起 ES / RocketMQ / ShardingSphere / Canal，
又慢又脆。这里只手动装配 `DataSource → MybatisSqlSessionFactoryBean → UserCouponMapper`
最小一条链路，并注册与生产 `MybatisPlusConfig` 一致的 `MetaObjectHandler`（自动填充
createdAt/updatedAt），其余一概不加载。

### 2.5 运行

```bash
# 跑全部集成测试（需要本机 Docker 可用）
mvn -pl local-life-server verify

# 只跑某一个
mvn -pl local-life-server failsafe:integration-test failsafe:verify -Dit.test=SeckillConcurrencyIT
```

实测结果：**单元 74 + 集成 5 = 79 个测试全绿**；首次运行会拉 `mysql:8.4`、
`redis:7.4-alpine` 等 Testcontainers 镜像。

## 三、代码覆盖率

### 3.1 Java：JaCoCo

`jacoco-maven-plugin` 三段式执行：`prepare-agent`（给单元测试注入探针）+
`prepare-agent-integration`（给集成测试注入，**append 到同一个 `jacoco.exec`**）+
`report`（verify 阶段合并出一份总报告）。一份报告同时反映单元 + 集成两层覆盖。

```bash
mvn -pl local-life-server verify
# 报告：local-life-server/target/site/jacoco/index.html
```

`mvn verify` 已接入 JaCoCo `check-core-coverage` 门禁，按核心类单独设置阈值：
Seckill/Coupon/Payment/Shop/Order/AuthInterceptor 低于当前基线会直接 fail build。

**被测核心业务类覆盖率**（指令 / 分支 / 行）：

| 类 | 指令 | 分支 | 行 |
|----|------|------|----|
| `SeckillService` | 94.5% | 96.4% | 94.1% |
| `AuthInterceptor` | 98.0% | 91.7% | 98.1% |
| `PaymentService` | 89.4% | 75.0% | 88.3% |
| `OrderService` | 68.7% | 67.4% | 67.5% |

> 全项目整体约 24.8%——因为 Controller、ES、MQ、Canal 这类 I/O 黏合层没做单元测试。
> **这是刻意的优先级**：把测试投在「写错就出资损/超卖/越权」的核心业务逻辑上，
> 而不是追求一个好看的总数字。面试时要主动讲这个取舍，而不是回避低的总覆盖率。

### 3.2 Python：pytest-cov

```bash
cd copilot-agent-service && source .venv/bin/activate
pip install -r requirements-dev.txt
pytest --cov --cov-report=term-missing --cov-report=html --cov-fail-under=45
# 报告：copilot-agent-service/htmlcov/index.html
```

配置见 `.coveragerc`（`branch = True` 开启分支覆盖，omit 掉测试/配置/启动入口）。

**被测核心模块覆盖率**：

| 模块 | 覆盖率 | 说明 |
|------|--------|------|
| `agent/tool_router.py` | 100% | RBAC 过滤 + 任务路由 |
| `guardrails/input_checker.py` | 100% | Prompt 注入防护 |
| `mcp/mcp_client.py` | 94.6% | JSON-RPC 客户端 |
| `rag/bm25_store.py` | 88.5% | BM25 检索 + 权限过滤 |
| `rag/pipeline.py` | 83.5% | RRF 融合检索 |

> 整体 35.2%——未覆盖的是 `nodes.py`（ReAct 循环，需真实 LLM）、`vector_store.py`（需真 Milvus）、
> `embedding/reranker`（需加载本地模型）这些重 I/O 模块，同样是刻意取舍。

## 四、秒杀压测

「压测」分两个层次，**两个都要有**：

### 4.1 并发正确性（已自动化）

即 §2.3 的 `SeckillConcurrencyIT`——它本质就是一个「正确性压测」：1000 线程瞬时并发，
断言零超卖。这部分进了 CI，每次 `mvn verify` 都验证，不依赖人肉观察。

### 4.2 吞吐 / 延迟压测（Locust）

**文件**：`performance-tests/locustfile_locallife_server.py`

针对 HTTP 接口的吞吐与 P99 延迟压测，三个场景：`LocalLifeUser`（读多写少混合流量）、
`SeckillUser`（秒杀专项）、`SearchUser`（ES 搜索）。

秒杀场景做了两个关键增强：
1. **不同虚拟用户用不同手机号**（`13900000000` 起的连号池）——否则所有用户同一个号，
   第一个抢到后其余全是「已领取」，测不出真实争抢。
2. **HTTP 层超卖核对**：结束时打印 `claimed / sold_out / duplicated / rate_limited / error`
   分布，`claimed` 总数必须 ≤ Redis 预置库存，否则 HTTP 层也能看出超卖。

```bash
cd performance-tests && pip install -r requirements.txt

# Web UI（实时图表）
locust -f locustfile_locallife_server.py --host http://localhost:8080

# 秒杀专项无头压测：200 并发，30 秒
locust -f locustfile_locallife_server.py --host http://localhost:8080 \
  --users 200 --spawn-rate 50 --run-time 30s --headless SeckillUser
```

性能基准（开发机本地 MySQL+Redis）：`/api/v1/seckill` P99 < 50ms（主路径在 Redis，DB 异步落库）。

> ⚠️ **压测前置数据准备（需手动）**：Locust 秒杀场景假设 DB 里已存在
> `13900000000` ~ `13900001999` 这批测试账号，且 Redis 已预热秒杀库存。
> 这部分依赖运行环境，未纳入自动化（见下「待办」）。

## 五、面试速答

**Q：你这个项目的测试都怎么做的？**
分三层：单元测试（JUnit+Mockito / pytest，~200 个，跑核心类的分支逻辑）；集成测试
（Testcontainers 拉真 MySQL/Redis，验证唯一索引、Lua 原子性这些 Mock 测不到的真实行为）；
压测（Locust 测吞吐/P99 + 一个 1000 线程的并发正确性测试验证秒杀不超卖）。

**Q：为什么用 Testcontainers 而不是 H2？**
H2 的方言和 MySQL 不一样——唯一索引冲突的异常、时间戳默认值、排序规则都有差异。
集成测试的意义就是测真实依赖，用 H2 等于测了个假环境。Testcontainers 用的镜像和生产同款，
容器随用随销毁，隔离且可重复。

**Q：怎么证明你的秒杀不超卖？**
有个集成测试：真 Redis 跑真 Lua 脚本，1000 个线程用 CountDownLatch 同时起跑抢 100 库存，
断言恰好 100 个成功、库存归零。Lua 在 Redis 单线程里原子执行，判重+扣减不会被打断。
要是改成普通 Redis 命令，这个测试立刻红。

**Q：覆盖率多少？为什么整体不高？**
核心业务类很高——秒杀 Service 指令覆盖 94%、分支 96%，鉴权 98%，支付 89%。整体只有 ~25%，
因为 Controller、ES、MQ 这些黏合层我没刷覆盖率。这是刻意的：测试投在写错会出资损/超卖/越权的
地方，而不是追求一个好看的总数。

**Q：单元测试 mock 了 Redis，那 Lua 到底测没测？**
单元测试（`SeckillServiceTest`）mock 掉了 `redisTemplate.execute`，只测「Lua 返回 0/1/2 时
Service 怎么分支」——它测不到 Lua 本身对不对。所以才单独写了集成测试用真 Redis 跑真 Lua。
两层各司其职：单元测分支逻辑，集成测真实原子性。

**Q：集成测试为什么不直接 @SpringBootTest？**
全上下文会连带启动 ES / RocketMQ / ShardingSphere / Canal，又慢又脆，且与被测点无关。
我只手动装配被测那条最小链路（比如 DataSource→Mapper，或 Redis→Service），启动快、稳定。
这也是「集成测试范围要尽量窄」的工程原则。
