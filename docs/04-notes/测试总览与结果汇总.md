## 二、测试金字塔（本项目落地形态）

```
         E2E / 压测 (4)
        ├─ Locust HTTP 压测（吞吐/P99）
        └─ 真实图 ReAct E2E（llm→tool→llm→final，2 例）
              ▲
       契约测试 (2)  ── Pact：Python McpClient ↔ Java MCP Server JSON-RPC
              ▲
      集成测试 (6)  ── Testcontainers 真 MySQL / 真 Redis
        ├─ SeckillConcurrencyIT：1000 线程抢 100，零超卖
        └─ UserCouponRepositoryIT：唯一索引兜底一人一单
              ▲
      单元测试 (330+) ── JUnit+Mockito / pytest+mock，纯逻辑分支
              ▲
      质量校验 ── 覆盖率（JaCoCo / pytest-cov）+ 变异测试（PIT / mutmut）
```

每升一层：更慢、更真、更少。单元测分支逻辑，集成测真实依赖行为，契约测跨服务接口一致性，
E2E 测整链路闭环，变异测「测试本身有没有用」。

## 三、各测试类型详情

### 3.1 单元测试

**Java（local-life-server，74 个）**：`AuthInterceptorTest`(10)、`SeckillServiceTest`(15)、
`OrderServiceTest`(19)、`PaymentServiceTest`(11)、`ShopServiceTest`(15)、`CouponServiceTest`(4)。
逐条动机见 [测试体系与面试答辩指南.md](./测试体系与面试答辩指南.md)。

**Python（copilot-agent-service）**：`test_guardrails`(31)、`test_tool_router`(27)、

`test_agent_graph`(29，ReAct 路由/分批/循环检测/压缩切分)、`test_agent_nodes`(10，异步节点)。

新增的 **ReAct 控制流测试**（`test_agent_graph.py` + `test_agent_nodes.py`）是这次的重点：
- `route_after_llm`：终止/HITL/压缩/反思/工具/收尾六向路由 + 优先级，逐分支覆盖
- `_partition_tool_calls`：并发安全分批（安全工具并发、高风险串行隔离，fail-closed）
- `_detect_loop`：同工具同参 ≥3 次死循环熔断
- `tool_node` / `llm_node` / `final_node`：mock LLM+MCP，测节点产出与三种终止原因

### 3.2 集成测试（Testcontainers，真实 MySQL + Redis）

| 测试 | 容器 | 验证 |
|------|------|------|
| `SeckillConcurrencyIT` | 真 Redis | 1000 线程并发抢 100 库存 → **恰好 100 成功，零超卖**；同一用户 50 并发 → 1 次 |

### 3.3 契约测试（Pact，消费者驱动）

### 3.4 E2E 测试

**已自动化（in-process，`tests/test_e2e_agent.py`，2 例）**：用 scripted LLM + mock MCP 驱动
**真实编译的 LangGraph 图**跑完整闭环：
- `llm_node →(决定调工具) tool_node →(总结) llm_node → final_node → END`，断言工具被调一次、

### 3.5 变异测试（测试有效性）

**Python — mutmut（已跑通）**：对 `tool_router.py` + `input_checker.py` 共 **216 个变异，杀死 110（≈51%）**。
- 存活的 106 个绝大多数是 **日志语句的字符串/参数变异**（如 `log.warning("x")`→`log.warning(None)`），
  测试本就不该断言日志内容——这是「等价变异/噪声」，不是盲区。
- **真实盲区**被揪出 1 个并已修复：DB 凭据检测正则带 `re.IGNORECASE`，但原测试只覆盖小写，
  「删掉 IGNORECASE」的变异能存活 → 补 `PASSWORD=` / `JDBC:` 大写用例将其杀死。
- 这正是变异测试的价值：行覆盖 100% ≠ 测试有效，它能指出「执行到了但没断言到」的地方。
- 运行：`pip install -r requirements-dev.txt && mutmut run && mutmut results && mutmut html`

**Java — PIT（已配置，当前 WSL 环境跑不起来，留作 CI）**：
- 已在 `local-life-server/pom.xml` 配好 `pitest-maven` + `pitest-junit5-plugin`，目标锁定
  Seckill/Order/Coupon/Payment 四个 Service。
- 本地 WSL2 环境下 PIT minion 子进程启动即崩（`UNKNOWN_ERROR`，无 Java 栈、无 hs_err），
  与具体测试无关（最简单的 CouponService 也崩）——属环境层问题。已加 `useClasspathJar`/
  `parseSurefireConfig=false` 等加固，仍未解决；**在标准 Linux CI 上可正常出分**。
- 运行：`mvn -pl local-life-server test-compile org.pitest:pitest-maven:mutationCoverage`，
  报告 `target/pit-reports/index.html`。

## 五、面试速答（合集）

**Q：项目测试体系怎么搭的？**
五层：单元（330+，JUnit+Mockito / pytest，跑分支逻辑）→ 集成（Testcontainers 真 MySQL/Redis，
验证唯一索引、Lua 原子性）→ 契约（Pact，跨语言 Agent↔MCP 接口一致性）→ E2E（真实 LangGraph 图跑
ReAct 闭环）→ 质量校验（JaCoCo/pytest-cov 覆盖率 + PIT/mutmut 变异测试）。

**Q：怎么证明秒杀不超卖？**
集成测试用真 Redis 跑真 Lua，1000 线程 CountDownLatch 同时起跑抢 100 库存，断言恰好 100 成功、
库存归零。改成普通 Redis 命令这测试立刻红。

**Q：覆盖率多少，为什么不追高？**
核心类很高（秒杀 Service 指令 94%/分支 96%、Coupon 100%、鉴权 98%），整体只有 Java 31%/Py 48%——
刻意的：测试投在出资损/超卖/越权/注入的核心逻辑，不刷 Controller/ES/MQ 黏合层。

**Q：覆盖率高就够了吗？**
不够。行覆盖只说明「执行到了」，不说明「断言到了」。所以加了变异测试：mutmut 对路由/防注入模块跑了
216 个变异，杀死率 51%，还揪出一个真实盲区——防注入正则的大小写不敏感没被测到，补了用例杀掉它。

**Q：跨语言的两个服务怎么保证接口不对错？**
消费者驱动契约测试（Pact）。Python 的 McpClient 作为消费者，对着 Pact mock 跑真实调用，生成契约文件；
契约再交给 Java MCP Server 做 provider 验证。任一方改坏 JSON-RPC 接口，重放契约即失败。

**Q：为什么集成测试用 Testcontainers 不用 H2？**
H2 方言和 MySQL 不一致（唯一索引冲突异常类型、时间默认值、排序规则）。集成测试要测真实依赖行为，
H2 等于测了个假环境。Testcontainers 用与生产同款镜像，容器随用随销毁，隔离可重复。

**Q：ReAct Agent 这种「跑起来才知道」的东西怎么测？**
分两层：纯逻辑（`route_after_llm` 的六向路由、并发分批、死循环熔断、上下文压缩切分）直接单测；
整图闭环用 scripted LLM + mock MCP 驱动真实编译的 LangGraph 跑一遍 llm→tool→llm→final，
断言工具被调、最终收口——既不花真实 LLM 的钱，又验证了接线。

## 六、已知缺口 / 待办（需环境或人工）

1. **PIT 在 WSL 跑不起来**：minion 子进程崩溃（环境问题）。在标准 Linux CI 上接入即可出变异分。
2. **Pact Provider 侧验证未接入**：需 MCP Server 运行 + pact-jvm/CLI，建议进 CI。
3. **压测数据 seed 已脚本化**（`scripts/seed-perf-data.sh`），但跑压测仍需手动起服务栈。
4. **重 I/O 模块仍 0/低覆盖**：Python 的 `vector_store`（需真 Milvus）、`session/*`（需 DB）、
   Java 的 Controller/ES/MQ/Canal——属刻意未覆盖，可按需用 Testcontainers（Milvus/MySQL）补。
5. **CI 接入**：把 `mvn verify`（Testcontainers）、`pytest --cov`、契约、变异测试编进 GitHub Actions，
   覆盖率/变异分作为门禁或 artifact。
