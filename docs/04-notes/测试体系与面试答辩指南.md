## 二、local-life-server 测试

### 2.1 AuthInterceptorTest（10 tests）

**被测类**：`AuthInterceptor` — JWT Bearer Token 鉴权拦截器，是所有需要登录接口的安全门卫。

#### 核心逻辑
1. 从 `Authorization: Bearer {token}` Header 提取 Token
2. 到 Redis 查 `login:token:{token}` 是否存在
3. 反序列化为 `LoginUserDTO`，校验账号状态
4. 写入 `UserContext`（ThreadLocal），供后续 Service 读取
5. 滑动刷新 Token TTL（7天）
6. `afterCompletion` 中清理 ThreadLocal 和 MDC

#### 为什么直接 `new`，不用 `@WebMvcTest`
- `AuthInterceptor` 有两个构造依赖（`StringRedisTemplate`、`ObjectMapper`），都可以轻量替身
- 直接 `new` 拿到真实实例，测试最快、最不脆弱
- 与 copilot 侧 `RbacFilterTest` 同一思路

#### 技术亮点：`doAnswer` 快照验证注入时机
多数用例不是断言"preHandle 返回后 UserContext 是什么"，而是用 Mockito `doAnswer` 在 `chain.doFilter()` 真正执行的瞬间拍快照：

```java
AtomicReference<LoginUserDTO> seenDuringChain = new AtomicReference<>();
doAnswer(inv -> { seenDuringChain.set(UserContext.get()); return null; })
    .when(chain).doFilter(any(), any());
```

这能区分"在 chain 执行前已正确注入"和"chain 执行后才注入、下游其实读到 null"这两种南辕北辙的实现。

#### 用例列表

| 用例 | 验证场景 |
|------|---------|
| `missingAuthHeader_throwsTokenMissing` | 无 Authorization Header → `AUTH_TOKEN_MISSING` |
| `malformedBearerPrefix_throwsTokenMissing` | 不以 "Bearer " 开头 → `AUTH_TOKEN_MISSING` |
| `tokenNotInRedis_throwsTokenExpired` | Redis 没有这个 Token → `AUTH_TOKEN_EXPIRED` |
| `disabledAccount_throwsAccountDisabled` | Redis 有 Token 但 status=DISABLED → `USER_ACCOUNT_DISABLED` |

### 2.2 SeckillServiceTest（15 tests）

**被测类**：`SeckillService` — Redis Lua 脚本原子秒杀、场次时间窗校验、多层查询兜底。

#### 核心逻辑
1. 从 DB 取场次，校验时间窗（before/after）和状态（PENDING/ACTIVE/ENDED）及 templateId 绑定
2. 执行 Redis Lua 脚本原子判断：减库存 + 写用户集合，返回 0/1/2/null
3. Lua 返回 0 → 发 Outbox 事件（异步写库）+ 写 RESULT_KEY=PENDING
4. `querySeckillResult` 多层兜底：DB → RESULT_KEY → USER_SET_KEY

#### 亮点1：Lua 脚本完全解耦
`SECKILL_SCRIPT` 是静态 final 字段，真正的 Lua 逻辑在 Redis 执行。测试直接 mock `execute()` 返回值，与脚本内容解耦，四条分支全覆盖：

```java
@SuppressWarnings("unchecked")
private void stubLuaResult(Long returnValue) {
    doReturn(returnValue).when(stringRedisTemplate).execute(any(), anyList(), any(Object.class));
}
```

#### 亮点2：`SetOperations.isMember` varargs 歧义
Spring Data Redis 3.x 同时存在 `isMember(K, Object)→Boolean` 和 `isMember(K, Object...)→Map`，`any()` 触发 varargs 版。修复：

```java
doReturn(Boolean.TRUE).when(setOps).isMember(anyString(), (Object) any());
```

### 2.3 OrderServiceTest（19 tests）

**被测类**：`OrderService` — 创单、幂等、门店/券校验、金额计算、取消、关单。

#### 核心逻辑
1. 幂等：Redis `setIfAbsent` 检查 idempotencyKey
2. 门店：selectById → 校验 ONLINE 状态
3. 券：4 个拒绝路径（已使用、已过期、门店不匹配、最低消费不满足）
4. 金额：服务端计算，CASH 直减、PERCENT 折扣，不信任客户端
5. 事务后钩子：`registerSynchronization(afterCommit)` 发延时 MQ 消息触发关单
6. `cancelOrder`：CAS 更新 `WHERE status=WAIT_PAY`，状态机保护
7. `handleOrderCloseDelayMessage`：幂等关单

#### 亮点1：MP 3.5.7 `initTableInfo` 需要真实 MapperBuilderAssistant
`LambdaUpdateWrapper` 在构建时就通过 `TableInfoHelper` 解析字段→列名，即使 mapper 是 mock。MP 3.5.7 不允许传 null，需要一个最小化的 `MapperBuilderAssistant`：

```java
@BeforeAll
static void initMybatisPlusLambdaCache() {
    MapperBuilderAssistant assistant = new MapperBuilderAssistant(new MybatisConfiguration(), "");
    TableInfoHelper.initTableInfo(assistant, UserCoupon.class);
    TableInfoHelper.initTableInfo(assistant, OrderInfo.class);
}
```

#### 亮点2：BaseMapper#insert 重载歧义（MP 3.5.7）
MP 3.5.7 同时有 `insert(T entity)` 和 `insert(Collection<T>)`，泛型擦除后 `any()` 无法区分。修复：

```java
doAnswer(inv -> {
    OrderInfo o = (OrderInfo) inv.getArgument(0);
    o.setId(ORDER_ID);
    return 1;
}).when(orderInfoMapper).insert((OrderInfo) any());
```

#### 亮点3：TransactionSynchronizationManager
`createOrder` 调用 `registerSynchronization()`，无活跃事务同步时抛 `IllegalStateException`。测试中激活同步而不启动真实事务：

```java
@BeforeEach void setUp() { TransactionSynchronizationManager.initSynchronization(); }
@AfterEach  void tearDown() { TransactionSynchronizationManager.clearSynchronization(); }
```

### 2.4 PaymentServiceTest（11 tests）

#### 核心逻辑
1. `createPayment`：校验订单归属、状态（非 WAIT_PAY 拒绝）、渠道（只支持 MOCK）

### 3.1 RbacContextTest（11 tests）

**核心动机**：`canAccessMerchant()` 是本系统唯一的"商家数据隔离"判定入口——merchant 角色不能看别家数据，条件写错一个就是真实越权漏洞。

#### 亮点：参数化覆盖所有角色×场景矩阵

```java
@ParameterizedTest
@MethodSource("accessDecisions")
void canAccessMerchant_decidesByRoleAndMerchantIdMatch(...) { ... }
```

8 个场景：admin/cs/merchant × 自己/别人/null merchantId，全部一次性覆盖。

#### 关键防御测试
- **大小写敏感性**：`"Merchant" ≠ "merchant"`，过滤器不做归一化
- **null role NPE 防护**：`"merchant".equals(role)` 常量在左，null role 安全返回 false
- **ThreadLocal 清理往返**：`set → get → clear` 的完整生命周期

### 3.2 RbacFilterTest（10 tests）

**核心动机**：HTTP Header 解析 + 身份注入的安全门卫，一处出错整条鉴权链路失效。

#### 亮点1：快照验证注入时机（与 AuthInterceptorTest 同样的技巧）

```java
doAnswer(inv -> { seenDuringChain.set(RbacContext.get()); return null; })
    .when(chain).doFilter(any(), any());
// 之后断言 seenDuringChain.get().getUserId() == 期望值
```

`McpController` 在 chain 执行期间调用 `RbacContext.get()`，所以只验证"之后"是不够的。

#### 亮点2：发现并修复了一个 Bug

**Bug**：非数字 Header（`X-User-Id: abc`）让 `Long.parseLong()` 抛 `NumberFormatException`，
冒泡成容器层 500，而不是过滤器应该负责的干净 401。

**修复**：在 `RbacFilter` 里加 try-catch，与"缺失 Header"统一走 401 + JSON 错误体。

**回归测试**：
```java
void nonNumericUserId_returns401_insteadOfLetting500Escape()
void nonNumericMerchantId_returns401_insteadOfLetting500Escape()
```

#### 亮点3：finally 块清理保证

```java
doThrow(new RuntimeException("downstream boom")).when(chain).doFilter(any(), any());
// 断言：异常继续传播，且 RbacContext.get() == null
```

线程池复用时，任何一个请求出异常如果不清理 ThreadLocal，下一个请求就会读到它残留的身份——这是严重越权风险。

### 3.3 QueryOrderToolTest（8 tests）

**被测类**：L1 只读工具，覆盖"参数提取 → RBAC 上下文检查 → DB 查询 → 结果整理"链路。

#### 亮点：防枚举攻击设计

"订单不存在"和"无权查询"统一返回 `not_found`，不区分两种情况：

```java
// 真的没有这条记录
when(orderMapper.selectOrderByOrderNo("xxx")).thenReturn(null);
// → ToolNotFoundException("订单不存在或无权查询: xxx")

// 空行（JOIN 过滤后没有匹配列）
when(orderMapper.selectOrderByOrderNo("xxx")).thenReturn(new HashMap<>());
// → 同样的 ToolNotFoundException
```

如果区分两种情况，攻击者可以通过不同错误码探测订单号是否存在（信息泄露）。

#### 亮点：safeStr() 的 null 防护

```java
// 场景：pay_status=null（未发起支付）
assertThat(payment).containsEntry("pay_status", null);  // 而不是 NPE
```

`safeStr(null)` 必须返回 null，而不是字符串 `"null"` 或 NPE。这是工具 `x-business-hint` 里"pay_status=null → 未发起过支付"判断逻辑的数据基础。

### 3.5 IssueCompensationCouponToolTest（11 tests）

**核心动机：修复一个 NPE Bug**

**Bug**：
```java
// 修复前
arguments.get("compensation_amount").asInt(0)
// JsonNode#get() 键缺失时返回 null（不是 MissingNode），
// null.asInt(0) → NullPointerException → Agent 收到 500，无法恢复
```

- 值为 0 → `ToolParameterException("amount 必须大于 0")`
- 值为负数（参数化：0/-1/-2990）→ 同上

### 3.6 ToolAuditServiceTest（5 tests）

**核心动机**：`toJson()` 方法曾存在两处缺口，任意一处都会让非法 JSON 写入 `JSON NOT NULL` 列，触发 `MysqlDataTruncation: Invalid JSON text`。

#### 两处旧缺口

```java
// 缺口 1：String 不加验证直接返回
if (obj instanceof String s) return s;  // "plain text" 不是合法 JSON

// 缺口 2：序列化失败时 toString() 兜底
} catch (JsonProcessingException e) {
    return obj.toString();  // toString() 结果同样不保证是合法 JSON
}
```

#### 修复后的行为（每条用例一个断言）

| 用例 | 输入 | 期望输出 | 验证点 |
|------|------|---------|--------|
| Map 序列化 | `{"a":1}` | `{"a":1}` | 正常路径 |
| 已是合法 JSON 的 String | `{"already":"json"}` | 原样返回 | 不双重编码 |
| 非 JSON 裸字符串 | `"plain text"` | `"\"plain text\""` | Jackson 加引号，合法 JSON |
| null | `null` | `null` | 返回 Java null（列允许 null） |
| Unserializable | `FAIL_ON_EMPTY_BEANS` 对象 | `"\"...(类名+toString)...\""` | 合法 JSON 字符串字面量，不崩溃 |

#### 技术亮点：不 mock ObjectMapper

用 `private static final class Unserializable { private final String hidden = "secret"; }` ——
Jackson 的 `FAIL_ON_EMPTY_BEANS`（默认开启）遇到无可见属性的类会抛 `InvalidDefinitionException`。
这比 mock ObjectMapper 内部更真实，且不依赖具体的内部实现细节。

### 3.7 McpControllerTest（19 tests）

**测试方式**：`@WebMvcTest` 切片——只启动 MVC 层，不启动数据库、Redis 等基础设施。

**关键 stub**：
```java
// @MapperScan 会把 ToolAuditMapper 注册为 MapperFactoryBean，
// 切片上下文没有 MyBatis，需要 RETURNS_DEEP_STUBS 让 checkDaoConfig() 空跑过去
@MockitoBean(answers = Answers.RETURNS_DEEP_STUBS)
private SqlSessionTemplate sqlSessionTemplate;
```

#### 亮点1：用"双重否决"证明执行顺序

```java
// 构造"限流 和 RBAC 都会拒绝"的场景
stubRegisteredTool("execute_refund", List.of("cs", "admin"));  // merchant 无权
when(rateLimiter.isAllowed("execute_refund")).thenReturn(false);  // 也触发限流

// merchant 角色发请求
// 如果先 RBAC 后限流 → 返回 permission_denied
// 如果先限流后 RBAC → 返回 tool_rate_limited（实际结果）
```

把隐含的"执行顺序"设计约束变成可执行的回归测试，一旦顺序被误改立即报红。

#### 亮点2：6 种 tool result 序列化形状

| 形状 | 期望 text 字段内容 | 技术点 |
|------|-----------------|--------|
| `Map` | JSON 对象 | 已被 section 5 覆盖 |
| `JsonNode` | JSON 原样输出 | `writeValueAsString(jsonNode)` |
| DTO（record） | JSON 对象 | Jackson 2.12+ 原生 record 支持 |
| `List.of(...)` | JSON 数组 | 如 `["item_a","item_b","item_c"]` |
| `null` | `"null"` | `writeValueAsString(null) = "null"` |
| `Unserializable` | → `internal_error` MCP 响应 | 而非 `.toString()` 垃圾值 |

### 3.8 ToolAuditServiceIntegrationTest（1 test）

**这是整个测试套件中技术含量最高的一条用例。**

#### 背景：两个相互独立的 Bug

**Bug 1：RbacContext 跨 @Async 线程边界丢失**
- `@Async` 方法体整体（含 `RbacContext.get()` 那行）被派发到执行器线程
- 普通 `ThreadLocal` 不跨线程传递，执行器线程上读到 `null`
- 纯单元测试里 `new ToolAuditService(...)` 是同步调用，完全看不到这个 bug

**Bug 2：表结构漂移（Schema Drift）**
- `V1__init_copilot_tables.sql` 声明 `user_id/user_role/session_id NOT NULL`
- 本地开发库实际是 `nullable=YES`（疑似手工 ALTER 解除了约束）
- `init-db.sh` 使用自制 `schema_migrations` 流水账表（只记录"版本是否执行过"），
  没有 checksum 校验，发现不了这类漂移——对比 Flyway/Liquibase 每次启动都 validate

**两个 Bug 的复合效果**：
- 本机（nullable=YES）→ INSERT 成功，`user_id/user_role` 永远是 NULL → **安静腐化**
- CI（NOT NULL，白板新库）→ INSERT 失败，被 catch(Exception) 吞掉，只有 WARN → **响亮失败**
- 空表至少让人起疑，半残的表只会让人误以为一切正常

#### 修复

**TaskDecorator**（Spring 官方推荐方案，Micrometer Tracing 也用这个）：

```java
executor.setTaskDecorator(runnable -> {
    RbacContext ctx = RbacContext.get();   // 提交线程上捕获快照
    return () -> {
        try {
            RbacContext.set(ctx);          // 执行线程上还原
            runnable.run();
        } finally {
            RbacContext.clear();           // 对称清理，防止线程复用泄漏
        }
    };
});
```

**DB 扶正**：ALTER TABLE 让三列恢复 NOT NULL。

#### 测试设计：双分支轮询（Race-Free Dual-Outcome Polling）

不直接 `Thread.sleep()`，也不只轮询一种终态信号，而是同时轮询两种：

```java
private AsyncOutcome awaitAsyncOutcome(String toolName, CapturedOutput output, Duration timeout) {
    while (System.currentTimeMillis() < deadline) {
        if (toolAuditMapper.exists(...))        return ROW_PERSISTED;     // 分支 A
        if (output.getOut().contains(warnMarker)) return WARNING_LOGGED;  // 分支 B
        Thread.sleep(100);
    }
    return NEITHER_HAPPENED_WITHIN_TIMEOUT;
}
```

- **分支 A（ROW_PERSISTED）**：TaskDecorator 修复后的预期路径，断言 `userId=88888, userRole="admin"`
- **分支 B（WARNING_LOGGED）**：意外失败的安全网，断言 WARN 包含 tool 名字和错误信息

**为什么不用 `@Transactional`**：
1. 异步写在另一条线程的独立事务上，rollback 够不到它
2. MySQL `REPEATABLE READ`：测试事务开启后，快照定格，轮询 SELECT 永远看不到另一个连接 commit 的行——不是变慢，是**永远等不到**，测试必然超时

## 七、面试高频问题快速答案

### Q1：为什么不用 InheritableThreadLocal 解决 @Async 跨线程问题？

`InheritableThreadLocal` 只在**线程创建时**从父线程继承——线程池里的工作线程是在服务启动时由第一个触发线程池初始化的线程创建的，不是由每个 HTTP 请求线程创建的。池里的线程被复用时不会重新继承。结果更糟：某次请求线程恰好创建了池线程，**那个创建者的 RbacContext** 会永久地留在那条工作线程上，下次不相关请求触发异步任务时，读到的是完全错误的身份——这是跨请求身份泄漏，比"永远 null"更危险。

### Q2：TaskDecorator 是怎么工作的？

Spring `@EnableAsync` + `AsyncConfigurer` 组合：当提交线程调用 `@Async` 方法时，Spring 的代理层在提交线程上调用 `decorator.decorate(runnable)`（此时 ThreadLocal 还能读到），返回一个包装的 Runnable，在执行线程上运行时先 `set()` 再 `run()` 再 `finally clear()`。这是 Micrometer Tracing、Spring Security 的 `DelegatingSecurityContextRunnable` 传播 TraceContext/SecurityContext 的标准做法。

### Q3：为什么集成测试不用 @Transactional？

1. **异步写在独立事务**：`@Async` 方法在另一条线程、另一个 JDBC 连接上执行，有自己的事务，rollback 够不到
2. **MySQL REPEATABLE READ 快照**：如果测试方法开了事务，第一条 SELECT 就会定格快照；异步线程随后 commit 的新行，在本事务内的轮询查询永远看不到——不是变慢，是永远等不到

### Q4：双分支轮询为什么不直接 Thread.sleep(固定时间)？

固定 sleep 是"写死等待时间"——太短可能 async 还没跑完，太长浪费时间。
轮询有相同问题：只轮询一种终态（如"行是否出现"），在 NOT NULL 约束环境下行永远不会出现，要等满超时才能"确认"。
双分支同时轮询两种终态，谁先出现就退出——在 NOT NULL 场景下 WARN 出现后立即退出，不等超时，测试快几秒。

### Q5：@WebMvcTest 里为什么要 mock SqlSessionTemplate？

`@MapperScan` 是 `@Import`-级别的注册，不受 `@WebMvcTest` 的切片过滤器影响，容器刷新时 `MapperFactoryBean.checkDaoConfig()` 要求存在 `SqlSessionFactory`/`SqlSessionTemplate`（切片上下文没有）。用 `@MockitoBean(answers = RETURNS_DEEP_STUBS)` 让 `mock.getConfiguration()` 返回深层 stub，使 `checkDaoConfig` 内部的 `hasMapper()/addMapper()` 能安全空跑。

### Q6：防枚举攻击是什么？

"订单不存在"和"商家无权查询他人订单"返回完全一样的错误码和消息。
如果区分（比如不存在返回 404，无权返回 403），攻击者通过枚举订单号就能知道哪些订单真实存在——这是信息泄露，可用于精准攻击。统一返回 `not_found`，攻击者无法区分两种情况。
