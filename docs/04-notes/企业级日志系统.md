# 企业级日志系统

本文说明 LocalLife 的集中式日志系统设计、启动方式和排障流程。目标不是把日志写到某个 `logs/app.log` 文件里，而是采用容器化项目更常见的链路：

```text
应用 stdout/stderr
  → Docker container logs
  → Promtail 采集、脱敏、打标签
  → Loki 持久化存储和保留策略
  → Grafana 查询、Dashboard、日志告警
```

业务审计数据单独保留在 MySQL：

```text
agent_session / agent_message / tool_audit_log / hitl_approval / langgraph_checkpoint
```

运行日志解决“系统哪里报错”；审计表解决“Agent 对谁做了什么、为什么做、谁审批”。

## 1. 架构组件

| 组件 | 作用 | 本项目位置 |
| --- | --- | --- |
| 应用日志 | Java/Python 输出到 stdout | 三个应用容器 |
| Promtail | 发现 Docker 容器日志、脱敏、打 `env/service/container/level` 标签 | `infra/promtail/promtail-config.yml` |
| Loki | 日志集中存储、索引、保留和日志告警规则 | `infra/loki/loki-config.yml` |
| Alertmanager | 接收 Loki Ruler 告警 | `infra/alertmanager/alertmanager.yml` |
| Grafana | 统一排障入口，查询 Prometheus、Loki、Zipkin | `infra/grafana/provisioning/` |
| Zipkin | Trace 可视化 | `infra/docker-compose.dev.yml` |
| MySQL 审计表 | Agent 会话、工具、审批和 checkpoint | `local-life-copilot/src/main/resources/db/migration/V1__init_copilot_tables.sql` |

官方参考：

- Kubernetes Logging Architecture: https://kubernetes.io/docs/concepts/cluster-administration/logging/
- Docker logging drivers: https://docs.docker.com/engine/logging/configure/
- OpenTelemetry Logs: https://opentelemetry.io/docs/specs/otel/logs/

## 2. 启动方式

启动应用和可观测性套件：

```bash
cd infra
docker compose -f docker-compose.dev.yml --profile app --profile observability up -d
```

只启动日志系统相关组件：

```bash
cd infra
docker compose -f docker-compose.dev.yml --profile observability up -d loki promtail alertmanager grafana
```

访问入口：

| 服务 | 地址 |
| --- | --- |
| Grafana | `http://localhost:3000`，默认 `admin/admin` |
| Loki API | `http://localhost:3100` |
| Alertmanager | `http://localhost:9093` |
| Zipkin | `http://localhost:9411` |

Grafana 会自动 provision 三个数据源：

- `Prometheus`
- `Loki`
- `Zipkin`

并自动加载 Dashboard：`LocalLife Logs`。

## 3. 持久化和保留策略

Loki 使用 Docker volume 持久化：

```yaml
volumes:
  loki_data:
  promtail_positions:
  alertmanager_data:
```

日志保留周期由 `.env` 控制：

```env
LOKI_RETENTION_PERIOD=168h
```

推荐：

| 环境 | 建议保留 |
| --- | --- |
| `dev` | 7 天 |
| `staging` | 14-30 天 |
| `prod` | 30-90 天，按公司合规要求决定 |

Loki compactor 会按 `retention_period` 清理旧日志。Promtail 的 `positions.yml` 也持久化，容器重启后不会从头重复采集。

## 4. 结构化日志

### Python Agent

`copilot-agent-service/config/logging.py` 已配置 `structlog` JSON 输出，关键字段：

```json
{
  "event": "tool_failed",
  "level": "warning",
  "timestamp": "...",
  "service": "copilot-agent",
  "env": "dev",
  "trace_id": "...",
  "http_method": "POST",
  "http_path": "/chat"
}
```

每个 HTTP 请求都会生成或继承 `X-Trace-Id`，并写入所有 Agent 日志。

### Java Server / Java MCP

Java 仍使用 Logback 控制台输出，但格式已包含：

```text
timestamp level thread traceId/spanId logger message
```

`local-life-server`：

- `TraceIdFilter` 会读取上游 `X-Trace-Id`，写入 MDC。
- 响应头会带回 `X-Trace-Id`。

`local-life-copilot`：

- `RequestTraceFilter` 会读取 Agent 传来的 `X-Trace-Id`，写入 MDC。
- `LocalLifeInternalClient` 调 Java 主服务时继续透传 `X-Trace-Id`。

### Promtail 标签

Promtail 从 Docker metadata 生成低基数标签：

```text
env
service
container
compose_service
compose_project
level
```

注意：`trace_id` 不作为 Loki label。traceId 基数极高，做 label 会让索引膨胀。正确做法是保留在日志正文/JSON 字段里，用 Grafana 搜索或 derived field 跳转。

## 5. 日志和 Trace 关联

调用链：

```text
Browser / curl
  → copilot-agent-service 生成 X-Trace-Id
  → McpClient 透传 X-Trace-Id
  → local-life-copilot RequestTraceFilter 写 MDC
  → LocalLifeInternalClient 透传 X-Trace-Id
  → local-life-server TraceIdFilter 写 MDC
```

排障时：

1. 在浏览器响应头、SSE 日志或 Agent JSON 日志里拿 `trace_id`。
2. Grafana Explore 选择 Loki。
3. 查询：

```logql
{env="dev"} |= "你的 trace_id"
```

如果 Java 主服务也上报 Zipkin span，可通过 Grafana derived field 跳转到 Zipkin。

## 6. 日志告警

Loki Ruler 规则在：

```text
infra/loki/rules.yml
```

当前内置告警：

| 告警 | 触发条件 |
| --- | --- |
| `LocalLifeErrorLogsHigh` | 某服务 5 分钟内出现多条 error/exception/failed/失败/异常 日志 |
| `CopilotAgentToolFailures` | Agent 5 分钟内多次 `tool_failed` |
| `CopilotRagUnavailable` | Milvus、Embedding、Reranker 或知识库入库异常 |
| `HitlApprovalWriteFailed` | HITL 审批写入失败 |

查看告警：

```text
http://localhost:9093
```

开发环境的 Alertmanager receiver 是 `local-dev`，默认不外发通知。生产可以改为企业微信、飞书、Slack、PagerDuty 或邮件。

## 7. 敏感信息治理

两层治理：

1. 应用侧：Python `structlog` 会对 key 名包含 `api_key/password/secret/token/authorization` 的字段打码，并对手机号、`sk-*`、Bearer token 做脱敏。
2. 采集侧：Promtail 再做一层正则 `replace`，防止 Java/Python 某处误打出敏感值。

禁止记录：

- 完整 API Key
- `Authorization` Header
- 数据库密码
- 完整手机号
- 完整 LLM Prompt 中的隐私数据
- 真实支付渠道密钥

允许记录：

- `user_id`
- `merchant_id`
- `session_id`
- `thread_id`
- `trace_id`
- 订单号、支付单号在开发演示环境可记录；生产环境要按公司数据分级策略决定是否脱敏。

## 8. 多环境隔离

通过 `.env` 配置：

```env
DEPLOY_ENV=dev
```

Promtail 会把 `env` 写入 Loki 标签。查询时必须带环境：

```logql
{env="dev", service="copilot-agent"}
{env="staging", service="local-life-server"}
{env="prod", service="local-life-copilot"}
```

生产建议：

- 每个环境独立 Loki 或至少独立 tenant/bucket。
- `prod` 禁止和 `dev` 共用 Grafana 管理员账号。
- `prod` 日志访问必须接入 SSO 和 RBAC。

## 9. 生产排障入口

统一入口：Grafana `LocalLife Logs` Dashboard。

常用 LogQL：

查某服务最近日志：

```logql
{env="dev", service="copilot-agent"}
```

查错误：

```logql
{env="dev"} |~ "(?i)(error|exception|failed|失败|异常)"
```

按 traceId 串全链路：

```logql
{env="dev"} |= "trace_id_value"
```

查 Agent 工具失败：

```logql
{env="dev", service="copilot-agent"} |= "tool_failed"
```

查 MCP 错误：

```logql
{env="dev", service="local-life-copilot"} |~ "(?i)(permission|tool|error|exception)"
```

查 RAG 异常：

```logql
{env="dev", service="copilot-agent"} |~ "milvus_unavailable|rag_no_candidates|rag_below_threshold|knowledge_base_ingest_failed"
```

查 HITL 写入失败：

```logql
{env="dev", service="copilot-agent"} |= "hitl_approval_write_failed"
```

如果是 Agent 业务行为问题，还要查 MySQL 审计表：

```sql
SELECT role, step_index, content, tool_calls, tool_results, created_at
FROM agent_message
WHERE session_id = ?
ORDER BY step_index, id;

SELECT tool_name, status, duration_ms, error_msg, created_at
FROM tool_audit_log
WHERE session_id = ?
ORDER BY created_at;

SELECT id, action_type, status, reason, approver_id, created_at, approved_at
FROM hitl_approval
WHERE session_id = ?
ORDER BY created_at DESC;
```

## 10. 本项目和企业级标准的差距

已补齐：

- 集中式日志：Loki + Promtail + Grafana
- 持久化和保留策略：Loki volume + compactor retention
- 结构化：Python JSON logs，Java 带 traceId 的规范化 console pattern
- Trace 关联：`X-Trace-Id` 从 Agent 传到 MCP，再传到 Java 主服务
- 日志告警：Loki Ruler + Alertmanager
- 敏感信息治理：应用侧和采集侧双层脱敏
- 生产排障入口：Grafana Dashboard + LogQL
- 多环境隔离：`DEPLOY_ENV` 标签

仍可继续增强：

- Java 日志改为完整 JSON encoder。
- 引入 OpenTelemetry Collector，统一 logs/traces/metrics 出口。
- 生产 Alertmanager 接入真实通知渠道。
- Loki 使用对象存储（S3/MinIO）而不是单机 filesystem。
- Grafana 接入 SSO/RBAC。
- 对 LLM prompt 和 tool arguments 做更细的数据分级脱敏。
