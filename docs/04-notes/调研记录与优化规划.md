## 三、调研综合结论（批判性整理）

### 3.1 哪些建议是真正有价值的（取）

| 建议来源 | 建议内容 | 对我们的价值 | 是否可信 |
|---------|---------|------------|---------|
| AgentGuide | 量化业务价值：「自动化率80%」 | ✅ 我们需要真实数字，不能编造 | 高可信 |
| Agent Evals 指南 | eval 集要包含「永远过不了的 case」+ CI 门禁 | ✅ 我们现在 50 条全能过，不真实 | 高可信 |
| 字节面经 | RAG 召回率必须有数字 | ✅ 我们有 73.3%，但需要来自真实测试 | 高可信 |
| 简历写法文章 | 四层表达 + 量化结果 | ✅ 我们描述框架多，结果少 | 中可信 |

### 3.2 哪些建议需要批判性看待（补短/辩证）

| 建议 | 盲目接受的风险 | 我们的立场 |
|------|-------------|---------|
| 「要加多Agent（Supervisor/Planner/Executor）」 | 为加而加，增加复杂度但不解决业务问题 | **我们不需要**：单 Agent + HITL 已经满足商家/客服场景 |
| 「一定要用 LangSmith」 | 依赖外部服务，不如自建 | **我们有更好的**：自建 tool_audit_log + Prometheus，可追溯性更强 |
| 「Recall@5 必须 90%+」 | 可能是针对搜索引擎场景的标准，不适用所有场景 | **73% 是真实数字**，我们应该展示优化过程（73% → 85%），而不是直接宣称 90% |
| 「只用 Function Calling，不用 ReAct」 | 忽视了复杂多步推理场景 | **两者各有用途**：简单查询 Function Calling，复杂排查 ReAct。我们的 Fast Path 就是这个思路 |
| 「加 Mem0 增强记忆」 | 我们的核心场景是订单排查，不是长期陪伴 | **当前不需要**：session 内的工具调用记录已经够用，Mem0 适合 B2C 长期对话 |

### 3.3 与优秀项目对比（我们的真实优劣势）

**我们比 yu-ai-agent / 小傅哥 项目强的地方**：
1. ✅ **真实完整的业务后端**：有秒杀、支付、MQ Outbox 等完整系统，不是 toy 项目
2. ✅ **MCP 实现更深**：L1-L4 工具分级 + RBAC + 工具审计 + 限流，不只是调 API
3. ✅ **HITL 完整流程**：MySQL Checkpoint + 审批恢复，生产级设计
4. ✅ **评测体系**：50 条用例 + 6 项指标（虽然还不完整）
5. ✅ **可观测性**：Prometheus 12 个业务指标 + traceId 跨服务

**我们比优秀项目差的地方**：
1. ❌ **没有真实 before/after 数据**：P99 18.5s 是估算不是实测；Recall@5 73.3% 是估算
2. ❌ **评测不完整**：没有 LLM-as-judge；没有对抗性测试；没有 CI 门禁
3. ❌ **缓存穿透防护缺失**：没有布隆过滤器
4. ❌ **延迟双删不完整**：只有一次删除
5. ❌ **Outbox 死信无自动恢复**
6. ❌ **RAG 召回纯向量**：没有 BM25 混合检索
7. ❌ **Prompt Caching 未使用**：成本节省没做
8. ❌ **简单查询走全 ReAct 浪费**：没有快速路由

- Q6: Agent 的上下文管理和记忆机制设计，如何避免上下文过长或信息污染

  → 面试必答：Bi-Encoder 近似语义相似度，Cross-Encoder 联合编码更精确，代价是不能预计算
- Q11: 如何评估 Rerank 的有效性，常用哪些指标？
  → 我们需要准备：MRR、NDCG、Recall@K before/after
- **Q13: 工具调用的安全控制与敏感接口限制如何实现？**
  → 我们的答案：L1-L4 工具分级 + RBAC ThreadLocal 身份注入 + 高风险工具 HITL
- **Q14: ReAct 与 Plan-and-Execute 有什么区别，分别适用于什么场景，为什么在项目中会选择 ReAct？**
  → 我们的答案：复杂多步订单排查 → ReAct（工具结果影响下一步决策）；简单数据查询 → Fast Path（无需推理）
- **Q20: MCP 和 Skills 有什么区别？它们分别解决什么问题？**
  → MCP: 工具标准化协议（我们实现了）；Skills: 编码特定领域能力（我们没有）

**非常完整的答案模板**，关键内容：

**Q5: LLM 如何学会调用工具？（关键亮点）**
```
2026年主流模型（Claude/GPT-5/Qwen3/DeepSeek-V3）原生支持 Function Calling
并支持「并行工具调用」（parallel tool calls）——一次返回多个 tool_use block，
由调用方并发执行后批量回填 tool_result，链路延迟可降到 1/N
```
→ **这正是我们「并行工具执行」优化的面试理论依据！**

**Q4: 短期 vs 长期记忆设计**
```
短期：context window + 摘要压缩（每N轮总结为摘要）
长期：向量库（Milvus）+ 结构化存储（Redis/DB）混合
Mem0: 自动提取+更新+检索，Zep: 会话记忆+事实提取
```
→ 我们有 agent_message 表（短期）+ Milvus RAG（长期），但没有跨会话的 Mem0

**Q: 1/N 延迟降低的理论依据**
