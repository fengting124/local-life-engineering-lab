## 三、调研综合结论（批判性整理）

### 3.1 哪些建议是真正有价值的（取）

| 建议来源 | 建议内容 | 对我们的价值 | 是否可信 |
|---------|---------|------------|---------|
| AgentGuide | 量化业务价值：「自动化率80%」 | ✅ 我们需要真实数字，不能编造 | 高可信 |
| Agent Evals 指南 | eval 集要包含「永远过不了的 case」+ CI 门禁 | ✅ 我们现在 50 条全能过，不真实 | 高可信 |
| 字节面经 | RAG 召回率必须有数字 | ✅ 我们有 73.3%，但需要来自真实测试 | 高可信 |
| 简历写法文章 | 四层表达 + 量化结果 | ✅ 我们描述框架多，结果少 | 中可信 |

### 3.2 哪些建议需要批判性看待（补短/辩证）

| 建议 | 盲目接受的风险 | 我们的立场 |
|------|-------------|---------|
| 「要加多Agent（Supervisor/Planner/Executor）」 | 为加而加，增加复杂度但不解决业务问题 | **我们不需要**：单 Agent + HITL 已经满足商家/客服场景 |
| 「一定要用 LangSmith」 | 依赖外部服务，不如自建 | **我们有更好的**：自建 tool_audit_log + Prometheus，可追溯性更强 |
| 「Recall@5 必须 90%+」 | 可能是针对搜索引擎场景的标准，不适用所有场景 | **73% 是真实数字**，我们应该展示优化过程（73% → 85%），而不是直接宣称 90% |
| 「只用 Function Calling，不用 ReAct」 | 忽视了复杂多步推理场景 | **两者各有用途**：简单查询 Function Calling，复杂排查 ReAct。我们的 Fast Path 就是这个思路 |
| 「加 Mem0 增强记忆」 | 我们的核心场景是订单排查，不是长期陪伴 | **当前不需要**：session 内的工具调用记录已经够用，Mem0 适合 B2C 长期对话 |

### 3.3 与优秀项目对比（我们的真实优劣势）

**我们比 yu-ai-agent / 小傅哥 项目强的地方**：
1. ✅ **真实完整的业务后端**：有秒杀、支付、MQ Outbox 等完整系统，不是 toy 项目
2. ✅ **MCP 实现更深**：L1-L4 工具分级 + RBAC + 工具审计 + 限流，不只是调 API
3. ✅ **HITL 完整流程**：MySQL Checkpoint + 审批恢复，生产级设计
4. ✅ **评测体系**：50 条用例 + 6 项指标（虽然还不完整）
5. ✅ **可观测性**：Prometheus 12 个业务指标 + traceId 跨服务

**我们比优秀项目差的地方**：
1. ❌ **没有真实 before/after 数据**：P99 18.5s 是估算不是实测；Recall@5 73.3% 是估算
2. ❌ **评测不完整**：没有 LLM-as-judge；没有对抗性测试；没有 CI 门禁
3. ❌ **缓存穿透防护缺失**：没有布隆过滤器
4. ❌ **延迟双删不完整**：只有一次删除
5. ❌ **Outbox 死信无自动恢复**
6. ❌ **RAG 召回纯向量**：没有 BM25 混合检索
7. ❌ **Prompt Caching 未使用**：成本节省没做
8. ❌ **简单查询走全 ReAct 浪费**：没有快速路由

  → 面试必答：Bi-Encoder 近似语义相似度，Cross-Encoder 联合编码更精确，代价是不能预计算
- Q11: 如何评估 Rerank 的有效性，常用哪些指标？
  → 我们需要准备：MRR、NDCG、Recall@K before/after
- **Q13: 工具调用的安全控制与敏感接口限制如何实现？**
  → 我们的答案：L1-L4 工具分级 + RBAC ThreadLocal 身份注入 + 高风险工具 HITL
- **Q14: ReAct 与 Plan-and-Execute 有什么区别，分别适用于什么场景，为什么在项目中会选择 ReAct？**
  → 我们的答案：复杂多步订单排查 → ReAct（工具结果影响下一步决策）；简单数据查询 → Fast Path（无需推理）
- **Q20: MCP 和 Skills 有什么区别？它们分别解决什么问题？**
  → MCP: 工具标准化协议（我们实现了）；Skills: 编码特定领域能力（我们没有）

#### 帖子 6：字节AI Agent框架面试完整版（7544字）

**非常完整的答案模板**，关键内容：

**Q5: LLM 如何学会调用工具？（关键亮点）**
```
2026年主流模型（Claude/GPT-5/Qwen3/DeepSeek-V3）原生支持 Function Calling
并支持「并行工具调用」（parallel tool calls）——一次返回多个 tool_use block，
由调用方并发执行后批量回填 tool_result，链路延迟可降到 1/N
```
→ **这正是我们「并行工具执行」优化的面试理论依据！**

**Q4: 短期 vs 长期记忆设计**
```
短期：context window + 摘要压缩（每N轮总结为摘要）
长期：向量库（Milvus）+ 结构化存储（Redis/DB）混合
Mem0: 自动提取+更新+检索，Zep: 会话记忆+事实提取
```
→ 我们有 agent_message 表（短期）+ Milvus RAG（长期），但没有跨会话的 Mem0

**Q: 1/N 延迟降低的理论依据**
→ 「并发执行后批量回填 tool_result，链路延迟可降到 1/N」—— 我们的并行工具执行优化从理论上是正确的

### 7.3 面经里的高频八股与项目追问

#### RAG 高频追问

- RAG 整体流程：query rewrite、embedding、向量召回、BM25、rerank、上下文拼接、生成、引用。
- chunk 策略：chunk size、overlap、父子分块、不同文档格式、表格/条款/代码块如何避免切散。
- 召回错误：query 改写、metadata filter、reranker、负样本、阈值、人工反馈闭环。
- 漏召问题：Hybrid Search、同义词、BM25 兜底、多路召回、RRF 融合。
- 评估指标：Recall@K、MRR、NDCG、faithfulness、answer relevance、hallucination rate。
- RAG vs 微调：RAG 解决知识与事实，微调解决风格和工具调用能力。

#### Agent 高频追问

- Agent 核心组件：LLM、Planning、Memory、Tools、Action、Observation、Guardrails。
- ReAct 流程：Thought -> Action -> Observation 循环，以及何时不适合 ReAct。
- ReAct vs Plan-and-Execute：不确定、短链路适合 ReAct；长任务、可拆解任务适合 Plan-Execute。
- 工具调用：Function Calling 和 MCP 的边界；tool schema、参数校验、错误返回、重试、超时。
- 上下文管理：短期记忆、摘要压缩、长期记忆、上下文污染、token 成本。
- 死循环：同工具同参数重复调用、最大步数、反思节点、失败终止条件。
- 安全：prompt injection、tool poisoning、权限最小化、HITL、审计日志。

#### Java 后端/秒杀高频追问

- 缓存一致性：Cache Aside、延迟双删、binlog/MQ、读写并发窗口、失败补偿。
- 缓存穿透/击穿/雪崩：布隆过滤器、互斥锁、逻辑过期、随机 TTL。
- 秒杀：Redis Lua 原子扣减、库存预扣、幂等、限流、异步下单、MQ 堆积。
- MQ：重复消费、顺序消息、事务消息、Outbox、死信恢复、最终一致性。
- MySQL：间隙锁、事务隔离级别、索引、慢 SQL、分库分表路由键。
- Java：线程池、锁、AQS、JVM 内存区域、GC、NIO。

### 8.1 对前序调研的审计结论

前序调研不是完全错误，但存在三类需要纠偏的问题：

| 问题 | 具体表现 | 纠偏原则 |
|------|----------|----------|
| 未实测数字容易被误用 | 文档中出现 `P99 18.5s -> 7s`、`Recall@5 73% -> 85%+`、`成本 -80%` 等表达 | 这些只能作为待验证假设或行业目标，不能写成项目结果。最终简历和面试文档只能使用本项目真实跑出来的数据 |

| 实施顺序一度偏离核心指令 | 第七节曾把 RAG/Agent 指标链放到 Java 后端可靠性之前 | 最终执行顺序以核心指令为准：先补 LocalLife Server 的经典后端可靠性能力，再推进 Agent/RAG 优化。原因是 Java 后端是主项目底座，且布隆/缓存/MQ 是高频八股与真实生产问题 |

### 8.2 最终规划的判断标准

后续每个功能是否进入规划，只看四个问题：

1. **业务痛点是否真实存在**：没有它时，系统在什么场景会失败或表现很差。
2. **面试是否能深挖 3 到 5 层**：能否讲清 Why、Tradeoff、极端情况、替代方案、失败兜底。
3. **指标是否可真实测量**：能否通过 Locust、evals、日志、usage 字段或数据库统计得到数据。
4. **是否服务统一叙事**：是否强化“本地生活交易平台 + 企业级 Copilot”的主线，而不是两个拼凑项目。

不满足这些条件的内容，即使很新，也不进入 P0/P1。

### 8.3 最终项目叙事：避免“两个项目拼一起”

最终简历和面试叙事应统一为：

> 基于本地生活交易平台构建企业级商家/客服 Copilot。底层平台覆盖秒杀、订单、支付、Outbox、缓存一致性、搜索、分表、限流等后端核心能力；Copilot 通过 MCP 工具层连接真实业务数据，支持 RBAC、审计、HITL、RAG 知识检索、Agent 评测与性能优化。项目重点不是“接了大模型 API”，而是把 AI Agent 落到真实业务系统、权限边界、可靠性和可观测指标上。

这个叙事解决三件事：

- 面向 Java 后端岗：证明不是只会 AI 包装，底层有秒杀、MQ、缓存、分表、限流等工程能力。
- 面向 AI 应用岗：证明不是教程 RAG/Agent demo，而是连接真实业务工具、权限、安全、HITL 和评测体系。
- 面向项目深挖：面试官从任意关键词追问，都能落到代码、指标、Tradeoff 和故障场景。

### 8.4 最终优先级路线

#### Phase 0：基线冻结与可测口径

在继续优化前，先确认基线，否则后续 before/after 没有可信度。

| 任务 | 产出 | 为什么必须先做 |
|------|------|----------------|
| 确认当前代码状态和可运行环境 | `git status`、服务启动说明、依赖状态 | 防止在半成品状态上继续堆功能 |
| 建立后端压测脚本口径 | Locust 场景：门店详情、秒杀抢券、支付回调、订单查询 | 布隆过滤器、缓存、Outbox 都需要有性能或失败恢复数据 |
| 建立 Agent eval 口径 | `python -m evals.metrics --real`，区分 query/diagnosis/knowledge/boundary | 并行工具、Fast Path、Hybrid RAG、LLM-as-Judge 都需要同一套评测口径 |
| 建立成本统计口径 | 记录 LLM usage、cache read/write tokens、平均 input/output tokens | Prompt Cache 不能只说节省，必须能从 usage 字段算出来 |

量化数据写法要求：

- 如果只完成代码但未跑压测，文档写“实现完成，数据待测试”，不能写提升百分比。
- 如果跑的是 mock eval，不能写成真实 Agent 指标。
- 如果跑的是单机开发环境，要标明环境条件，不能包装成生产数据。

#### Phase 1：LocalLife Server 后端可靠性（先做）

这部分是主项目底座，也是 Java 后端面试最稳定的深挖区。

| 优化 | 业务痛点 | 设计要求 | Tradeoff 必须准备 | 验证方式 |
|------|----------|----------|-------------------|----------|
| 布隆过滤器 | 非法 shopId/userId/couponId 大量请求会穿透缓存打到 DB | Redisson `RBloomFilter`；启动时从 DB 批量加载现有 ID；新增数据同步 add；误判率 1% | 误判率和内存占用；不支持删除；Redis 宕机后重建；为什么不只缓存空值 | 构造不存在 ID 压测，统计 DB 查询次数下降；记录布隆命中/拦截日志 |
| 延迟双删 | 写 DB 和删缓存之间存在并发读旧值回填缓存的窗口 | 写方法先删缓存，事务写 DB，事务后 `@Async` 延迟 500ms 再删 | 500ms 依据；双删额外 Redis 开销；不能绝对保证一致；为什么不用 Canal/binlog | 并发更新和读取压测，统计旧值缓存残留概率；至少做日志验证 |
| Outbox 死信自动恢复 | RocketMQ 短暂不可用后消息进入 FAILED，只靠人工恢复不适合生产 | V9 加 `auto_retry_count`；每小时扫 `FAILED and auto_retry_count < 3`；每批 100；重置为 PENDING | 自动恢复可能重复投递；消费者幂等前提；为什么不无限重试；如何告警 | 模拟发送失败到 FAILED，再恢复 MQ，验证自动回到 SENT；统计恢复次数 |

这三项不算“新”，但非常适合面试深挖。它们的价值不在于技术名词，而在于能把缓存、Redis、MQ、事务、幂等、故障恢复讲完整。

#### Phase 2：Agent 延迟、成本、死循环与工具层治理

这部分体现 AI 应用工程深度，重点是让 Agent 从 demo 变成可控系统。

| 优化 | 业务痛点 | 设计要求 | Tradeoff 必须准备 | 验证方式 |
|------|----------|----------|-------------------|----------|
| 并行工具执行 | 订单异常排查常需同时查订单、支付、券、MQ；串行工具导致长尾延迟 | `tool_node()` 对同一轮 LLM 返回的多个独立 tool call 使用 `asyncio.gather(return_exceptions=True)` | 并行只适合无依赖工具；有依赖的步骤仍需 ReAct 下一轮；失败工具不能影响成功工具结果 | 对包含多工具调用的诊断 case 跑真实 eval，对比端到端 P50/P99 |
| 工具 Schema TTL 缓存 | 每轮 LLM 前都 `tools/list`，工具 schema 变化频率低，HTTP 往返浪费 | `mcp_client.list_tools()` 加 300s TTL；按 role/merchant 权限过滤仍在 Agent 侧执行 | schema 变更延迟生效；管理员热更新工具时如何清缓存 | 统计 list_tools 调用次数和单轮耗时 |
| Prompt Caching | System Prompt 和工具说明长，重复 token 成本高 | `SystemMessage` 增加 `cache_control={"type": "ephemeral"}`；只缓存稳定前缀，不缓存用户消息 | 不同模型支持不同；缓存不保证每次命中；不能把动态权限上下文错误缓存 | 读取 usage 中 cache read/write tokens，计算真实节省 |
| 循环检测 | Agent 可能同工具同参数重复调用，浪费 token 且无法收敛 | 同工具同参数达到 3 次，写入错误 ToolMessage 或 stop_reason，进入 final/reflection | 阈值太低可能误杀正常重试；需要区分参数修正后的重试 | 构造工具失败 case，验证不会无限循环 |
| Fast Path | 简单确定性指标查询不需要 ReAct，多轮推理反而慢且不稳定 | 在 `/chat` 前置识别“今天/昨天/本月 + 订单/GMV/销售额”等，直接调 `shop_metrics_query` | 规则覆盖有限；误判复杂问题会损失解释能力；需要 fallback 到 ReAct | 简单查询 case 对比 Fast Path 与 ReAct 延迟 |

新度来自“Agent/Workflow 混合架构 + Agent Harness 治理”，不是盲目加 Multi-Agent。

#### Phase 3：RAG 召回、评测与幻觉治理

RAG 是牛客 AI 面经里最容易深挖的部分，必须从“会用向量库”升级为“能解释召回和评测”。

| 优化 | 业务痛点 | 设计要求 | Tradeoff 必须准备 | 验证方式 |
|------|----------|----------|-------------------|----------|
| Hybrid RAG | 纯向量检索对专有名词、订单术语、短查询不稳定 | BM25 + Milvus 多路召回；RRF 融合；再用 Cross-Encoder rerank | BM25 对语义弱；向量对关键词弱；RRF 的 k 值；BM25 内存索引重启重建 | `--category knowledge` 跑真实 eval，记录 Recall@5、MRR 或关键词覆盖变化 |
| Chunk 策略复盘 | 面经会问文档数、chunk 数、切片粒度、是否切散条款 | 保留 500/50 作为初始方案，但必须记录文档数、chunk 数，并准备 250/500/1000 对比实验 | 小 chunk 召回准但上下文碎；大 chunk 上下文完整但噪声多 | 统计不同 chunk size 的 Recall@5 和 rerank top score |
| LLM-as-Judge | 只看关键词覆盖不足以判断回答是否忠实于证据 | `evals/judge.py` 评估 faithfulness、relevance、hallucination_detected；保留原规则指标 | LLM judge 有偏差；需要固定 prompt、温度和抽样；不能替代人工抽检 | 输出 judge report，标注模型和参数 |
| 证据约束回答 | RAG 幻觉主要来自无证据生成 | 对低相关度检索结果拒答；回答中带来源；无文档时明确说明知识库未覆盖 | 拒答会降低覆盖率；过低阈值会引入幻觉，过高会误拒答 | boundary/knowledge case 中统计误拒答和幻觉 |

RAG 的“新度”不靠复杂名词，而靠完整链路：多路召回、精排、证据约束、评测、失败样例。

#### Phase 4：只保留有明确价值的扩展项

| 扩展项 | 最终定位 | 原因 |
|--------|----------|------|
| 多 LLM Provider | P2，可做但不优先 | 有助于模型对比和成本优化，但当前没有证明单模型成为瓶颈 |
| Multi-Agent | 暂不做 | 当前任务复杂度未证明单 Agent 不够；加了会增加调度、延迟和一致性成本 |
| Mem0/Zep 长期记忆 | 暂不做，保留为未来方向 | 商家偏好记忆有业务价值，但当前核心痛点是工具、RAG、评测和可靠性 |
| A2A/Skills | 只写入知识储备，不实施 | 面试可被问到概念区别，但不一定要在项目里落地 |
| RAG-Fusion/HyDE | P3 实验项 | 对召回有帮助，但先做 BM25 + 向量 + RRF 这种更稳定可解释的方案 |

### 8.5 文档与面试指南规划

`docs/04-notes/深度拷打与面试指南.md` 应在代码优化和可测数据完成后写，不能先写“漂亮数字”。结构固定如下：

```text
### X.X 场景名称

业务场景：这个技术解决什么真实业务问题
技术实现：精确到文件、类、方法
为什么这样做：和 2-3 个替代方案对比
极端情况：至少 2 个失败/降级/恢复场景
量化数据：真实测试数据；没有就写“待测试后填入”
五层拷打 Q&A：从是什么到为什么、代价、异常、改进
```

章节顺序建议：

| 章节 | 核心技术 | 面试价值 |
|------|----------|----------|
| 1. 门店详情与缓存穿透 | 布隆过滤器、缓存空值、Redis 宕机重建 | Redis/缓存八股 + 生产异常 |
| 2. 缓存一致性 | Cache Aside、延迟双删、事务后异步删除 | 读写并发窗口、Tradeoff |
| 3. 秒杀抢券 | Lua 原子扣减、限流、幂等、异步下单 | 高并发后端主战场 |
| 4. 支付与 Outbox | 本地消息表、RocketMQ、死信恢复、幂等消费 | 分布式事务与最终一致性 |
| 5. 订单分片 | ShardingSphere、user_id 路由键、广播查询代价 | 分库分表设计 |
| 6. MCP 工具层 | JSON-RPC、Tool Schema、RBAC、审计 | AI Agent 与后端系统连接点 |
| 7. Agent Harness | 并行工具、循环检测、Fast Path、HITL | 新度与工程治理 |
| 8. RAG 检索与评测 | Hybrid RAG、Rerank、LLM-as-Judge、证据约束 | AI 应用面经高频深挖 |
| 9. 可观测性与评测 | Locust、Prometheus、evals、usage 统计 | 真实指标与防吹牛 |

### 8.6 最终实施顺序

最终执行顺序以“核心指令 + 面试证据链 + 可测性”为准：

1. **先冻结基线**：确认服务能跑、评测能跑、压测口径能跑。
2. **先做 LocalLife Server 三项**：布隆过滤器、延迟双删、Outbox 自动恢复。
3. **再做 Agent 工具治理**：并行工具、工具 schema TTL、Prompt Cache、循环检测、Fast Path。
4. **再做 RAG 评测链**：Hybrid RAG、chunk 对比、LLM-as-Judge、证据约束。
5. **最后写深度面试指南**：只写真实代码和真实数据；没有数据就标待测。

### 8.7 当前规划的专业性底线

后续任何人继续工作时，必须遵守：

- 不把“调研文章中的数字”写成“本项目数据”。
- 不把“理论上可降低”写成“已经降低”。
- 不为新度牺牲可解释性，宁愿少做也要能讲穿。
- 不逃避弱点：个人项目、单机压测、BM25 内存索引、布隆过滤器不支持删除、LLM judge 偏差，都要诚实准备话术。
- 不写空泛简历句式：“使用 LangGraph/MCP/RAG”必须替换为“解决了什么业务问题、如何设计、怎么验证、有什么代价”。

## 九、给下一个实施者的简短结论

前序调研的方向是对的：牛客面经确实高频追问 RAG、Agent、MCP、工具调用、安全、评测、缓存一致性和 MQ。需要纠正的是表达方式和实施顺序：不要把未验证数字当成果，不要被“新技术名词”牵着走。最终路线应以 LocalLife 真实业务为底座，先补 Java 后端可靠性，再补 Agent Harness 和 RAG 评测链，所有结果都必须可测试、可复盘、可被面试官追问。
