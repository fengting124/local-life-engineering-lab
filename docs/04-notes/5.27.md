#  发笔记链路

## MQ 事件设计是什么
某个业务事实已经发生，通知其它系统去处理
MQ 负责了异步处理审核，索引和feed。
- feed 是信息流：比如系统把B发的动态放到用户A的关注流里面
	分发的两种模式：
				拉模式：在用户打开关注页的时候临时查询  优点：写入简单，不会提前产生大量数据 缺：查询慢，关注多人时性能差。
				推模式：商家发布内容时，提前把内容写入粉丝的feed收件箱 优点:用户体验好 缺：写入压力大
- 搜索索引：一般用Elasticsearch，是一个搜索引擎和分析引擎

二者都接受短暂的延迟，所以可以使用MQ

**使用MQ的优点**：
- 服务解耦
- 主流程更快
- 消弱流量峰值
- 可重试失败信息

**MQ 事件 = 系统发生了一件事，发出去通知别人**  
  
**事件消费者 = 收到这件事并处理的人或服务**  
  
**Elasticsearch = 高性能搜索引擎**  
  
**分发关注 Feed = 把内容推送到关注者的信息流里**

# 抢券链路

**lua**：是一种轻量级脚本语言。Redis 支持直接执行 Lua 脚本，常用于把多个 Redis 命令合并成一个原子操作

MySQL: 券模板；秒杀场次；订单
Redis：秒杀库存；用户抢券记录
RocketMQ ：抢券成功事件

**为什么 lua 和 redis 能做到原子操作，防止重复抢券**
如果用java多步操作，高并发下会出问题，导致超发或者重复请求绕过判断。
使用lua脚本的时候，中间不会插入其它redis命令。

## **消费者幂等设计** 幂等——同一个操作执行一次和多次，结果一样
幂等键： user_id + coupon_template_id + seckill_session_id 
消费者创建订单之前，必须做这个
存放到 MySQL 中。
为什么要靠唯一的幂等键兜底呢？ 因为先查询再插入不是绝对安全的。

**通俗的设计逻辑：**
	MQ 消息可能重复来。
	消费者每次收到“创建秒杀订单”的消息时，先用 用户ID + 券模板ID + 秒杀场次ID 查一下订单表。
	如果已经有订单，说明这条消息之前处理过，直接当作成功。
	如果没有订单，就创建订单。
	但高并发下，可能两个消费者同时查到没有订单，然后同时插入。
	所以数据库必须加唯一索引，强制同一个用户、同一个券、同一个场次只能有一条订单。
	如果插入时报唯一键冲突，说明另一个线程已经创建成功了。
	这不是业务失败，应该当作消息消费成功，避免 MQ 一直重试。

## 抢券的失败补偿
这个流程，涵盖了 Redis，MQ，MySQL， 延时消息 每一步都有可能失败，失败后都要有补救。
1. 当redis lua 执行失败的时候，无法判断库存和限购，应该直接停下来，返回系统繁忙
2. Lua 成功，但是MQ 发送失败 ，这个时候库存减一，但是因为MQ消息没有发出去，消费者不知道创建订单 
	处理方法：把Redis 库存加回去，把用户从已抢set里面删除，记录失败日志
3. 消费者失败，此时订单会创建延迟
	处理方法：RocketMQ重新发送，若仍然失败则进入死信
4. 订单创建成功但延时关单消息失败，此时未支付的订单无法自动关闭
	 处理方法：定时任务扫描超时订单兜底
5. Redis 和 MySQL 不一致，Redis预扣和MySQL记录的不一致
	现象：redis少了，mysql没有订单，导致了少卖
		mysql 有订单，但redis用户set里面没有记录，可能是用户重复抢了
		redis剩余库存和Mysql 数量对不上  **处理方法：用MySQL 订单反推出真实的库存，修正Redis 中的库存和用户集合**

**补偿优先级**：同步可回滚则回滚 Redis，同步不可确认则写失败记录，异步失败用MQ 重试和死信；最终不一致，用mysql 订单信息对账修复
