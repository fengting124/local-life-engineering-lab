# 给 Codex 的核心指令：这个项目的真实目的和工作要求

## 一、这个项目是什么，最终目的是什么

这个项目叫 **LocalLife × LocalLife Copilot**，是一个**双线项目**：

- `local-life-server`：Java Spring Boot 本地生活交易平台（秒杀/订单/支付/MQ/ES/分表）
- `local-life-copilot` + `copilot-agent-service`：企业级 AI Agent（MCP + LangGraph + RAG + HITL）

**最终目的是两个，缺一不可**：

### 目的 1：通过大厂后端 + AI 应用方向的面试

这意味着：
- 每一个技术决策必须能**说清楚为什么**，不只是「我用了 XXX」
- 每一个数字（P99、Recall@5、成本、QPS）必须**真实可测量**，不能编造
- 每一个弱点必须**诚实承认并有应对话术**，不能回避
- 面试官会追问 5 层深，你写的每一行代码背后都要有「Why」
- **常见死法**：答得太完整像背书、被追问就崩、说了「高并发」但说不出数据

### 目的 2：成为真实可上线的项目，不是玩具 Demo

这意味着：
- 业务逻辑要考虑真实的极端情况（Redis 宕机、MQ 积压、DB 分片挂一个）
- 安全要考虑真实的攻击场景（Prompt Injection、越权查询、重放攻击）
- 性能要有真实的测试数据，不是理论估算
- 错误处理要考虑用户真实会遇到的场景，不能只写 HappyPath

## 二、什么不能做（红线）

**❌ 不能为了堆技术而加功能**
- 加 Mem0 不是因为「别人都用」，是因为「商家跨会话偏好确实有价值」
- 加 Multi-Agent 不是因为「听起来高级」，是因为「当前单 Agent 确实不够用了」
- 每个新功能都必须能回答「没有它的时候出了什么问题？」

**❌ 不能写没有 Tradeoff 分析的代码**
- 布隆过滤器：它能防穿透，但它有误判率（False Positive），面试官会问「你的误判率是多少？设置了几个哈希函数？」
- 延迟双删：它能降低不一致窗口期，但删两次有什么代价？500ms 如何确定？
- 分布式 ID（雪花算法）：时钟回拨问题你怎么处理？

**❌ 不能写无法验证的数字**
- 「P99 从 18.5s 降到 7s」必须是跑 Locust 测出来的，不是估算的
- 「Recall@5 从 73% 提升到 85%」必须是跑 Evals 测出来的
- 没有真实数字的优化只是「声称优化」，面试官一问就穿帮

**❌ 不能忽略安全和权限**
- Agent 的每个工具调用都必须验证身份（merchant 不能查别人的订单）
- 高风险操作（退款、补券）必须有 HITL，不能 AI 自主执行
- 输入必须过 Guardrails（Prompt Injection 防护）

**❌ 不能只写代码不写注释和文档**
- 面试时要能「指着代码讲」，注释是你的演讲稿
- 每个关键决策旁边必须有一句注释说明「为什么不用其他方案」

## 三、每做一件事，必须问自己这三个问题

**Q1: 面试官追问到第3层，我能回答吗？**
```
示例拷打链：
「你用了布隆过滤器」
→ 「布隆过滤器的误判率你设的多少？」
→ 「为什么用这个误判率？太低了有什么代价？」
→ 「如果门店被删除了，布隆过滤器里的数据怎么处理？」
→ 「你用 Redisson 的原因是什么？直接用 Redis 的 Bitmap 不行吗？」
```

**Q2: 这个功能在真实生产中会出什么问题？**
```
布隆过滤器：
- 误判（false positive）：正常门店被误判不存在，用户查不到
- 不支持删除：门店下线后布隆里还有记录
- 重建成本：Redis 宕机后需要从 DB 重建，10万门店重建需要多久？
```

**Q3: 有没有更简单的方案能达到同样效果？为什么没选它？**
```
防缓存穿透：
- 方案A：布隆过滤器（我们选的）
- 方案B：缓存空值（设一个短 TTL 的 null 缓存）
- 方案C：ID 校验前置（先 SELECT COUNT(*) 判断存在性）
为什么选A不选B/C？B会浪费缓存空间且有时间窗口；C会增加DB查询次数
这就是面试时的「Tradeoff 分析」
```

## 四、具体工作要求

### 4.1 LocalLife Server 端

必须做的优化（有真实面试价值，不是堆砌）：

**布隆过滤器**（防缓存穿透，面试必考）
- 用 Redisson 的 RBloomFilter，不要自己实现
- 初始化时从 DB 批量加载所有 shop.id（`@PostConstruct`）
- 误判率设为 0.01（1%），面试时要能说清楚这个数字是怎么来的
- 代码里必须有注释：为什么选误判率 0.01？门店数量规模？内存占用？
- **必须准备的追问**：「门店被删除后怎么办？」→ 回答：布隆过滤器不支持删除，我们定时（每天凌晨）重建，或者接受极低概率的 false positive（已删门店偶尔被允许查 DB，DB 返回 null 后缓存空值兜底）

**延迟双删**（缓存一致性，面试必考）
- 写方法：先删缓存 → 写 DB → 延迟500ms → 再删一次缓存
- 500ms 的依据：MySQL 主从同步延迟通常 < 200ms，500ms 是 2.5 倍余量
- 用 `@Async` 执行延迟删除，不阻塞主流程
- **必须准备的追问**：「延迟双删解决了什么问题？」→ 解决「读线程在写事务未提交时读到旧值并写回缓存」的脏数据窗口期

**Outbox 死信自动恢复**
- 每小时扫 status=FAILED 且 auto_retry_count < 3 的消息
- 分批处理（每批 100 条），避免锁表
- 自动重置为 PENDING 重新投递
- V9 Migration 加 `auto_retry_count` 字段

### 4.2 Copilot Python Agent 端

**并行工具执行**（最高价值优化）
- 改 `tool_node()` 里的串行 for 循环为 `asyncio.gather()`
- 理论依据：2026年主流 LLM 都支持并行工具调用（parallel tool calls），延迟可降到 1/N
- 必须用 `return_exceptions=True`，一个工具失败不影响其他
- 代码注释要写：「哪些工具调用是独立的（可并发），哪些有数据依赖（必须串行）」
- **真实测试**：用 `time.time()` 记录 before/after，放进评测报告

**Prompt Caching**（成本优化）
- 给 SystemMessage 加 `cache_control={"type": "ephemeral"}`
- 测试：观察 Claude API 响应中的 `cache_read_input_tokens` 字段
- 计算节省：平均系统提示 2000 tokens × 0.003$/1k = $0.006/次，缓存后几乎为0

**Hybrid RAG**（召回率改进）
- 新建 `rag/bm25_store.py`，用 rank-bm25 库
- ingest 时同步构建 BM25 索引（内存，重启后需要重建）
- retrieve() 时同时调 Milvus（向量）+ BM25（关键词），RRF(k=60) 融合
- **必须测试**：`python -m evals.metrics --real --category knowledge`，记录 Recall@5 before/after

**LLM-as-Judge**（评测升级）
- 新建 `evals/judge.py`，用轻量模型（claude-haiku）评测
- 评测三个维度：faithfulness（有事实依据）、relevance（切题）、hallucination_detected（是否幻觉）
- 在评测报告里加这三项数字
- **面试话术**：「我们用 LLM-as-judge 评测忠实度，当前幻觉率是 X%，比基线降低了 Y%」

**循环检测**
- 同工具同参数 ≥ 3 次 → 强制停止，触发 final_node
- 日志记录：哪个工具在循环，循环了几次
- 面试说：「这是 Agent Harness 的核心能力，65%的企业AI失败来自Harness缺陷」

**Fast Path 路由**（Agent/Workflow混合）
- 识别简单模式：「今天/昨天/本月/上周 + 卖了多少/订单量/GMV」
- 匹配到 → 直接调 shop_metrics_query 返回，stop_reason="fast_path"
- 不匹配 → 走完整 LangGraph ReAct
- **面试说法**：「我们实现了 Agent/Workflow 混合架构，简单确定性查询走 Fast Path（<500ms），复杂多步推理走 ReAct（P99 ~7s）」
