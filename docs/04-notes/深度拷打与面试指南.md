## 1. 门店详情与缓存穿透防护

### 业务场景

用户在 App 刷到附近门店列表，点进某个门店查看详情。系统每秒有大量门店详情请求。
**问题**：如果攻击者用不存在的 shopId 构造请求（枚举攻击），每个请求都会穿透缓存打到 MySQL，导致 DB 压力暴增。

### 技术实现

**三层防护链**（文件：`ShopService.java`）：

```
请求
  ↓ 布隆过滤器（O(1)，前置拦截）
  ↓ Redis 缓存（L2，TTL 30min）
  ↓ MySQL（真实数据源）
```

**布隆过滤器**（`BloomFilterService.java`）：
- 启动时 `@PostConstruct` 从 DB 加载所有门店 ID（只查 id 字段）
- 参数：100 万期望容量，1% 误判率（FP），内存约 1.2 MB
- Redisson `RBloomFilter` 存 Redis，进程宕机重建只需几秒

**缓存读写**（`ShopCacheService.java`）：
- Key 格式：`cache:shop:detail:{shopId}`，TTL 30min
- 缓存命中 → 直接返回，校验 status 是否 ONLINE

**ShopService.getShopDetail() 三步流程**：
```java
if (!bloomFilterService.mightContainShopId(shopId)) throw SHOP_NOT_FOUND;  // 一定不存在，拦截
Shop cached = shopCacheService.getShopDetail(shopId);
if (cached != null) return toVO(cached);  // 缓存命中
Shop shop = shopMapper.selectById(shopId); // 查 DB
shopCacheService.putShopDetail(shop);       // 写入缓存
```

### 极端情况

**Redis 宕机（布隆过滤器数据丢失）**：
重启时 `@PostConstruct` 重建布隆过滤器，需要扫全表 shop.id，约需 5-30 秒（取决于门店数量）。
这段时间布隆过滤器为空，所有请求都会穿透到 Redis 缓存和 DB，短暂压力增大。
降级方案：布隆过滤器 `tryInit` 失败时跳过，缓存空值兜底（虽然防穿透能力下降）。

**门店被永久关闭（CLOSED 状态）后布隆过滤器仍有记录**：
布隆过滤器不支持删除。CLOSED 的门店 ID 会产生 false positive：
请求通过布隆过滤器 → 查 Redis → 查 DB → DB 返回 CLOSED 状态 → 抛 SHOP_NOT_FOUND。
代价：一次额外 DB 查询。接受程度：关闭门店是低频操作，代价可忽略。

### Tradeoff

| 方案 | 优点 | 缺点 | 选择理由 |
|------|------|------|---------|
| 布隆过滤器（当前）| O(1) 前置拦截，内存 1.2MB | 不支持删除；1% 误判率 | 防穿透最彻底 |
| 只缓存空值 | 简单，支持删除 | 空值 TTL 短（5min），高并发窗口期仍穿透 | 补充手段，不能单独依靠 |
| 接口 ID 校验 | 无误判 | 每次都查 DB，没有减少 IO | 不适合 |

### 五层追问 Q&A

**Q1：布隆过滤器 1% 误判率是怎么确定的？**

> 误判率越低，内存越大。1% 误判率在 100 万容量下只需 1.2MB；
> 如果要 0.1%，需要约 2.4MB（翻倍），收益递减。
> 对于「不存在的门店偶尔通过」这种场景，1% 是合理的 trade-off。
> 进阶：Redisson `tryInit` 的两个参数 `expectedInsertions` 和 `falseProbability`，
> 底层用 `ceil(-n*ln(p) / (ln(2)^2))` 个 bit 和 `ceil(-ln(p)/ln(2))` 个哈希函数。

**Q2：布隆过滤器不支持删除，门店下线了怎么办？**

> 两种情况：
> 1. 门店临时下线（OFFLINE）：布隆过滤器有记录，请求通过 → Redis/DB 返回 OFFLINE 状态 → 抛 SHOP_NOT_FOUND。业务上正确，只是多了几次 IO。
> 2. 门店永久关闭（CLOSED）：同上。代价可接受。
> 如果需要支持精确删除，可升级为 Counting Bloom Filter（每个 bit 改为计数器），内存增加 3-4 倍。

**Q3：Redis 里的 RBloomFilter 是什么数据结构？为什么用 Redis 而不是内存？**

> Redisson 的 RBloomFilter 在 Redis 里存成 `BITSET`，多个哈希函数映射到不同 bit 位。
> 用 Redis 而不是纯内存的原因：应用多实例部署时，内存布隆过滤器各实例独立（需要各自从 DB 加载），
> Redis 共享一份，节省 DB 扫描压力。

**Q4：如果布隆过滤器判断「可能存在」但 DB 也查不到，会发生什么？**

> 这是 false positive（1% 概率）。流程是：通过布隆 → Redis miss → DB 查不到 → 抛 SHOP_NOT_FOUND。
> 攻击者需要猜中 1% 的误判 ID，且每次都要经过 Redis miss + DB 查询，成本较高。
> 可以加空值缓存（5min TTL）作为补充：同一个不存在的 ID 第一次打到 DB，之后走缓存。

**Q5：高并发下，`@PostConstruct` 加载门店 ID 时，新建的门店请求怎么处理？**

> 布隆过滤器初始化期间可能有极短的窗口期（新建门店还没有加载）。
> 解决方案：`createShop()` 成功后立即 `bloomFilterService.addShopId(shop.getId())`（代码已实现）。
> 即使在初始化窗口内，新建门店 ID 也会被实时加入，不会被拦截。

## 2. 缓存一致性：从延迟双删到 Canal+binlog

> **架构演进提要**：本节描述的方案已从「延迟双删」升级为「Canal + binlog 事件驱动」
> （`ShopCacheInvalidationListener.java`），下文先讲清楚旧方案要解决什么问题、
> 有什么不优雅之处，再讲新方案如何更彻底地解决它——这正是面试中「你做过什么架构升级」
> 类问题的标准展开方式：讲清楚 before/after 和升级动机，而不是只背新方案的实现细节。

### 业务场景

商家在后台修改门店信息（名称、地址、营业时间），用户端应该尽快看到更新后的数据。
**问题**：Cache Aside 模式（「读时缓存，写时删缓存」）存在并发竞态：

```
线程 A（写）：删缓存 → 写 DB → [此时线程 B 读到旧缓存并回填] → A 写 DB 完成
线程 B（读）：miss → 查 DB（读到A写前的旧值）→ 写入缓存（旧值！）
```

用户可能在数秒内仍看到旧数据。

### 旧方案：延迟双删，以及它不优雅在哪

**延迟双删**（已移除的 `ShopCacheService.delayedDeleteShopDetail()`）曾经这样做：

```java
// ShopService.updateShop() / onlineShop() / offlineShop() 的写流程（旧版）
shopCacheService.deleteShopDetail(shopId);        // 1. 写前先删（清旧缓存）
shopMapper.updateById(shop);                      // 2. 写 DB
shopCacheService.delayedDeleteShopDetail(shopId); // 3. 事务后延迟 500ms 再删（@Async）
//   实现：Thread.sleep(500L) 后再调用 deleteShopDetail——
//   500ms 依据「MySQL 主从同步延迟通常 < 200ms，给 2.5 倍安全余量」
```

这个方案能工作（约 99% 的并发场景下有效），但有两个本质问题，也是驱动升级的真正原因：

1. **500ms 是一个"猜出来的"经验值**：本质是在猜"数据变更需要多久才能让所有读路径
   感知到"。猜对了相安无事，猜小了竞态窗口仍然存在（生产环境网络抖动时主从延迟
   可能远超 200ms），猜大了则让缓存不必要地脏更久。这个数字背后没有任何确定性保证，
   只是一个"看起来够用"的拍脑袋余量。
2. **缓存失效逻辑和业务写逻辑强耦合**：`updateShop`/`onlineShop`/`offlineShop`
   每一个会改 `shop` 表的写路径，都必须记得在写 DB 后补一行
   `delayedDeleteShopDetail(shopId)`。这是一种"靠人记住"的一致性保证——
   一旦未来新增一个写路径（比如后台批量改价、运营批量下线），漏掉这一行调用，
   就会产生一个只有在生产环境高并发场景下才会暴露的隐蔽 bug，且不会有任何编译期
   或测试期的信号提醒你漏掉了。

### 新方案：Canal + binlog 事件驱动（已完成升级）

核心思路是把"猜延迟"换成"听事件"——MySQL 的 binlog 是"数据确实变了"这件事
**唯一可信的信号源**，与其在业务代码里猜测"数据变更大概什么时候能被所有人感知到"，
不如直接订阅这个信号源，数据一变就立刻清缓存：

```
写 DB 前：deleteShopDetail(shopId)         ← 同步、立即，处理"写之前缓存里还有旧值"的简单情形（保留）
写 DB
写 DB 后：（不再需要业务代码显式删除）
         └─ MySQL 写入 binlog（行级变更日志）
            └─ Canal Server 订阅 binlog 并解析（伪装成 MySQL 的"从库"）
               └─ ShopCacheInvalidationListener（应用内瘦客户端，TCP 长轮询拉取）
                  └─ deleteShopDetail(shopId)   ← 由数据变更事件精确驱动，不是定时器猜出来的
```

**关键实现要点**（`ShopCacheInvalidationListener.java`）：

```java
@PostConstruct
public void start() {
    if (!enabled) {                       // canal.enabled=false 时不订阅，
        return;                           // 退化为「写前删缓存 + 30min TTL」，业务仍然正确
    }
    running = true;
    executor.submit(this::consumeLoop);   // 单独的后台线程，长轮询是阻塞调用
}

private void consumeLoop() {
    while (running) {                                      // 外层：连接生命周期管理（断线重连）
        connector.connect(); connector.subscribe(filter);
        while (running) {                                  // 内层：正常消费循环
            Message message = connector.getWithoutAck(batchSize);
            try {
                handleEntries(message.getEntries());       // 解析 RowChange，过滤 shop 表的行变更
                connector.ack(batchId);                    // 处理成功才确认位点
            } catch (Exception e) {
                connector.rollback();                      // 失败回退到上次 ack 位点，Server 重新推送
            }
        }
    }
}
```

`handleEntries` 按 `EntryType.ROWDATA` 过滤掉事务标记/心跳，按表名过滤出 `shop` 表，
按 `EventType` 过滤出 INSERT/UPDATE/DELETE（DDL 不带行数据，过滤掉），
再从 `RowData` 中提取主键 `id`（DELETE 用 `beforeColumns`，因为行数据已从 DB 消失；
INSERT/UPDATE 用 `afterColumns`，要拿变更后的最新值），调用 `deleteShopDetail(shopId)`。

### 为什么 Canal 方案"显著更好"，而不只是"换了个实现"

| 维度 | 延迟双删 | Canal + binlog |
|------|---------|----------------|
| 失效触发条件 | 猜测的固定延迟（500ms） | 真实的数据变更事件（binlog） |
| 失效时机精度 | 可能早（竞态仍在）也可能晚（不必要的脏读窗口） | 数据一变就触发，通常百毫秒级 |
| 与业务代码的耦合 | 每个写路径都要记得调用，容易漏 | 业务代码无需感知，监听的是表而不是代码路径 |
| 新增写路径的成本 | 必须同步补充缓存失效调用 | 零成本——Canal 监听的是 binlog，新路径自动覆盖 |
| 引入的外部依赖 | 无 | Canal Server（需要独立部署、MySQL 开 binlog） |

可以看到，升级后的方案在"一致性精度"和"代码解耦"两个维度都显著更好，
唯一的代价是引入了一个需要独立运维的外部中间件——这正是下面要讨论的取舍。

### 灰度与回退：`canal.enabled` 开关

Canal Server 不是开箱即用的基础设施（需要 MySQL 开启 `--log-bin --binlog-format=ROW`、
配置 dump 账号、独立部署运维）。为了让这次升级"可灰度、可回退"，
`application.yml` 中加了一个 `canal.enabled` 开关（默认 `false`）：

- **关闭时**：`ShopCacheInvalidationListener.start()` 直接返回，不启动订阅线程，
  缓存一致性退化为「写 DB 前同步删一次缓存 + 30 分钟 TTL 兜底」——业务依然正确，
  只是不一致窗口从"百毫秒级"变回"最长 30 分钟"。
- **打开时**：Canal Server 就绪后接管"写后失效"的职责，把窗口压缩到百毫秒级。

这是"新基础设施分阶段上线"的标准做法：先让代码"在没有这个依赖时也能正确工作"，
再逐步在环境中接入依赖、打开开关、观察效果，而不是一刀切地强绑定。

### 极端情况：Canal 也会失败，靠什么兜底？

**消费处理失败**：单批 entries 处理异常时整体 `rollback()`（回退到上次 ack 的位点），
Canal Server 会重新推送这批未确认的事件——不会丢失，只会重复消费
（`deleteShopDetail` 是幂等操作，删多次和删一次效果相同，重复消费无害）。

**Canal Server 长时间不可用 / 监听器进程崩溃**：
`ShopCacheService.SHOP_DETAIL_TTL = Duration.ofMinutes(30)` 兜底——
缓存最多脏 30 分钟后自然过期重建，不会永久不一致。
**这正是整套架构里最值得讲清楚的一点**：Canal 不是"防止数据永久错误"的最后防线
（TTL 才是），它的价值是把"正常情况下"的不一致窗口从"最长 30 分钟"压缩到
"百毫秒级"。这与 `OrderService`「MQ 延时消息为主链路、定时任务兜底」的分层防御
思路一脉相承——快速路径专注于"快"，把"绝对不能错"的责任交给慢但可靠的兜底机制，
而且这次连"新增兜底代码"都不需要，因为 30 分钟 TTL 本来就已经存在。

### Tradeoff

| 方案 | 一致性 | 复杂度 | 适用场景 |
|------|--------|--------|---------|
| 延迟双删（已替换） | 约 99%（极短并发窗口，依赖经验阈值） | 低 | 读多写少、短暂不一致可接受、不想引入外部依赖 |
| Canal + binlog（当前） | 接近 100%（事件驱动，百毫秒级） | 中（需要独立部署 Canal Server，但应用侧只是瘦客户端） | 一致性精度要求更高、写路径会持续增多的场景 |
| 分布式锁保护读写 | 100% | 高（显著拖慢读性能） | 强一致性、低并发场景 |

### 五层追问 Q&A

**Q1：为什么要从延迟双删升级到 Canal？仅仅是因为它更"高级"吗？**

> 不是为了用新技术而用新技术，而是延迟双删暴露了两个具体问题：
> 一是 500ms 是拍脑袋的经验值，本质是在猜"数据变更多久能被所有读路径感知"，
> 猜错了要么一致性不够、要么缓存不必要地脏更久；
> 二是失效逻辑和业务写路径强耦合，每加一个写路径都要记得手动触发，容易漏。
> Canal 把"猜延迟"换成"听 binlog 事件"，从根上解决了这两个问题——
> 这是"用正确的工具解决正确的问题"，不是炫技。

**Q2：Canal 引入了一个外部中间件依赖，会不会让系统更脆弱？**

> 不会成为正确性的单点依赖。即便 Canal Server 长时间不可用、
> `ShopCacheInvalidationListener` 重连耗尽，Redis 缓存本身的 30 分钟 TTL 会
> 兜住——缓存最多脏 30 分钟后自然过期重建。Canal 的价值是把"正常情况"下的
> 不一致窗口从分钟级压缩到毫秒级，而不是变成"一旦它挂了数据就永久错误"的单点。
> 而且我们用 `canal.enabled` 开关做了灰度：关闭时退化为"写前删缓存 + TTL 兜底"，
> 业务依然正确，可以分阶段接入这个依赖。

**Q3：消费 binlog 事件失败了怎么处理？会不会丢事件？**

> 不会丢。单批处理异常时调用 `connector.rollback()` 回退到上次 ack 的位点，
> Canal Server 会重新推送这批未确认的事件。即使因此重复消费同一批事件也无妨——
> `deleteShopDetail` 是幂等操作，删多次和删一次的最终效果完全一样。
> 这也是为什么我们选择"先处理成功再 ack"而不是"先 ack 再处理"：后者一旦处理失败
> 事件就真的丢了，前者把"失败"转化成"重复"，而重复对幂等操作来说是安全的。

**Q4：为什么用单线程消费，而不是多线程提高吞吐？**

> binlog 是严格有序的（同一行的 INSERT→UPDATE→DELETE 顺序不能错乱）。
> 多线程并发消费会打乱这个顺序，可能出现"先处理了 UPDATE 事件的缓存失效，
> 后处理 INSERT 事件的缓存失效"这种乱序，反而引入新的不一致。
> 而且这里的吞吐瓶颈本来就不在消费端——`shop` 表的写频率很低（商家偶尔改信息），
> 单线程长轮询绰绰有余，강行并发只会增加复杂度而没有收益。

**Q5（MySQL MVCC 映射）：binlog 是在事务的哪个阶段写入的？这对"事件驱动缓存失效"意味着什么？**

> binlog 在事务**提交（commit）时**写入（两阶段提交：先写 redo log prepare，
> 再写 binlog，最后 redo log commit，保证崩溃恢复时两者一致）。
> 这意味着 Canal 收到的 binlog 事件，对应的数据**一定已经持久化提交**——
> 不会出现"缓存已经失效，但 DB 里还是旧值"的反向不一致。
> 这也是 binlog 比"猜延迟"更可靠的根本原因：它是在数据真正落盘之后才发出的信号，
> 而不是一个与实际持久化时机无关的固定等待时间。

## 3. 秒杀抢券：Redis Lua 原子操作

### 业务场景

大促期间商家发起秒杀活动，平台限量发券（如 100 张），用户并发抢购。
**问题**：如何在高并发下保证「不超卖」「不重复领取」，同时保持低延迟？

### 技术实现

**Lua 脚本三步原子操作**（`seckill.lua`）：

```lua
-- KEYS[1] = seckill:stock:{sessionId}:{templateId}（库存 Key）
-- KEYS[2] = seckill:user:{sessionId}:{templateId}（已抢用户集合）
-- ARGV[1] = userId

-- 步骤 1：判重（O(1)，Set 成员检查）
if redis.call('SISMEMBER', user_key, user_id) == 1 then return 2 end  -- 已领取

-- 步骤 2：判库存
local stock = tonumber(redis.call('GET', stock_key))
if stock == nil or stock <= 0 then return 1 end  -- 库存不足

-- 步骤 3：原子扣减（两步中间不会被打断）
redis.call('DECR', stock_key)        -- 库存 -1
redis.call('SADD', user_key, user_id) -- 记录用户
return 0  -- 成功
```

**返回码**：0=成功，1=库存不足，2=已领取

**为什么必须 Lua 脚本**（面试核心）：

```
不用 Lua 时的竞态：
  线程 A: SISMEMBER → false（没抢过）
  线程 B: SISMEMBER → false（没抢过）
  线程 A: GET stock = 1（有货）
  线程 B: GET stock = 1（有货）
  线程 A: DECR → 0 ✓
  线程 B: DECR → -1 ← 超卖！

Lua 脚本在 Redis 单线程内原子执行，B 要等 A 的脚本全部完成后才能执行。
从根本上消除了竞态条件。
```

**异步写 DB（复用 Outbox）**（`SeckillService.publishSeckillSuccessEvent()`）：
Lua 预扣成功后不再同步 INSERT user_coupon，而是把「写券」封装成 `SeckillSuccessEvent`，
复用支付域已经验证过的 `OutboxService`：同事务内写 outbox_message → Relay 异步投递 RocketMQ
→ `SeckillSuccessConsumer` 用 Redis SETNX 幂等模板异步落库（详见 LocalLife 项目接口教程 §7.10）。
关键改造点：把 `saveToOutbox(PaymentSuccessEvent, ...)` 泛化为 `saveToOutbox(Object, String eventId, ...)`，
让落表逻辑与具体事件类型解耦，秒杀域才能直接复用而不必另起一套机制。
前端轮询 `GET /seckill/result` 时依据 `RESULT_KEY`（`seckill:result:{sessionId}:{templateId}:{userId}`）
判断 PENDING/SUCCESS，再叠加 DB 查询和 `USER_SET_KEY` 兜底，构成三层判定。

### 极端情况

**Redis 宕机（秒杀 Stock Key 丢失）**：
Stock Key 消失，所有 `GET stock_key` 返回 nil，Lua 脚本判断 `stock == nil` 返回 1（库存不足）。
所有后续秒杀请求失败。这是「宁可少卖也不超卖」的设计选择：安全性优先于可用性。

降级方案：
1. 应用层本地令牌桶（`AtomicInteger`）提供粗粒度兜底，防止打爆 DB
2. 监控 `stock_key` TTL，接近过期前异步延期

**Lua 预扣成功但异步写库迟迟未完成（升级为 Outbox 后，"DB 写失败"已不在主链路上）**：
Redis 已 DECR，`outbox_message` 也已落表（与「预扣成功」原子绑定在同一事务内），
但 Relay 投递、消费者写库这些环节都是异步的，可能因 MQ 抖动、消费失败而延迟。
这正是复用 Outbox 的核心收益——不再需要在请求主链路里设计「DB 失败回滚 Redis」这种
**非原子、易崩溃**的补偿逻辑：
- Outbox 落表与预扣同事务，保证「锁定名额」这件事不会丢
- Relay 失败有指数退避重试，超限进 FAILED 触发告警
- 消费者失败有 RocketMQ 自动重试（16 次）+ 死信队列兜底
- 即使所有环节都最终失败，`outbox_message` 记录仍在，运维可人工重放，不会出现
  「Redis 库存 -1 但永远查无此券」的不可追溯状态——这比「直接回滚 Redis」更可靠，
  因为回滚本身也可能失败（INCR 成功但 SREM 失败），而 Outbox 链路是「记录在先、投递在后」，
  失败只是"慢"，不是"丢"。

### Tradeoff

| 方案 | 优点 | 缺点 |
|------|------|------|
| Lua 原子脚本（当前）| 高并发安全，延迟 <10ms | 逻辑在 Redis 侧，可读性差 |
| 分布式锁 + Redis | 可读性好 | 锁有 2 次 RTT，吞吐低 |
| 数据库乐观锁 | 强一致性 | 高并发下大量重试，DB 压力高 |

**同步写 DB vs 异步 MQ**（当前已选择异步 MQ，复用 Outbox 基础设施）：
同步写：延迟高（响应路径上多一次 INSERT），实现简单，「抢到后立刻能用」。
异步 MQ（当前方案）：Lua 预扣成功即返回，延迟最低；DB 写入压力被 Relay + 消费者削峰填谷，
不会在抢购瞬间打满连接池；代价是用户需要轮询 `GET /seckill/result`
（依据 `RESULT_KEY = seckill:result:{sessionId}:{templateId}:{userId}`）确认是否已真正出券。
面试时的关键论证：「秒杀场景里响应延迟和瞬时写压力远比『抢到后能不能立刻用』更重要——
用户体感上『抢到名额』和『券真正到账』是两件事，可以接受『抢到了，正在为你出券』这种过程态反馈」。

### 五层追问 Q&A

**Q1：为什么要用 Set 记录已抢用户，不直接判断 DB 有无记录？**

> 判断 DB 要走 MySQL，耗时 5-15ms 且受高并发冲击。Set SISMEMBER 是 O(1)，单次 <1ms。
> 秒杀的核心设计原则：尽量减少对 DB 的写入，让 DB 成为「最终确认」而非「实时判断」。
> 此外，DB 写入在事务中（有 WAL 日志刷盘），每次 INSERT 都比 Redis 写慢 1-2 个数量级。

**Q2：Stock Key 的 TTL 应该设置多少？不设 TTL 有什么风险？**

> 应该设 TTL = 秒杀活动结束时间 + 一定余量（如 +1 小时）。
> 不设 TTL 的风险：活动结束后 Key 永久占用 Redis 内存；如果 Key 意外没有被清理，
> 下次同 sessionId 的活动会读到上次残余的 stock 数值，出现错误。
> 实现：在秒杀场次初始化时 `SET stock_key {stock} EX {ttl}`，Lua 脚本执行前 stock_key 已存在。

**Q3：如果秒杀场次支持每人最多抢 2 张，Lua 怎么改？**

> 将 Set 改为 Hash（存用户 → 已领张数）或 ZSet（score = 张数）：
> ```lua
> local count = tonumber(redis.call('HGET', user_hash_key, user_id) or '0')
> if count >= 2 then return 2 end   -- 已达上限
> redis.call('DECR', stock_key)
> redis.call('HINCRBY', user_hash_key, user_id, 1)
> return 0
> ```

**Q4（JVM 视角）：秒杀期间大量 SeckillRequest、SeckillResultVO 对象的 GC 行为？**

> 这些对象是短生命周期对象（请求结束即可回收），会大量在 Eden 区分配。
> 高并发时 Eden 满 → Minor GC（Young GC）频率高，每次 GC 约 1-5ms Stop-The-World。
> 优化方向：
> 1. 增大 Eden 区（`-Xmn` 参数），减少 Minor GC 频率
> 2. 使用 G1 GC（`-XX:+UseG1GC`），降低单次 STW 时间
> 3. 对象复用：SeckillRequest 用 ThreadLocal 复用（减少频繁 new）

**Q5：为什么秒杀接口要同时加 @RateLimit（每用户 5s 限 1 次）和 Lua 脚本？不是重复了吗？**

> 两者解决的问题不同：
> - `@RateLimit` 是「频率限制」：防止用户在 5 秒内疯狂点击（防脚本刷接口）
> - Lua 脚本是「业务幂等」：防止两个请求同时到达 Redis 时超卖
> 
> `@RateLimit` 在拦截器层（HTTP 层），Lua 在 Redis 层。即使 @RateLimit 失效（如两台服务器），
> Lua 仍然保证库存不超卖。两层防护各自独立，互为补充。

## 4. 支付与 Outbox 消息可靠性

### 业务场景

用户支付成功后，平台需要触发后续流程：发券、更新统计、发用户通知等。
**问题**：支付成功（DB 写入）和发 MQ 消息不是原子操作：DB 提交后 MQ 发送失败 → 消息永久丢失。

### 技术实现

**Transactional Outbox 模式**（`OutboxService.java`）：

```
BEGIN TX
  UPDATE payment_order → SUCCESS
  UPDATE order_info → PAID
  INSERT outbox_message (PENDING)  ← 与业务数据同一事务！
COMMIT TX

[Relay 任务，每 10 秒]
  SELECT PENDING messages
  rocketMQTemplate.send()
  UPDATE → SENT
```

**关键设计**：
- `saveToOutbox()` 用 `Propagation.MANDATORY`：强制要求外层事务，防止在无事务环境误调用
- Relay 任务不用 `@Transactional`：每条消息独立发送，发一条确认一条，不互相影响
- 多实例并发安全：`markAsSent WHERE status='PENDING'`，只有一个节点 affected=1

**指数退避重试**：
- 第 1 次失败：10s 后重试
- 第 2 次失败：30s 后重试
- 第 3 次失败：60s 后重试
- 第 4 次失败：标记 FAILED，停止（等自动恢复任务或人工干预）

**死信自动恢复**（`OutboxService.autoRecoverFailedMessages()`）：
- 每小时扫描 `status='FAILED' AND auto_retry_count < 3` 的消息
- 重置为 PENDING，`auto_retry_count +1`
- MQ 短暂不可用导致的 FAILED 批量消息，MQ 恢复后自动重新投递

**支付回调幂等（`PaymentService.handleCallback()`）**：
```java
// 第一道防线：CAS UPDATE
int affected = paymentOrderMapper.updateStatusOnSuccess(paymentOrder.getId(), ...);
if (affected == 0) return;  // 已处理，幂等跳过

// 数据库兜底：uk_payment_channel_trade_no(channel, trade_no)
```

### 极端情况

**MQ Broker 宕机 10 小时，积累大量 FAILED 消息**：
- `auto_retry_count < 3` 的消息：MQ 恢复后 1 小时内自动恢复投递
- `auto_retry_count >= 3` 的消息：需要人工干预（说明是持续性故障，不是临时抖动）
- 人工恢复：`UPDATE outbox_message SET auto_retry_count=0, status='FAILED' WHERE ...`（或直接重置为 PENDING）
- 告警：监控 `SELECT COUNT(*) FROM outbox_message WHERE status='FAILED' AND auto_retry_count >= 3`

**支付回调重复 3 次（渠道重试）**：
- 第 1 次：`UPDATE WHERE pay_status='PENDING'` → affected=1，处理成功
- 第 2、3 次：`UPDATE WHERE pay_status='PENDING'` → affected=0（已是 SUCCESS），幂等跳过
- DB 兜底：`uk_payment_channel_trade_no` 唯一索引，即使 UPDATE 判断有时序问题也会被数据库拦截

### Tradeoff

| 方案 | 可查性 | MQ 依赖 | 实时性 | 选择理由 |
|------|--------|---------|--------|---------|
| Outbox（当前）| 完整日志 | 弱（可换 MQ）| 10s 延迟 | 故障排查、运营要求 |
| RocketMQ 事务消息 | 无 | 强（绑 RocketMQ）| 实时 | 需要 broker 支持事务特性 |
| 同步发 MQ | 无 | 强 | 实时 | 发送失败直接丢消息 |

**为什么 Outbox 的 10s 延迟可接受？**
支付成功后发券、发通知属于「最终一致性」场景，用户能接受几秒内发到。
如果需要实时（< 1s），可以在 Outbox 写入后立即触发一次 Relay（Outbox + Push 混合）。

### 五层追问 Q&A

**Q1：Relay 任务用 `fixedDelay` 而不是 `fixedRate`，有什么区别？**

> `fixedDelay`：上次执行完成后等 10s 开始下次。
> `fixedRate`：无论上次是否完成，每隔 10s 触发一次。
> 选 `fixedDelay` 是因为：Relay 任务处理时间不固定（扫 100 条可能耗时 2-3 秒），
> 用 `fixedRate` 可能导致前一次还没结束下一次就开始，并发执行产生重复发送。
> Outbox 设计已通过 `WHERE status='PENDING'` 幂等，并发多少次都安全，
> 但 `fixedDelay` 是更稳健的默认选择。

**Q2：Outbox Relay 多实例部署（3 个节点）时，同一条消息会被发 3 次吗？**

> 不会（绝大多数情况）。`markAsSent` 的 `WHERE status='PENDING'` 是 CAS（Compare And Swap）：
> 3 个节点同时执行，只有 1 个节点的 UPDATE 返回 affected=1（成功），其余 2 个返回 0（幂等跳过）。
> 极端情况：3 个节点都发送成功但只有 1 个更新状态（网络分区），消息可能被重复发送。
> 防御：消费者通过 `eventId` Redis SETNX 幂等（已实现）。

**Q3：`Propagation.MANDATORY` 是什么语义？为什么 saveToOutbox 要用它？**

> `MANDATORY`：要求调用方**必须已经有事务**，否则抛 `IllegalTransactionStateException`。
> `REQUIRED`（默认）：如果有事务就加入，没有就创建新事务。
> 
> saveToOutbox 用 MANDATORY 是一种防御性编程：强制要求与业务操作（更新 payment_order/order_info）在同一事务中执行。
> 如果有人在没有事务的地方调用 saveToOutbox（bug），会立即报错，而不是默默创建一个独立事务（导致消息和业务数据不一致）。

**Q4：Outbox 表的 `event_id` 为什么不能用 `id` 代替？**

> `id` 是存储 ID（每次 INSERT 新生成），可能因重试产生新 id；
> `event_id` 是业务语义 ID，与业务操作绑定（格式：`paymentOrderId + "_paid"`）。
> 业务操作无论重试多少次，`event_id` 应该相同（幂等标识）。
> 如果用 `id` 作为幂等键，每次重试都是新 id，消费者无法判断是否重复消费。

**Q5（Spring 事务传播深挖）：如果 saveToOutbox 用 REQUIRES_NEW，会有什么问题？**

> `REQUIRES_NEW` 挂起外层事务，启动独立新事务写 outbox_message。
> 如果外层业务事务随后回滚（如支付金额校验失败），outbox 消息**已经提交**（独立事务），
> 导致「消息已写入但业务未完成」的幽灵消息——消费者会处理一个「支付失败」的支付成功事件。
> 这正是 Outbox 模式要解决的问题：outbox 和业务**必须同一事务**，要么都成功，要么都回滚。

**Q6（架构复用）：Outbox 现在已经不止服务支付域了，秒杀域也在用，改造时遇到了什么坑？**

> 最初 `saveToOutbox(PaymentSuccessEvent event, String topic, String tag)` 的签名写死了
> `PaymentSuccessEvent`，内部直接调 `event.getEventId()`。秒杀域要发布 `SeckillSuccessEvent`，
> 传不进去。
>
> 改法：把签名泛化为 `saveToOutbox(Object event, String eventId, String topic, String tag)`，
> 让 `eventId` 由调用方显式传入而不是从 event 对象里取——`saveToOutbox` 因此只需要关心
> 「序列化 + 落表」，完全不用知道事件长什么样。这是「按依赖倒置原则解耦」的一个小例子：
> 通用基础设施不应该依赖某个具体业务类型。
>
> 改造的连带影响也要评估：grep 全项目只有 `PaymentService` 一处调用点，改完同步把
> `outboxService.saveToOutbox(event, MqTopics.XXX, ...)` 改成
> `outboxService.saveToOutbox(event, event.getEventId(), MqTopics.XXX, ...)` 即可，
> 没有破坏既有调用方。这种「先评估调用点数量再决定怎么改签名」的习惯，
> 是大型项目里改公共方法时必须要有的防护意识。

**Q7（技术选型边界感）：既然 Outbox 基础设施已经现成可复用，订单超时关闭的延时消息为什么不也接进来？**

> 这是「升级 `OrderService.closeExpiredOrders` 到 RocketMQ 延时消息」时刻意做的取舍——
> 答案是：**不是所有 MQ 消息都值得用 Outbox**，要看这条消息「丢了会怎样」。
>
> Outbox 解决的核心问题是「事件必须可靠投递，丢了就会数据不一致」（支付成功事件丢了，
> 用户付了钱订单却显示未支付；秒杀成功事件丢了，用户抢到了券却拿不到）。
> 但订单关闭延时消息恰恰相反——它**允许丢**：
> - 它不是触发关单的唯一途径，`closeExpiredOrders()` 定时任务仍然按 `expire_at` 兜底扫描，
>   消息丢了订单也只是退化为"等定时任务扫到再关"，不会出现"永远关不掉"的数据问题
> - 关单操作本身是 `WHERE status='WAIT_PAY'` 的 CAS 更新，天然幂等，重复触发没有副作用
>
> 用 Outbox 的"落库 + Relay 轮询投递"两段式反而是过度设计：多一次 DB 写入，
> 多一段轮询延迟（当前 Relay 是每 10 秒扫一次），却没有换来"必须"的可靠性收益。
> 选择是直接 `rocketMQTemplate.syncSend(...)` 同步发送、best-effort、失败只记日志不重试——
> 用一个 `TransactionSynchronizationManager.registerSynchronization(...)` 的 `afterCommit`
> 钩子保证"事务提交后才发"就够了，比引入 Outbox 轻量得多。
>
> 这道题真正考察的不是"记不记得 Outbox 怎么用"，而是**有没有按场景判断"这条消息需要多强的
> 可靠性保证"的意识**——可靠性投入应该和事件的重要程度匹配，把每条消息都按最高规格走一遍
> "落库 + 轮询 + 重试 + 死信"，看似严谨，实则是对基础设施的滥用。

## 5. 订单分片：ShardingSphere user_id 路由

### 业务场景

随着平台增长，order_info 表行数可能超过千万级（每天 10 万订单，3 年 ≈ 1 亿行）。
MySQL 单表超过 5000 万行后 B+ 树层数增加（4 层变 5 层），查询 IO 次数增加，P99 延迟明显上升。

### 技术实现

**分片策略**（`sharding.yaml`）：
- 逻辑表：`order_info`（应用层看到的表名）
- 物理表：`order_info_0 ~ order_info_3`（数据库中真实存在的表）
- 分片键：`user_id`
- 算法：`MOD 4`（`user_id % 4` → 0/1/2/3）
- 开发环境：单 MySQL 实例 4 张表（节省资源）；生产可扩展为 2 库 × 2 表

**ShardingSphere Driver 接管 JDBC**：
应用层只写 `INSERT INTO order_info WHERE user_id = ?`，ShardingSphere 自动改写为 `INSERT INTO order_info_1 WHERE user_id = ?`。
对 MyBatis-Plus 完全透明，不需要改实体类和 Mapper。

**UPDATE 必须携带 user_id（精准路由）**：
```java
// OrderInfoMapper.updateStatusFromWaitPay — 传入 userId 精准路由
@Update("UPDATE order_info SET order_status = #{targetStatus} " +
        "WHERE id = #{orderId} AND user_id = #{userId} AND order_status = 'WAIT_PAY'")
int updateStatusFromWaitPay(@Param("orderId") Long, @Param("userId") Long, @Param("targetStatus") String);
```
不传 user_id 时 ShardingSphere 广播到 4 个分片，4 倍 DB 压力。

### 极端情况

**某个分片（order_info_2）挂了**：
`user_id % 4 = 2` 的用户订单全部不可访问（约 25% 用户）。
依赖 MySQL 主从复制：每个分片应该有独立 Slave，主库宕机后切换到 Slave。
ShardingSphere 不自动处理 failover，需要配合 MySQL Proxy 或 MHA 实现。

**按 order_no 查询（不含 user_id）**：
ShardingSphere 广播到 4 个分片，汇总结果。
优化方案：建立 ES 订单索引（`order_no → order_id + user_id`），先查 ES 得到 user_id，再精准路由。

### Tradeoff

| 分片键 | 「我的订单」查询 | 跨用户查询 | 数据均匀性 |
|--------|----------------|------------|----------|
| `user_id`（当前）| ✅ 单分片精准 | ❌ 广播 4 分片 | 依赖用户数均匀分布 |
| `order_id` | ❌ 广播 4 分片 | ✅ 精准 | 均匀 |
| 时间分片 | ❌ 可能跨分片 | 取决于时间范围 | 近期热点分片压力大 |

**选 user_id 的核心理由**：C 端核心查询是「我的订单」（按 user_id 过滤），C 端比 B 端流量大 10 倍以上。

### 五层追问 Q&A

**Q1：ShardingSphere 5.5.0 有哪些已知限制？**

> 1. `LIMIT` 分页：`SELECT ... LIMIT 10` 会变成每个分片各取 10 条（共 40 条），内存合并后再取 10 条，内存放大 4 倍。改进：主键分页（`WHERE id > last_id LIMIT 10`）。
> 2. 聚合函数（GROUP BY）：需要 ShardingSphere 内存聚合，数据量大时内存压力高。
> 3. 子查询：部分包含分片键的子查询不支持，需要改写。
> 4. 跨分片 JOIN：不支持，需要应用层分步查询。

**Q2：想扩展到 2 库 4 表（生产），需要改哪些配置？**

> 只需修改 `sharding.yaml`：
> ```yaml
> dataSources:
>   ds0: { jdbcUrl: mysql://db-master-0:3306/... }  # 新增
>   ds1: { jdbcUrl: mysql://db-master-1:3306/... }
> tables:
>   order_info:
>     actualDataNodes: ds${0..1}.order_info_${0..1}  # 修改
>     databaseStrategy:
>       standard: { shardingColumn: user_id, shardingAlgorithmName: dbMod }
>     tableStrategy:
>       standard: { shardingColumn: user_id, shardingAlgorithmName: tableMod }
> ```
> **应用代码零改动**，这是 ShardingSphere JDBC 的核心价值。

**Q3：ShardingSphere 5.5.0 的 `shardingsphere-test-util` 缺失 bug 是怎么回事？**

> 这是 5.5.0 版本的打包 bug：`shardingsphere-test-util` 是测试工具，被错误地声明为非 test 作用域依赖，但它没有发布到 Maven Central。
> 解决方案：在 `pom.xml` 的 dependency 里添加 `<exclusion>` 排除它：
> ```xml
> <exclusions>
>   <exclusion>
>     <groupId>org.apache.shardingsphere</groupId>
>     <artifactId>shardingsphere-test-util</artifactId>
>   </exclusion>
> </exclusions>
> ```
> 这个 bug 反映了开源项目的版本管理问题，也是我们在使用时遇到并解决的真实工程问题。

**Q4：分片后如何做「今日所有用户下单量」的全局统计？**

> 方案一：ShardingSphere 广播查询（4 个分片各 COUNT，汇总）。对于统计类查询可接受，频率低。
> 方案二：实时写入统计表（业务代码里每次下单后更新统计表，单表统计）。适合高频统计。
> 方案三：数仓 ETL（离线）：每天定时把 4 个分片数据同步到 ClickHouse，跑聚合查询。适合复杂报表。
> 
> 我们目前用的是 Prometheus 统计（`BusinessMetrics.recordOrderCreated(shopId)`），Grafana 展示。

**Q5（分布式 ID 深挖）：雪花算法的时钟回拨问题是什么？MyBatis-Plus 怎么处理？**

> 时钟回拨：机器时钟回调（如 NTP 同步）导致生成了与历史 ID 相同的时间戳，可能产生 ID 冲突。
> MyBatis-Plus 的 `ASSIGN_ID` 使用 Snowflake，底层是 `IdentifierGenerator`，遇到时钟回拨会：
> 1. 等待时钟追上（如果回拨量 < 5ms）
> 2. 抛出 `RuntimeException`（如果回拨量 > 5ms）
> 
> 生产建议：使用 `百度 uid-generator` 或 `美团 Leaf`，这两个方案都处理了时钟回拨问题（用历史时间戳 + 序列号偏移）。

## 6. MCP 工具协议层

### 业务场景

Copilot Agent 需要调用 LocalLife 的业务数据（订单、支付、门店），但直接调 REST API 有问题：
- API 设计给用户调用，需要 JWT Token（Agent 没有用户 Token）
- 没有权限控制边界（Agent 可能越权查其他商家数据）
- 没有审计（谁调了什么参数，返回了什么，无记录）

**MCP（Model Context Protocol）**解决这个问题：标准化 Agent 与工具的通信协议，类似 USB-C。

### 技术实现

**JSON-RPC 2.0 over HTTP**（`McpController.java`）：
```
POST /mcp
{
  "jsonrpc": "2.0", "id": "req-001",
  "method": "tools/call",
  "params": {
    "name": "query_order",
    "arguments": {"order_id": "1234567890"}
  }
}
```

**工具分级（L1-L4）**（`McpTool.java`）：
- L1 只读查询：`query_order` / `shop_metrics_query` 等（RBAC + 审计）
- L2 草稿生成：`campaign_draft_generate`（审计 + 用户确认）
- L4 高风险执行：`execute_refund` / `issue_compensation_coupon`（HITL + 审批 + 审计）

**RBAC 身份注入（`RbacFilter.java`）**：
```
请求 Header：X-User-Id: 10001, X-User-Role: merchant, X-Merchant-Id: 20001
             ↓ RbacFilter 解析 → ThreadLocal
SQL 查询：WHERE shop.merchant_id = #{threadLocal.merchantId}
```
Agent 不能自行修改 merchant_id；服务端注入，强制边界。

**自实现 JSON-RPC 而不用 Spring AI 官方 SDK 的原因**：
官方 `@Tool` 注解方案只提供简单工具注册，没有工具分级、RBAC、审计、限流的细粒度控制。
我们自实现是为了在协议层直接内置治理能力（`McpController → ToolRegistry → RBAC → execute → Audit`）。

### 五层追问 Q&A

**Q1：MCP 和 Function Calling 有什么区别？**

> Function Calling 是 LLM 的「输出格式约定」（让模型输出 JSON 格式的工具调用指令）。
> MCP 是工具调用的「网络传输协议」（定义 Client 和 Server 之间的通信标准）。
> 类比：Function Calling 是「说什么语言」，MCP 是「用什么信道传输」。
> 两者不互斥：LLM 通过 Function Calling 输出调用决策，Agent 通过 MCP 协议执行调用。

**Q2：MCP 和 A2A 协议有什么区别？**

> MCP（Model Context Protocol）：LLM/Agent 调用工具/数据源的协议（纵向，Agent→Tool）。
> A2A（Agent-to-Agent）：Agent 之间协作委托的协议（横向，Agent→Agent）。
> MCP 解决「工具集成标准化问题」（一个 Agent 调用多个工具），
> A2A 解决「Agent 间通信问题」（一个 Orchestrator 分配任务给多个 Sub-Agent）。

**Q3：工具限流是怎么实现的？为什么在工具层而不是在 HTTP 层？**

> 工具限流在 `ToolRateLimiter.java`（Redis INCR，100次/分钟/工具 + 30次/分钟/用户）。
> 为什么在工具层：
> - Agent 可能在一次对话中多次调用同一工具（ReAct 循环），HTTP 请求是合法的，不能在 HTTP 层拒绝
> - 工具层限流针对「同一用户对同一工具的频率」，与 HTTP 请求频率不同
> - 防止 Agent 失控循环（Bug 导致无限调用）消耗工具资源

**Q4：RBAC 的 merchant_id 是谁设置的？Agent 能伪造吗？**

> merchant_id 由 Python Agent Service 注入（从数据库根据 user_id 查询），通过 HTTP Header 传递。
> Agent（LLM）只能影响请求的 `arguments` 字段，不能修改请求 Header。
> MCP Server 从 Header 读取 merchant_id（`X-Merchant-Id`），注入到 ThreadLocal，工具执行时强制过滤。
> 即使 Prompt Injection 攻击让 LLM 生成了含 `merchant_id: 99999` 的 arguments，SQL 也会用 ThreadLocal 里的合法 merchant_id 覆盖。

**Q5：结构化错误（5 种 reason）的设计价值是什么？**

> 普通错误：`{"error": "查询失败"}`  → Agent 不知道怎么处理
> 结构化错误：`{"reason": "parameter_error", "detail": "order_id 格式错误", "hint": "应为纯数字"}`
> → Agent 可以根据 `reason` 自动决策：
> - `parameter_error` → 修正参数重试（Agent 可以自动修复）
> - `permission_denied` → 停止调用，告知用户权限边界（不重试）
> - `tool_timeout` → 重试一次（可能是临时网络问题）
> 这是「让工具调用失败变得可处理」的关键设计，降低 Agent 卡住和发散的概率。

## 7. Agent Harness：并行工具 + 循环检测 + Fast Path + HITL

### 业务场景

「65% 的企业 AI 失败来自 Harness 缺陷，而非模型能力不足」（来自 Firecrawl Agent Harness Engineering 研究）。
三类主要缺陷：
1. **State Degradation**（状态降级）：HITL 挂起后服务重启，状态丢失
2. **Context Drift**（上下文漂移）：对话轮数增多后 Agent 偏离任务
3. **Schema Misalignment**（Schema 不匹配）：工具调用参数类型不一致导致静默错误

我们的 Agent Harness 针对性解决了这三类问题。

### 技术实现

**并行工具执行 + 并发安全分批**（`nodes.py tool_node` / `tool_router.py`）：
```python
# 原来：串行 for 循环，延迟 = t1 + t2 + t3
# 后来：不分青红皂白地 asyncio.gather 全部，延迟 = max(t1, t2, t3) ≈ 1/N，
#       但 query_order 和 execute_refund 这种查询/高风险写操作混在一起并发跑，
#       会有竞态和审计隔离问题（参考 Claude Code 的 partitionToolCalls 设计）
# 现在：先按并发安全性分批，安全批并发、不安全/高风险批单独隔离串行
batches = _partition_tool_calls(tool_calls)   # fail-closed：未登记工具一律视为不安全
for mode, indices in batches:
    if mode == "concurrent":
        batch_results = await asyncio.gather(
            *[_execute_single_tool(tool_calls[i]) for i in indices],
            return_exceptions=True,            # 一个失败不影响其他
        )
    else:                                       # execute_refund / issue_compensation_coupon 等
        batch_results = [await _execute_single_tool(tool_calls[i]) for i in indices]
```
分级表 `TOOL_CONCURRENCY_SAFE` 只收录只读查询类工具，资金类高风险写操作
（`execute_refund` / `issue_compensation_coupon`）和有副作用的生成型操作
（`campaign_draft_generate`）都不在其中——fail-closed，未显式登记 = 按不安全处理。

理论依据：主流 LLM（Claude/GPT-5）原生支持 parallel tool calls，一次返回多个 `tool_use block`；
但"能并发"不等于"该并发"，并发安全性应该由工具自身的副作用/破坏性决定，而不是无脑全量并发。

**循环检测**（`nodes.py _detect_loop`）：
```python
def _detect_loop(messages, tool_name, args):
    """同工具同参数在最近10条消息中出现 ≥ 3 次 → 触发停止"""
    identical_calls = [tc for m in messages[-10:]
                       if hasattr(m, "tool_calls")
                       for tc in m.tool_calls
                       if tc["name"] == tool_name and tc["args"] == args]
    return len(identical_calls) >= 2  # 已有2次，当前是第3次
```

**Prompt Caching**（`nodes.py llm_node`）：
```python
system_msg = SystemMessage(
    content=_build_system_prompt(tools),
    additional_kwargs={"cache_control": {"type": "ephemeral"}},  # 缓存5分钟
)
```
系统提示（角色定义 + 工具说明）是稳定内容，每次重复计费是浪费。缓存后节省约 80% input tokens。

**Fast Path 路由**（`api/chat.py`）：
```python
fast_path_result = await _try_fast_path(message, user_role, merchant_id, ...)
if fast_path_result is not None:
    return StreamingResponse(fast_stream(fast_path_result), ...)  # 跳过 LangGraph
```
识别模式：「今天/昨天 + 销售额/订单量」→ 直接调 `shop_metrics_query`，延迟 <500ms。
约覆盖 60% 的商家日常查询。

**MySQL Checkpoint（HITL 状态持久化）**（`session/checkpointer.py`）：
```python
class AsyncMySQLCheckpointer(BaseCheckpointSaver):
    async def aput(self, config, checkpoint, ...):
        # 序列化 Agent 状态（含消息历史）→ 写 langgraph_checkpoint 表
    async def aget_tuple(self, config):
        # 从 DB 恢复，支持服务重启后继续 HITL 流程
```

### HITL 完整 9 步流程

```
1. Agent 决定调用 execute_refund → 识别 L4 高风险工具
2. hitl_node 写 hitl_approval（PENDING）到 MySQL
3. LangGraph 执行到 hitl_node → END → checkpoint 写入 MySQL
4. SSE 推送：event: hitl_request, data: {thread_id, action, approval_id}
5. 运营在审批工作台看到申请（GET /hitl/pending）
6. 运营通过审批（POST /hitl/{id}/approve）
7. 前端调用恢复接口（POST /chat/resume with thread_id）
8. Agent Service 从 MySQL Checkpoint 恢复 Agent 状态（aget_tuple）
9. Agent 继续执行，调用 execute_refund → LocalLife Internal API
```

### Tradeoff

| 组件 | 选择 | 替代方案 | 选择理由 |
|------|------|---------|---------|
| Checkpoint 存储 | MySQL | Redis / MemorySaver | Redis 需 RDB/AOF；MemorySaver 重启丢失 HITL 状态 |
| 循环检测阈值 | 3 次 | 2 次或 5 次 | 2 次太激进（合理重试被误杀）；5 次浪费太多 token |
| Fast Path 规则 | 关键词 | LLM 意图识别 | 关键词更快（<1ms）；LLM 识别更准但要多 1 次 LLM 调用 |

### 五层追问 Q&A

**Q1：并行工具执行只适合哪些工具？哪些必须串行？**

> 同一轮 LLM 返回的 tool_calls 通常没有数据依赖（有依赖的话 LLM 会拆成多轮 ReAct，
> 不会在同一轮返回两个互相依赖的工具调用），这是"能并发"的前提。
> 但"没有数据依赖"不等于"可以放心并发"——还要看工具本身是否有副作用/破坏性。
> 我们按并发安全性把工具分成两类（`tool_router.TOOL_CONCURRENCY_SAFE`，fail-closed）：
> 只读查询类（`query_order`/`query_payment`/`shop_metrics_query`...）互不影响数据状态，
> 合并成并发批吃满 1/N 的延迟收益；`execute_refund`/`issue_compensation_coupon` 这类
> 资金类高风险写操作，以及未显式登记的任何工具（含未来新增的），一律单独隔离成串行批——
> 既避免和其他调用产生竞态，也方便审计时单独追溯每一次高风险动作。
> `_partition_tool_calls()` 负责把一轮 tool_calls 切成"连续安全合并并发批 + 不安全单独隔离串行批"，
> 直接借鉴了 Claude Code Tool 接口的 `isConcurrencySafe`/`partitionToolCalls` 设计。

**Q2：Prompt Caching 的缓存是如何失效的？会不会缓存了错误内容？**

> Claude 的 ephemeral 缓存 TTL 是 5 分钟，5 分钟后自动失效，下次请求重新缓存。
> 失效场景：工具列表更新（如部署新版 MCP Server）→ 需要等最多 5 分钟生效（因为 Schema TTL 缓存也是 5 分钟）。
> 不会缓存错误内容：只有 SystemMessage 被标记为 ephemeral，用户消息不缓存（每次都是新鲜内容）。
> 风险：如果 tool schema 更新了但 system prompt 还在缓存，Agent 可能用旧 schema 生成参数。
> 防御：MCP Server 工具更新时提供 `POST /mcp/cache/invalidate` 端点（当前未实现，标记为 TODO）。

**Q3：LangGraph Checkpoint 和 Session Cache 有什么本质区别？**

> Session Cache（如 `agent_message` 表）：存消息历史，用于会话回放和审计。
> LangGraph Checkpoint：存 Agent 的**完整执行状态**（消息历史 + 图的执行位置 + 待执行的 pending_action）。
> 
> 关键区别：Checkpoint 可以从**中断点恢复执行**。Session Cache 只能重放历史，不能继续执行。
> HITL 必须依赖 Checkpoint：服务重启后，从 Checkpoint 恢复不仅要知道「说了什么」，
> 还要知道「执行到哪一步」「审批通过后要做什么」。

**Q4：Agent 卡死了（一直在 reflection_node 和 tool_node 之间循环）怎么处理？**

> 防御机制（多层）：
> 1. 循环检测（已实现）：同工具同参数 ≥ 3 次 → 强制停止
> 2. max_steps（已实现）：最多 15 步，超限 → final_node
> 3. token 预算（已实现）：50000 tokens/会话，超限 → final_node
> 4. reflection_node 反思：每 5 步触发，让 LLM 自我评估是否在绕圈
> 5. 告警（未实现）：监控单会话步数 > 10，触发告警通知 on-call

**Q5：为什么 HITL 审批用 MySQL 而不是 Redis？**

> HITL 审批可能需要等待数小时（甚至 24 小时），期间服务可能重启。
> Redis 的数据需要配置 RDB/AOF 持久化，且有内存压力；MySQL 天然持久化，已有 HA（主从）。
> 此外，`hitl_approval` 表还需要支持：查询待审批列表（JOIN session）、审批历史审计，这些操作在 MySQL 更自然。
> 面试说法：「HITL 状态必须跨重启存活，MySQL 已有可靠持久化和 HA，不引入新依赖。」

## 8. RAG 检索与评测：Hybrid RAG + LLM-as-Judge

### 业务场景

商家问「双十一活动券的退款规则是什么」，Agent 需要在平台规则文档中检索相关内容，基于真实规则回答，不能编造。

**问题**：
1. 纯向量搜索对「ORDER_12345」「5 折券」等精确词召回不稳定（Recall@5 约 52%）
2. 只看关键词覆盖率不知道「Agent 有没有编造规则」

### 技术实现

**Hybrid RAG 流水线**（`rag/pipeline.py`）：

```
查询
  ↓ Query Rewrite（去空白，标准化）
  ↓ 并行双路召回
    ├── Milvus 向量搜索 Top 20（语义相似度，multilingual-e5-base 768维）
    └── BM25 关键词搜索 Top 20（rank_bm25，精确词匹配）
  ↓ RRF 融合（Reciprocal Rank Fusion，k=60）
  ↓ Cross-Encoder Reranker 精排（ms-marco-MiniLM-L-6-v2）
  ↓ 相关性阈值过滤（score < 0.3 → 拒答，避免编造）
```

**RRF 融合公式**：
```
score(d) = Σ 1/(k + rank_i(d))
k=60（来自原始 RRF 论文，防止排名 1 得分过高）
```
不依赖原始分数的量纲（向量相似度和 BM25 分数无法直接相加）。

**分块策略**（`pipeline.ingest_document`）：
- 块大小：500 字，重叠：50 字
- 依据：平台规则文档按章节组织，500 字覆盖一个完整章节
- 对比（⚠ 待实测）：250/500/1000 三种块大小，需要跑 evals 验证最优

**LLM-as-Judge**（`evals/judge.py`）：
- 用 Claude Haiku（cheap, fast）评测 Claude Sonnet（业务模型）的输出
- 评测维度：Faithfulness（忠实度）/ Relevance（相关性）/ Hallucination（幻觉检测）
- 固定 temperature=0，保证评测可重复性
- 人工抽检 10% 校准 Haiku 判断偏差

**运行评测**：
```bash
# 基础评测（6项代码规则指标）
python -m evals.metrics --real

# 完整评测（含 LLM-as-Judge，有额外成本）
python -m evals.metrics --real --judge

# 只跑知识类（验证 Hybrid RAG 效果）
python -m evals.metrics --real --category knowledge
```

### 极端情况

**Milvus 不可用（连接失败）**：
`vector_store.search()` 返回 `_mock_search()` 结果（只有一条 Mock 文档）。
Agent 工具调用收到空/Mock 结果 → 拒答（拒答优于编造）。
BM25 内存索引仍可用，提供关键词检索兜底。

**知识库内容过期（政策更新后未重新入库）**：
Agent 依据旧规则回答，是信息准确性问题而非幻觉（幻觉是编造 KB 里没有的内容）。
解决方案：定期 `python -m rag.ingest` 重建知识库；生产中接 Canal 监听规则文档变更触发自动重建。

### Tradeoff

| 方案 | Recall@5（预期）| 成本 | 实现复杂度 |
|------|----------------|------|---------|
| 纯向量（原）| ~52%（⚠ 待实测）| 低 | 低 |
| Hybrid RAG（当前）| ~80%（⚠ 待实测）| 中（BM25 内存）| 中 |
| RAG-Fusion（多角度改写）| ~85%+ | 高（多次 LLM 改写）| 高 |

**为什么选 Hybrid RAG 而不是 RAG-Fusion？**
RAG-Fusion 需要对每个查询生成 3-5 个改写版本，调用 LLM 多次，成本高 3-5 倍。
Hybrid RAG 用 BM25（无 LLM 成本）互补，性价比更高。

### 五层追问 Q&A

**Q1：向量检索和 BM25 的分数量纲不一样，为什么 RRF 可以合并它们？**

> 这正是 RRF 的核心价值：它**不使用原始分数**，只使用**排名**。
> 无论向量相似度是 0.85（余弦相似度，0-1）还是 BM25 分数是 12.3（无上界），
> RRF 只关心「这个文档在这一路结果中排第几」，用 `1/(k + rank)` 统一量化。
> 这解决了「不同检索方法分数无法直接相加」的问题，是 RRF 广泛应用的原因。

**Q2：Cross-Encoder Reranker 比向量检索更准，为什么不直接用它做召回？**

> Cross-Encoder 计算量正比于 `查询 × 文档数`，不能预计算（每次查询都要重新计算）。
> 如果对全量知识库（哪怕只有 1000 篇）用 Cross-Encoder，每次查询需要计算 1000 次前向传播，
> 耗时约 2-10 秒（GPU），无法接受。
> Bi-Encoder（向量搜索）可以预计算文档 embedding，查询时只算查询向量，
> 再做近似最近邻（ANN），毫秒级完成。两阶段（粗召回 + 精排）是平衡精度和速度的最佳实践。

**Q3：BM25 内存索引服务重启后丢失，怎么保证 Hybrid RAG 在重启后还能工作？**

> BM25 索引丢失后，`_rrf_merge` 的 `bm25_docs` 为空，代码里有 fallback：
> ```python
> if bm25_docs:
>     merged = _rrf_merge([vector_docs, bm25_docs], k=60)
> else:
>     merged = vector_docs  # 降级为纯向量
> ```
> 即降级为纯向量检索，功能不中断（只是召回率会下降）。
> 要完全恢复 BM25：重启后调用 `python -m rag.ingest` 重建知识库。
> 生产中可在 `main.py` 的 `lifespan` 里自动触发 ingest（如果知识库是静态内容）。

**Q4：LLM-as-Judge 用 Haiku 评测 Sonnet 的输出，会有「宽容评分」偏差吗？**

> 是的，LLM judge 有已知偏差：
> 1. 倾向于给较长、更完整的回答更高分（即使有幻觉）
> 2. 倾向于给与 judge 模型风格一致的回答更高分
> 3. Haiku 比 Sonnet 弱，可能对 Sonnet 的微妙错误不敏感
> 
> 缓解措施（已在 `judge.py` 中体现）：
> - 固定 temperature=0（减少随机性）
> - 明确的评分标准（prompt 里有 0.0/0.3/0.7/1.0 对应的具体标准）
> - 人工抽检 10% 做 ground truth 校准（judge 偏差校正）

**Q5：知识库只有 2 篇文档，Recall@5 能代表真实场景吗？**

> 诚实说：不完全能。当前 2 篇文档是演示/开发用，生产环境应该有 50-200 篇文档
> （平台规则手册、商家 FAQ、故障复盘案例等）。
> 
> 当前评测的 50 条 knowledge cases 使用的是固定关键词匹配（代码规则评测），
> 不是真实 Milvus 搜索结果评测（需要先 ingest 足够文档）。
> 
> 面试说法：「当前知识库是 MVP 状态，有 2 篇示例文档。Recall@5 指标待知识库充实到生产规模后实测。
> 评测框架（50 用例 + 多指标）已搭建好，运行 `python -m evals.metrics --real --category knowledge` 即可验证。」

## 附录 C：弱点承认与话术

弱点要诚实说，但要配合「升级路径」或「设计权衡」，不能只说弱点没有对策。

### 1. Outbox 轮询延迟（10s）

> **弱点**：Relay 任务每 10 秒轮询，支付成功到发券最多延迟 10 秒。
>
> **话术**：「当前 Outbox 是轮询实现，优点是可查日志、故障排查方便；缺点是有 10 秒的批次延迟。升级方向是用 Canal 监听 outbox_message 的 binlog，变更时触发实时投递，延迟降到毫秒级。当前场景下发券延迟 10 秒用户可以接受。」

### 2. Canal 引入了一个需要独立运维的外部中间件

> **弱点**：相比"纯应用内"的延迟双删，Canal + binlog 方案多了一个 Canal Server 依赖——
> 需要 MySQL 开启 `--log-bin --binlog-format=ROW`、配置专门的 dump 账号、独立部署监控这个中间件。
>
> **话术**：「这是一个真实的取舍：用"多一个外部依赖"换"消除一个拍脑袋的经验值 + 解耦缓存失效逻辑与业务代码"。
> 我们用 `canal.enabled` 开关做了灰度——关闭时退化为「写前删缓存 + 30 分钟 TTL 兜底」，业务依然正确；
> 即使打开后 Canal Server 不可用，TTL 也兜底到「最长 30 分钟」的不一致窗口，不会影响正确性，只是
> 退化为升级前的体验。所以这次升级是"加分项"而不是"新的单点故障"——它只在正常工作时提供额外的及时性，
> 从不在异常时降低系统的正确性下限。」

### 3. ShardingSphere 5.5.0 SQL 兼容性限制

> **弱点**：`LIMIT` 分页内存放大 N 倍；不支持跨分片 JOIN；部分聚合 SQL 需要改写。
>
> **话术**：「ShardingSphere 5.5.0 的 SQL 兼容性不是 100%，这是分布式中间件的普遍问题。我们通过主键分页（`WHERE id > last_id`）代替 `OFFSET` 分页；跨分片统计走 Prometheus 实时指标或 ElasticSearch；已知 bug（test-util 缺失）通过 exclusion 解决。」

### 4. BM25 内存索引重启丢失

> **弱点**：BM25 索引是进程内内存，服务重启后需要重建。
>
> **话术**：「BM25 内存索引在服务重启后丢失，需要重新 ingest 知识库（约几秒），期间 Hybrid RAG 降级为纯向量检索（功能不中断）。生产中可以在启动时自动触发 ingest，或者把 BM25 索引持久化到 Redis（以牺牲一定内存为代价）。」

### 5. LLM-as-Judge 评测有偏差

> **弱点**：用 Haiku 评测 Sonnet 的输出，存在「宽容评分」等固有偏差。
>
> **话术**：「LLM-as-judge 有已知偏差，但它依然比纯关键词匹配更接近语义质量。我们通过三个手段缓解：固定 temperature=0 减少随机性；明确的评分标准（prompt 里写了 0/0.3/0.7/1.0 的对应标准）；人工抽检 10% 做偏差校准。这与 RAGAS 框架的最佳实践一致。」

### 6. 项目是个人项目，不是公司生产系统

> **弱点**：没有真实的高并发流量验证，性能数据来自本地测试。
>
> **话术**：「这是个人作品，业务场景和技术实现都基于真实的生产级设计（秒杀/支付/MQ 等均参照头部电商系统的常见方案），但确实没有经过生产流量验证。性能数据来自本地 Locust 压测和单机环境，与生产有差距。简历会标注「个人项目，基于真实业务场景设计」，不夸大为公司项目。」
