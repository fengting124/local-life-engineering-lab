# Findings

## Local Spec
- 当前规范强调三条并行主线：功能推进、代码理解、复盘沉淀
- 当前规范强调 AI 的角色是架构陪跑、代码协作、联调辅助、复盘追问
- 当前规范强调每个模块都要按目标定义、小步实现、即时验证、主链路复述、关键代码精读、文档沉淀推进
- 当前规范强调项目学习的目标是能复述、能定位、能改动、能解释技术取舍
- 当前规范明确反对纯跟教程和纯抄代码，主张把项目设计成学习现场
- 当前规范允许经典项目二次构建，也允许 AI 直接辅助设计新项目，关键在于分阶段递进和真实落地

## External Research
- 牛客近一年后端面经和讨论高频聚焦在缓存一致性、热 key、本地缓存和分布式缓存边界、Redis 宕机兜底、秒杀库存预扣减、QPS 压测、慢 SQL 排查、MQ 重复消费和一致性补偿
- 牛客 AI Agent 相关讨论高频聚焦在 Tool 与 Skill 分层、MCP 的协议定位、工具集合治理、RAG 分块与召回、SSE 流式返回、异常兜底和评测体系
- 官方 Spring AI 文档确认 MCP 在 Java 侧已经有成熟的 Boot Starter 与注解方式，适合 Java 主栈项目落地
- 官方 MCP 规范与 SDK 文档确认远程 MCP 更推荐 Streamable HTTP，SSE 在 MCP 传输层属于兼容旧客户端的 legacy 方案
- 牛客项目求助帖的共识是：经典项目依然适合打基础，但直接原样写进简历同质化严重；需要改题材、补模块、补指标、补异常场景、补压测和补偿链路
- 牛客多篇项目建议都把“亮点是否能被追问”放在“题材是否新颖”前面，重点不是换壳，而是是否能讲出为什么这么设计
- Agent 项目相关帖子反复强调：简历亮点要写工具边界、Router、Controller、工作流、观测指标和策略优化，不能只写 Prompt 或模型调用

## Project Direction
- 后端项目不能停留在点评、外卖、商城式 CRUD，必须主动设计工程问题并给出方案对比
- AI 项目不能停留在接模型 API，必须体现检索、路由、控制、评测、协议设计和可观测性
- 两条线要共享一部分基础设施与工程方法：测试、压测、监控、故障演练、复盘文档
- 双线项目应围绕同一业务域组织，形成相互加成的简历叙事
- 面向 Java 求职，最佳路线是传统后端作为主项目，AI 应用作为进阶项目，前者证明工程底座，后者证明方向感和增量能力
