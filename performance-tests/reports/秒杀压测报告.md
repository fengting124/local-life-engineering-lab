# 秒杀压测报告

## 1. 目标

验证 `POST /api/v1/seckill` 在高并发下的三件事：

| 维度 | 验收口径 |
|------|----------|
| 正确性 | `claimed_success <= Redis 总库存`，不允许超卖 |
| 幂等性 | 重复领取返回业务失败或 429，不产生重复券 |
| 延迟 | k6 自定义指标 `seckill_latency` P95 < 200ms，P99 < 500ms |
| 稳定性 | `unexpected_error_rate < 1%`，业务 400/429 不算系统错误 |

## 2. 被测链路

```text
k6 虚拟用户
  -> POST /api/v1/auth/login
  -> POST /api/v1/seckill
  -> RateLimitInterceptor
  -> SeckillService
  -> Redis Lua: 库存扣减 + 用户判重
  -> MySQL user_coupon / outbox_message
```

秒杀主判断在 Redis Lua 内完成，MySQL 负责最终落库和幂等兜底。压测重点不是单纯打出高 QPS，而是证明“成功领取数不会超过库存”。

## 3. 数据准备

```bash
bash scripts/seed-perf-data.sh
```

默认准备：

| 数据 | 默认值 |
|------|--------|
| 手机号池 | `13900000000` ~ `13900001999` |
| 登录验证码 | `123456` |
| 秒杀场次 | `1:1`, `2:2` |
| Redis 库存 | 每个场次 `1000` |

## 4. k6 执行命令

本机需要先安装 k6；未安装时可先用下方 Locust 对照方案演示同一业务目标。

```bash
k6 run performance-tests/k6/seckill.js
```

推荐面试演示命令：

```bash
USERS=500 RUN_TIME=60s EXPECTED_STOCK=1000 \
SECKILL_SESSIONS=1:1,2:2 \
k6 run performance-tests/k6/seckill.js
```

输出文件：

```text
performance-tests/reports/seckill-k6-summary.json
```

## 5. 结果判定

压测结束后重点看控制台 summary：

| 指标 | 含义 | 通过条件 |
|------|------|----------|
| `claimed_success` | 真正抢到券的成功次数 | `<= EXPECTED_STOCK * 场次数` |
| `business_sold_out` | 库存不足、未抢到 | 可接受 |
| `business_duplicate` | 重复领取拦截 | 可接受 |
| `rate_limited` | 用户级限流触发 | 可接受 |
| `unexpected_error` | 5xx、鉴权失败、未知响应 | 越接近 0 越好 |
| `oversold` | 是否超卖 | 必须是 `NO` |

当前仓库已补齐可执行 k6 脚本和报告模板。正式性能数字需要在目标机器上运行上述命令后，以本机实际 CPU、内存、Docker 配置和 Redis/MySQL 状态为准，不在文档中伪造结果。

## 6. Locust 对照方案

项目原有 Locust 脚本也能跑秒杀专项：

```bash
cd performance-tests
pip install -r requirements.txt
locust -f locustfile_locallife_server.py \
  --host http://localhost:8080 \
  --users 500 --spawn-rate 50 --run-time 30s \
  --headless --only-summary SeckillUser
```

k6 更适合快速 CLI 演示和产出 JSON summary；Locust 更适合 Web UI 观察 RPS、P99 和错误分布。
